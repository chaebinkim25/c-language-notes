반갑습니다, 사이버 보안학과 신입생 여러분.

지금까지 우리는 데이터를 다루는 변수, 흐름을 제어하는 `if`와 `for`, 그리고 많은 데이터를 저장하는 배열과 메모리 주소를 다루는 포인터까지 숨 가쁘게 달려왔습니다. 
오늘 읽기 수업의 핵심인 **11장 함수 _function_** 은 지금까지 배운 모든 문법을 하나로 묶어내는 조립 공정이자, 해커의 공격 피해를 최소화하는 보안 격벽 _bulkhead_ 입니다.

함수는 복잡한 문제를 작은 실행 단위로 쪼개어 가독성과 유지보수성을 높여주지만, 반대로 외부의 위험한 데이터가 내부로 들어오는 통로 _input gate_ 가 되기도 합니다. 
오늘은 1장부터 11장까지의 지식을 총동원하여, 함수를 호출하고 값을 주고받는 과정에 어떤 보안 취약점들이 숨어 있는지 코드로 알아보겠습니다.

### \[실습 1\] 철저한 격리: 값 복사 _call by value_ 와 격벽
* **난이도:** ★★☆☆☆ 
* **관련 이론:** 11.6.2 함수 호출시 값은 복사된다, 11.6.1 변수 이름이 유효한 범위

다음은 사용자의 권한을 관리자(9)로 올리려고 시도하는 코드입니다. `main` 함수가 끝날 때 `my_level`은 얼마일까요?

```c
void upgrade_privilege(int level)
{
        /* 관리자 권한 부여 시도 */
        level = 9; 
}

int main(void)
{
        int my_level = 1; /* 1: 일반 유저 */
        
        /* 권한 상승 함수 호출 */
        upgrade_privilege(my_level);
        
        return 0;
}
```

#### \[해설\]
- **정답:** `my_level`은 여전히 **1**입니다.
- **해석:** C 언어에서 함수를 호출할 때는 인자의 값만이 매개변수에 복사되어 전달됩니다 _call by value_.
  `upgrade_privilege` 함수 안의 `level`은 `main` 함수의 `my_level`과는 완전히 다른 별개의 메모리 상자로 동명이인입니다.
  따라서 함수 안에서 값을 아무리 바꿔도 원본에는 영향을 주지 않습니다.
- **보안 포인트:** 이를 보안 용어로 격벽 _compartmentalization_ 원리라고 합니다.
  함수는 자신만의 독립적인 메모리 공간을 쓰기 때문에, 해커가 특정 함수를 장악하더라도 다른 함수의 지역 변수까지 마음대로 오염시키지 못하도록 피해를 최소화해 줍니다.
  원본을 꼭 바꾸고 싶다면 10장에서 배운 주소를 넘겨주어야 합니다.

### \[실습 2\] 침묵의 대가: 반환값 무시로 인한 _fail fast_ 위반
- **난이도:** ★★★☆☆ 
- **관련 이론:** 11.2 함수의 세 가지 요소, 6.6.1 방어적 프로그래밍

다음 코드는 보안 인증 함수를 호출하는 로그인 로직입니다. 무엇이 문제일까요?

```c
int check_auth(int password)
{
        if (password != 1234) {
                return -1; /* 인증 실패 에러 코드 반환 */
        }
        return 0; /* 인증 성공 */
}

int main(void)
{
        int user_pw = 0000; /* 틀린 비밀번호 */
        
        /* 인증 함수 호출 */
        check_auth(user_pw);
        
        /* ... 시스템 접근 허용 로직 ... */
        
        return 0;
}
```

#### \[해설\]
- **정답:** 인증이 실패하여 `-1`을 반환했음에도, `main` 함수가 반환값을 무시하고 시스템 접근을 허용해 버립니다.
- **해석:** `check_auth` 함수는 임무 실패 시 `-1`을 뱉어내어 상황을 보고했습니다.
  하지만 `main` 함수 사장님이 `check_auth` 함수 직원의 보고서를 아예 읽어보지 않고 폐기해 버렸습니다.
- **보안 포인트:** 보안 시스템에서 함수가 실패했는데도 정상처럼 행동하는 것은 가장 위험한 설계입니다.
  반환값을 무시하는 것은 _MISRA C_ 규정을 위반하는 행위입니다.
  에러가 발생하면 즉시 프로그램을 멈추거나 차단하는 조기 실패 _fail fast_ 원칙을 반드시 지켜야 합니다.

### \[실습 3\] 절대 닿을 수 없는 방어막: _unreachable code_
- **난이도:** ★★★☆☆ 
- **관련 이론:** 11.5.1 `return`은 즉시 탈출이다

다음은 해킹 시도를 감지하면 함수를 즉시 종료하고 로그를 남기려는 코드입니다. 여기서 심각한 결함은 무엇일까요?

```c
int process_data(int is_hacked)
{
        if (is_hacked == 1) {
                return -1; /* 해킹 감지 시 즉시 탈출 */
        
                /* 해킹 시도 기록 (로그 남기기) */
                int alert = 1; 
        }        
        return 0;
}
```

#### \[해설\]
- **정답:** `int alert = 1;` 코드는 영원히 실행되지 않습니다.
- **해석:** `return` 문은 단순히 값을 돌려주는 배달부가 아니라, 함수의 실행을 그 즉시 종료하는 '마침표'입니다.
  `return` 뒤에 남아있는 코드는 실행 흐름이 영원히 닿을 수 없는 도달할 수 없는 코드 _unreachable code_ 가 됩니다.
- **보안 포인트:** 보안 표준 _CWE-561_ 에서는 실행되지 않는 코드를 단순한 낭비가 아니라 논리적 착각으로 보며 엄격히 금지합니다.
  해킹을 방어하는 코드를 작성해 놓고도 `return` 뒤에 배치하는 실수로 인해, 실제로는 아무런 방어막이 없는 상태로 방치되는 아찔한 보안 사고가 발생할 수 있습니다.

### \[실습 4\] 잃어버린 경계선: 배열 매개변수와 버퍼 오버플로우
- **난이도:** ★★★★★ 
- **관련 이론:** 11.8 배열을 함수로 넘기기, 8.1 배열과 반복문

네트워크 패킷 3개를 검사하기 위해 배열을 함수로 넘기는 코드입니다. 해커에게 어떤 빌미를 제공하게 될까요?

```c
/* 배열을 받아 검사하는 함수 */
void inspect_packets(int arr[])
{
        int i;
        /* 배열 크기를 몰라서 임의로 10번 반복 */
        for (i = 0; i < 10; i = i + 1) { 
                arr[i] = 0; /* 패킷 초기화 */
        }
}

int main(void)
{
        int packets[11] = {1, 2, 3}; /* 3칸짜리 배열 */
        
        inspect_packets(packets); /* 배열 전달 */
        
        return 0;
}
```

#### \[해설\]
- **정답:** 배열의 실제 크기(3칸)를 넘어서 10칸까지 값을 덮어쓰는 **버퍼 오버플로우(Buffer Overflow)**가 발생합니다.
- **해석:** 함수 매개변수 자리에 `int arr[]`라고 적으면, C 언어는 배열 전체를 통째로 복사하는 것이 아니라 단순히 **'첫 번째 원소의 시작 위치(주소값)'**만을 전달합니다 (배열의 포인터 퇴화). 따라서 `inspect_packets` 함수 안에서는 이 배열이 원래 몇 칸짜리인지 절대 알 길이 없습니다.
- **보안 포인트:** 크기를 모르는 상태에서 반복문을 돌리면 허가되지 않은 메모리를 파괴하게 되며, 이는 해커가 돌아갈 목적지 주소(return address)를 조작하여 프로그램 제어권을 빼앗는 스매싱 공격(Smash the Stack)의 핵심 원리가 됩니다. 배열을 함수로 넘길 때는 반드시 배열의 크기도 또 다른 매개변수로 함께 넘겨주는 것이 기본 철칙입니다.

### \[실습 5\] 필요 없는 건 무시하기: 포인터와 `NULL` 체크
- **난이도:** ★★★★☆ 
- **관련 이론:** 11.7.2 출력 인자, 11.7.3 `NULL` 체크의 매력

다음은 암호화 키 두 개를 생성하여 돌려주는 함수입니다. 
`main` 함수에서는 두 번째 키 `key2`가 필요 없어서 `NULL`을 전달했습니다. 
그러면 어떻게 동작할까요?

```c
/* 포인터를 이용해 2개의 결과값을 내보내는 함수 */
void generate_keys(int *key1, int *key2)
{
        if (key1 != NULL) {
                *key1 = 1234; /* 첫 번째 키 발급 */
        }
        if (key2 != NULL) {
                *key2 = 5678; /* 두 번째 키 발급 */
        }
}

int main(void)
{
        int primary_key = 0;
        
        /* 두 번째 인자에는 NULL 전달 */
        generate_keys(&primary_key, NULL);
        
        return 0;
}
```

#### \[해설\]
- **정답:** `primary_key`에는 `1234`가 담기고, 두 번째 키 생성 코드는 안전하게 건너뛰어 에러가 발생하지 않습니다.
- **해석:** 함수는 `return`으로 값을 하나밖에 반환할 수 없기 때문에, 여러 개의 결과를 얻어내기 위해 "그릇의 주소를 줄 테니 담아줘"라는 출력 인자 _output parameter_ 패턴을 사용합니다. 
  필요 없는 결과에 대해서는 `NULL`(가리키는 곳이 없음)을 전달할 수 있습니다.
- **보안 포인트:** 함수 내부에서 `if (key != NULL)`을 통해 유효한 주소인지 검사하지 않고 무작정 역참조 `*key2 = ...`를 했다면 미정의 동작이 발생해 프로그램이 즉시 종료 _crash_ 되었을 것입니다.
  포인터를 매개변수로 받을 때는 반드시 `NULL` 체크를 하여 안전장치를 마련하는 동시에 함수의 유연성도 높여야 합니다.

> ### \[마무리 당부\]
> 함수는 단순히 코드를 줄여주는 타자기 같은 도구가 아닙니다. 입력 통로인 매개변수를 통해 들어오는 재료를 철저히 의심하고, 안전한 격리 구역인 블록에서 작업한 뒤,
> 정확한 결과인 반환값을 보고하는 하나의 작고 견고한 요새가 되어야 합니다.
> 
> 오늘 본 5가지 취약점은 해커들이 대규모 시스템을 침투할 때 사용하는 단골 메뉴들입니다.
> 1장부터 11장까지 배운 제어문과 메모리, 포인터의 원리를 결합하여 "함수의 벽"을 뚫리지 않게 설계하는 진정한 보안 전문가로 성장하시기 바랍니다. 
>

### \[실습 6\] 환영의 그림자: 변수 이름 가려짐 _shadowing_
- **난이도:** ★★★★☆ 
- **관련 이론:** 11.6.1 변수 이름이 유효한 범위, 이름 가려짐 현상

다음은 사용자의 권한을 검사하고, 문제가 있으면 임시로 권한을 부여하여 로직을 처리하려고 만든 함수입니다. 함수가 끝난 후 반환되는 `is_admin`의 값은 무엇일까요?

```c
int check_auth(int is_admin) {
        if (is_admin == 0) {
                /* 권한이 없으면 1로 덮어쓰기 시도 */
                int is_admin = 1; 
        }
        
        return is_admin;
}

int main(void) {
        int result = check_auth(0);
        return 0;
}
```

#### \[해설\]
- **정답:** `result`는 여전히 **0**입니다. 권한 부여에 실패했습니다.
- **해석:** `if`문 블록 `{ }` 안에서 `int is_admin = 1;`이라고 선언하는 순간, 이 변수는 바깥에서 받아온 매개변수 `is_admin`과는 이름만 같은 완전히 다른 새로운 상자가 됩니다.
  안쪽 블록이 끝날 때까지 바깥쪽 변수는 가려져서 접근할 수 없게 되며, 안쪽 변수는 블록이 닫히는 `}` 순간 메모리에서 치워져 사라집니다.
- **보안 포인트:** 이를 보안 용어로 이름 가림 _shadowing_ 이라고 합니다.
  개발자는 권한을 승격시켰다고 착각하지만, 실제로는 아무런 상태도 변하지 않았습니다.
  의도치 않은 변수에 값을 잘못 저장하여 치명적인 권한 제어 실패를 유발할 수 있으므로, _CERT_ 와 _MISRA C_ 에서는 이름 가림 현상을 엄격히 금지하고 취약점으로 분류합니다.

### \[실습 7\] 착각의 크기: 매개변수로 넘어온 배열과 `sizeof`
- **난이도:** ★★★★★ 
- **관련 이론:** 11.8.1 배열을 매개변수에 쓰는 법, 10.5.1 크기 계산 연산 `sizeof`

다음은 해커가 남긴 악성 데이터인 100칸짜리 배열의 흔적을 `0`으로 덮어써서 지우려는 메모리 초기화 함수입니다. 배열을 완벽하게 청소할 수 있을까요?

```c
void clear_buffer(int arr[]) {
        int i;
        /* 배열의 전체 크기만큼 반복하며 0으로 청소 시도 */
        for (i = 0; i < sizeof(arr); i = i + 1) {
                arr[i] = 0;
        }
}

int main(void) {
        int buffer[100] = { /* 악성 데이터 100개 */ };
        
        clear_buffer(buffer);
        
        return 0;
}
```

#### \[해설\]
- **정답:** 완벽하게 청소하지 못합니다. 전체 배열 100칸 중 아주 앞부분인 1칸 또는 2칸만 지우고 반복문이 멈춥니다.
- **해석:** C 언어에서 배열이 함수의 인자로 넘어갈 때는 배열 전체가 넘어가는 것이 아니라 제일 앞 원소의 시작 주소로 자동 변환되어 전달됩니다.
  따라서 `clear_buffer` 함수 안의 `arr`는 진짜 배열이 아니라 단순히 주소를 담는 '포인터 변수'일 뿐입니다.
  포인터에 `sizeof` 연산을 하면 배열의 크기인 400바이트가 아니라 단순히 주소 변수의 크기인 4바이트 또는 8바이트가 나옵니다.
- **보안 포인트:** 방 전체를 청소하라고 지시했지만, 문 앞 타일 하나만 닦고 청소가 끝났다고 보고하는 꼴입니다.
  배열의 뒷부분에 남아있는 악성 데이터나 중요 비밀번호는 그대로 메모리에 잔존하게 되어 심각한 정보 노출 _information disclosure_ 취약점으로 이어집니다.
  배열을 인자로 넘길 때는 반드시 배열의 실제 크기도 별도의 매개변수로 함께 넘겨야 합니다.

### \[실습 8\] 예측 불가: 인자 계산 순서의 룰렛 게임
- **난이도:** ★★★★☆ 
- **관련 이론:** 11.4.1 호출식의 피연산자는?, 인자의 계산 순서는 보장되지 않는다

다음은 시스템에 사용자와 관리자의 접속 기록을 각각 남기고, 그 결과를 모아서 최종 처리하는 로직입니다. 과연 `user`와 `admin` 중 누구의 접속 기록이 먼저 남게 될까요?

```c
/* 각각 기록을 남기고 1을 반환하는 함수라고 가정 */
int log_user(void);
int log_admin(void);
void process_logs(int u, int a);

int main(void) {
        /* 두 함수의 실행 결과를 인자로 전달 */
        process_logs(log_user(), log_admin());
        
        return 0;
}
```

#### \[해설\]
- **정답:** 알 수 없습니다. 컴파일러 마음대로 결정됩니다.
- **해석:** C 언어 표준은 함수를 호출하기 전에 인자를 먼저 계산하도록 규정하지만, 여러 개의 인자 중 어떤 것을 먼저 계산할지 그 순서는 정해두지 않았습니다 _unspecified behavior.
  `log_user()`가 먼저 실행될 수도 있고, `log_admin()`이 먼저 실행될 수도 있습니다.
- **보안 포인트:** 보안 감사 시스템 _audit logging_ 에서 기록의 선후 관계는 해킹 사고를 추적할 때 결정적인 단서가 됩니다.
  인자 계산 순서에 의존하여 코드를 작성하면, 시스템을 옮기거나 컴파일러를 바꿀 때마다 로그의 순서가 뒤바뀌는 끔찍한 일이 발생합니다.
  순서가 중요한 함수 호출은 반드시 개별 변수에 먼저 대입하여 실행 순서를 명확히 강제해야 합니다.

### \[실습 9\] 침묵의 시한폭탄: 반환값 누락으로 인한 제어 흐름 붕괴
- **난이도:** ★★★★★ 
- **관련 이론:** 11.5.2 `return` 문의 정식 문법, 반환 값이 없는 함수

다음은 외부에서 들어온 입력값이 안전한지 검사하는 함수입니다. 
입력값이 양수여서 안전하면 아무 문제가 없다고 보고 `return 0;`을 해야 하는데, 개발자가 이를 깜빡했습니다. 
그 결과 어떤 재앙을 불러올까요?

```c
int validate_input(int value) {
        if (value < 0) {
                return -1; /* 에러 시 -1 반환 */
        }
        /* 정상일 때 return 0; 을 작성해야 하는데 빼먹음! */
}

int main(void) {
        /* 정상적인 양수 100을 전달 */
        int status = validate_input(100);
        
        if (status == 0) {
                /* 안전한 시스템 접근 허용 */
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** `status`에 무슨 값이 들어갈지 알 수 없으며, 정상적인 입력임에도 시스템 접근이 차단되거나 엉뚱한 로직이 실행될 수 있습니다.
- **해석:** 반환 타입이 `int`인 함수에서 값을 반환하지 않고 블록 끝 `}`에 도달하는 것은 C 표준에서 미정의 동작 _undefined behavior_ 입니다.
  함수는 메모리에 남아있던 알 수 없는 쓰레기 값을 반환하게 됩니다.
- **보안 포인트:** 함수가 약속된 결과값 `0`을 주지 않고 쓰레기 값을 던지기 때문에, `main` 함수에 있는 `if (status == 0)` 검사는 실패할 확률이 높습니다. 
  보안 용어로 제어 흐름의 붕괴 _control-flow corruption_ 라고 합니다.
  정상 사용자의 서비스 가용성 _availability_ 를 해치며, 해커가 메모리 구조를 조작해 이 '쓰레기 반환값'을 자신이 원하는 숫자로 조작하면 방어 로직을 무력화시킬 수도 있는 치명적인 실수입니다.
