반갑습니다, 사이버 보안학과 신입생 여러분.

지금까지 우리는 1장부터 6장까지 C 언어의 기본 구조와 데이터의 처리, 그리고 if문을 통한 논리적 판단 방식을 배웠습니다. 
오늘 다룰 **7장 반복문 _for_** 는 컴퓨터가 가진 파괴력의 핵심이자, 해커와 보안 전문가가 가장 치열하게 맞붙는 전장입니다.

해커가 비밀번호를 알아내기 위해 모든 경우의 수를 눈 깜짝할 새에 대입해보는 
무차별 대입 공격 _Brute Force Attack_ 은 바로 이 반복문을 통해 이루어집니다. 

컴퓨터는 인간과 달리 지치지 않고 수천, 수만 번의 공격과 방어를 완벽하게 똑같이 반복할 수 있기 때문에, 
해커와 보안 전문가 모두에게 반복문이 중요합니다. 

오늘은 1장부터 7장까지의 모든 지식을 총동원하여, 
반복문 속에 숨겨진 치명적인 보안 취약점을 찾아내는 읽기 훈련을 시작하겠습니다. 

코드는 모두 C89 표준에 맞추어 작성되었으며, 반복문 원리에 집중하기 위해 `i++` 대신 `i = i + 1`을 사용했습니다.

## 1부: 반복문 자체의 문법적 허점

### \[실습 1\] "딱 한 번만 더!" : 오프 바이 원 _Off-by-one_ 오류
- **난이도:** ★★★☆☆ 
- **관련 이론:** 7.3 for문의 기본 모양, 6.3 비교 기호

다음은 로그인 시스템에서 비밀번호 입력 기회를 3번으로 제한하는 방어 코드입니다. 
해커에게 총 몇 번의 입력 기회를 주게 될까요?

```c
int main(void)
{
        int retry;
        int attempt_count = 0;
    
        /* 비밀번호 입력 3회 제한 */
        for (retry = 0; retry <= 3; retry = retry + 1) {
                attempt_count = attempt_count + 1;
                /* 로그인 시도 로직 (생략) */
        }
    
        return 0;
}
```

#### \[해설\]
- **정답:** 기회를 4번 줘서 `attempt_count`는 4가 됩니다.
- **해석:** `retry` 변수가 0, 1, 2, 그리고 3일 때까지 조건 `retry <= 3`이 참이 되므로 총 4번 반복됩니다.
- **보안 포인트:** 이를 보안 용어로 **오프 바이 원 오류 _Off-by-one error_** 라고 합니다.
  초보자는 조건식에 `<`를 써야 할지 `<=`를 써야 할지 자주 헷갈립니다.
  단 한 번의 추가 실행 차이가 허가되지 않은 메모리를 건드려 정보를 유출하거나 데이터를 오염시키는
  _Out-of-Bounds, OOB_ 취약점의 핵심 원인이 됩니다.
  항상 "경계선이 어디인가?"를 긴장하며 확인해야 합니다.

### \[실습 2\] 반복이 남긴 유령: C89의 넓은 유효 범위
* **난이도:** ★★★★☆
* **관련 이론:** 7.4 for문 실행 순서, 7.5 C99 이후: 범위 축소로 더 안전해진 변수 선언

다음은 서버 접속을 3회 시도하고, 실패하면 프로그램을 종료하는 로직입니다. 
반복문이 끝난 직후, if문은 참이 될까요, 거짓이 될까요?

```c
int main(void)
{
        int i;
        
        for (i = 0; i < 3; i = i + 1) {
                /* 접속 시도, 실패했다고 가정 */
        }
        
        /* 3번 모두 실패했는지 확인 */
        if (i == 2) {
                /* 시스템 잠금 */
                return -1;
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** `if (i == 2)`는 **거짓 `0`** 이 되어 시스템 잠금이 실행되지 않습니다.
- **해석:** 반복문이 끝난 뒤의 `i` 값은 마지막으로 실행된 값 `2`가 아니라,
  조건을 거짓으로 만들어 반복을 끝내버린 값인 `3`이 됩니다.
- **보안 포인트:** 개발자가 `i`의 값이 2일 것이라고 착각하고 재사용하면,
  3회 제한이라는 보안 정책이 스스로 무너지는 심각한 논리적 버그가 발생합니다.

### \[실습 3\] 스스로 망가뜨린 브레이크: 무한 루프와 DoS
* **난이도:** ★★★★☆
* **관련 이론:** 7.6 for의 권장 코딩 스타일, 7.2 반복을 만드는 재료 3가지

다음은 네트워크 트래픽 5개를 검사하는 코드입니다. 프로그램이 언제 끝날까요?

```c
int main(void) {
        int i;
        int packet_checked = 0;
        
        for (i = 0; i < 5; i = i + 1) {
                packet_checked = packet_checked + 1;
                
                /* 복잡한 검사 로직 후 변수 엉킴 */
                i = 0; 
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** 영원히 끝나지 않는 무한 루프가 발생합니다.
- **해석:** for문의 조건이 무색하게, 반복문 본문 안에서 루프 카운터 변수 `i`를 0으로 계속 덮어쓰고 있습니다.
  따라서 `i < 5`라는 반복 조건, 다른 말로 하면 안전 브레이크는 영원히 발동하지 않습니다.
- **보안 포인트:** 이런 스파게티 코드 _Spaghetti code_ 는 프로그램의 실행 흐름을 꼬이게 만들어 가독성을 망칩니다.
  더 큰 문제는 루프가 끝나지 않으면 프로그램이 CPU 자원을 100% 점유하게 되어,
  해커가 고의로 시스템을 마비시키는 서비스 거부 _DoS_ 공격과 똑같은 피해를 입힌다는 점입니다.
  자동차 소프트웨어 표준 _MISRA C_ 에서도 루프 본문 내 카운터 변수 변경을 엄격히 금지합니다.

### \[실습 4\] 허공에 발차기: 세미콜론 ;의 착시
* **난이도:** ★★★★★
* **관련 이론:** 6.7.2 if(조건);의 함정, 7.3 for문의 기본 모양

다음 코드는 해커 IP들의 목록에서 1000개를 스캔하는 반복문입니다. 실행 후 `scan_count`는 얼마가 될까요?

```c
int main(void)
{
        int i;
        int scan_count = 0;
        
        /* 1000번 반복하며 스캔 횟수 증가 */
        for (i = 0; i < 1000; i = i + 1);
        {
                scan_count = scan_count + 1;
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** `scan_count`는 `1`이 됩니다.
- **해석:** 6장에서 `if(조건);` 뒤에 실수로 세미콜론을 찍으면 빈 문장 _null statement_ 으로 인식되어
  if문이 끝나버린다는 것을 배웠습니다.
  for문 역시 괄호 안의 세미콜론 2개는 부품을 나누는 칸막이 역할을 하지만,
  괄호가 끝난 직후에 찍힌 세미콜론은 반복 실행할 문장이 "아무것도 하지 마라"임을 뜻합니다.
- **보안 포인트:** 눈으로 보기엔 1000번 스캔한 것 같지만, 컴퓨터는 허공에 1000번 빈 반복을 한 뒤,
  밑에 묶여있는 `scan_count = scan_count + 1;` 블록을 단 한 번만 실행합니다.
  보안 검사가 완전히 무력화되는 치명적 실수입니다.

### \[실습 5\] 문을 닫아버린 경비원: break와 continue의 오남용
* **난이도:** ★★★☆☆
* **관련 이론:** 7.7.1 탈출구 break, 7.7.2 건너뛰기 continue

다음은 악성 파일 10개를 순서대로 검사하는 백신 로직입니다. 
만약 1번째 파일이 안전한 파일이어서, `file_status`가 `0`이라면, 이 코드는 어떻게 동작할까요?

```c
int main(void)
{
        int i;
        int file_status = 0; /* 0: 안전, 1: 악성 */
        
        for (i = 1; i <= 10; i = i + 1) {
                /* 파일 상태 확인 (생략) */
                
                if (file_status == 0) {
                        /* 안전하면 다음 파일로 건너뜀? */
                        break; 
                }
                
                /* 악성 파일 처리 로직 */
        }
        
        return 0;
}
```
#### \[해설\]
- **정답:** 1번째 파일만 확인하고, 나머지 9개는 검사조차 하지 않고 전체 반복문을 즉시 종료해 버립니다.
- **해석:** `break`는 자신이 속한 반복문을 즉시 종료시키고 밖으로 나가는 '탈출구'입니다.
  안전한 파일이므로 해당 회차만 무시하고 다음 파일을 검사하게 하려면,
  이번만 패스하고 갱신 단계로 점프하는 `continue`를 썼어야 합니다.
- **보안 포인트:** 해커가 이 구조를 알게 되면 어떻게 공격할까요?
  악성 파일 9개 앞에 '정상 파일' 딱 1개만 미끼로 던져두면 됩니다.
  백신은 정상 파일을 보자마자 안심하고 `break`를 호출해 작동을 멈추게 되고,
  뒤따라오는 악성 파일들은 아무런 제재 없이 시스템을 장악하게 될 것입니다.

> **\[마무리 - 1부\]** > 여러분, 반복문 `for`는 코드 몇 줄로 수백만 번의 작업을 지시하는 마법 같은 문법입니다.
> 하지만 그 강력함 이면에는 한 번의 부등호 실수 `<` vs `<=`,
> 단 하나의 세미콜론 `;` 오타가 수백만 번의 재앙으로 증폭될 수 있다는 무서운 진실이 숨어있습니다.
> 우리는 컴퓨터의 흐름을 통제할 모든 도구 `int`, `if`, `for` 를 이제 손에 쥐었습니다.
> 오늘 배운 내용을 바탕으로 복습을 진행하시기 바랍니다.
> 수업 마칩니다.

앞서 다룬 5가지 실습이 반복문 자체의 문법적 실수나 구조적 허점을 찔렀다면, 이번에 추가할 4가지 실습은 1장부터 7장까지 배운 모든 지식(참/거짓, 변수 초기화, 나눗셈, 다중 제어)이 for문과 결합할 때 발생하는 고차원적인 함정들입니다. 자, 다시 컴퓨터의 눈으로 아래 코드들의 진짜 동작을 파헤쳐 봅시다.

### [실습 6] 빠져나갈 수 없는 미로: 중첩 반복문과 break
* **난이도:** ★★★★★
* **관련 이론:** 7.7.1 탈출구 break, 7.3 for문의 기본 모양

다음은 서버 3대의 각 포트 3개씩을 순서대로 스캔하다가, 해킹 흔적(`hacked == 1`)이 발견되면 즉시 전체 스캔을 중단하려는 코드입니다. 이 코드는 설계자의 의도대로 작동할까요?

		int main(void) {
		    int i;
		    int j;
		    int hacked = 0;
		    int scan_count = 0;

		    /* i: 서버 번호 (0~2), j: 포트 번호 (0~2) */
		    for (i = 0; i < 3; i = i + 1) {
		        for (j = 0; j < 3; j = j + 1) {
		            
		            if (hacked == 1) {
		                break; /* 해킹 감지 시 스캔 중단? */
		            }
		            
		            scan_count = scan_count + 1;
		            hacked = 1; /* 첫 번째 스캔에서 즉시 해킹 감지됨 */
		        }
		    }

		    return 0;
		}

#### [해설]
* **정답:** 설계자의 의도대로 멈추지 않고, 바깥쪽 루프는 계속 돌아갑니다. (`scan_count`는 3이 됨)
* **해석:** for문 안에 for문이 있는 중첩 반복문에서 `break`를 호출하면, 가장 안쪽의 for문 하나만 종료됩니다. 즉, 0번 서버 스캔 중 해킹을 감지해서 내부 루프를 깨고 나왔지만, 바깥쪽 루프는 멈추지 않았으므로 1번 서버와 2번 서버에 대한 스캔이 계속 진행됩니다.
* **보안 포인트:** 중첩 루프 탈출 실패는 시스템 자원을 불필요하게 낭비하거나, 이미 오염된 상태에서 검증되지 않은 다음 데이터를 계속 처리하게 만드는 심각한 로직 오류를 낳습니다. 전체를 깨고 싶다면 상태를 나타내는 플래그(flag) 변수를 바깥쪽 루프의 조건식에도 추가하여 확실히 차단해야 합니다.

### [실습 7] 부등호가 사라진 브레이크: 암시적 참/거짓 판단
* **난이도:** ★★★★☆
* **관련 이론:** 6.1 C에서 참/거짓 표현, 7.2 반복을 만드는 재료

다음은 보안 타이머를 카운트다운하는 코드입니다. for문의 조건 칸(`count;`)에 부등호가 없습니다! 이 코드는 몇 번 반복될까요?

		int main(void) {
		    int count;
		    int execute_times = 0;

		    /* 조건식에 부등호(<, >)가 없다! */
		    for (count = 3; count; count = count - 1) {
		        execute_times = execute_times + 1;
		    }

		    return 0;
		}

#### [해설]
* **정답:** 3번 반복됩니다. (`execute_times`는 3)
* **해석:** 6장에서 배웠듯 C 언어는 0을 거짓(False)으로, 0이 아닌 모든 값을 참(True)으로 봅니다. for문의 가운데 칸은 반드시 부등호 비교식일 필요가 없습니다.
  * `count`가 3, 2, 1일 때는 '0이 아니므로' 참이 되어 반복합니다.
  * `count`가 0이 되는 순간 거짓으로 평가되어 루프가 종료됩니다.
* **보안 포인트:** 해커들이나 고급 개발자들은 코드를 짧게 만들기 위해 이런 암시적 조건을 자주 사용합니다. 보안 분석가는 값이 정확히 0이 되는 순간이 언제인지를 추적할 수 있어야 합니다.

### [실습 8] 독이 든 항아리: 누적 변수 초기화 누락
* **난이도:** ★★★☆☆
* **관련 이론:** 7.0 값 누적하기, 5.4.3 쓰레기 값(Garbage Value)

다음은 네트워크로 들어온 5개의 패킷 크기를 모두 더해(누적) 총 트래픽을 계산하는 코드입니다. `total_traffic`에는 어떤 값이 들어갈까요?

		int main(void) {
		    int i;
		    int total_traffic; /* 앗! 초기화 누락 */

		    /* 5번 반복하며 값을 누적 (accumulating) */
		    for (i = 0; i < 5; i = i + 1) {
		        total_traffic = total_traffic + 100;
		    }

		    return 0;
		}

#### [해설]
* **정답:** 알 수 없는 쓰레기 값
* **해석:** 7장에서 배운 '값 누적하기(accumulating)'의 기본은 빈 항아리를 준비하는 것입니다. 하지만 `total_traffic` 변수를 선언만 하고 0으로 초기화하지 않았습니다. 메모리 상자 안에는 이전에 쓰다 남은 알 수 없는 값(쓰레기 값)이 들어 있으며, 여기에 100을 5번 더해봐야 여전히 예측 불가능한 값일 뿐입니다.
* **보안 포인트:** 과금 시스템이나 트래픽 제어 로직에서 누적 변수 초기화를 빼먹으면, 고객에게 터무니없는 요금이 청구되거나 방화벽이 오작동하는 등 치명적인 무결성(Integrity) 침해 사고가 발생합니다.

### [실습 9] 잘려나간 꼬리: 정수 나눗셈과 반복 횟수
* **난이도:** ★★★★☆
* **관련 이론:** 5.6.2 int의 나눗셈, 7.3 for문의 기본 모양

다음은 10MB의 데이터를 3MB씩 쪼개서 암호화 처리하는 반복문입니다. 이 반복문은 총 몇 번 실행될까요?

		int main(void) {
		    int i;
		    int total_data = 10;
		    int chunk_size = 3;
		    int encrypted_count = 0;

		    /* 10 / 3 번 만큼 반복? */
		    for (i = 0; i < (total_data / chunk_size); i = i + 1) {
		        encrypted_count = encrypted_count + 1;
		    }

		    return 0;
		}

#### [해설]
* **정답:** 3번 반복됩니다. (`encrypted_count`는 3)
* **해석:** 수학적으로 10 나누기 3은 3.333... 이므로 4번에 걸쳐 처리해야 데이터를 모두 암호화할 수 있을 것 같습니다. 하지만 5장에서 배웠듯 C 언어의 `int`끼리의 나눗셈은 소수점 아래를 무조건 0을 향해 버립니다(truncate toward zero). 따라서 `total_data / chunk_size`의 결과는 정확히 3이 되고, for문은 `i = 0, 1, 2`까지만 실행됩니다.
* **보안 포인트:** 나머지 1MB의 데이터는 암호화되지 않은 평문(Plaintext) 상태로 고스란히 남게 됩니다! 데이터 분할이나 버퍼 처리를 반복문과 결합할 때는 나머지(`%`) 연산자를 활용하여 자투리 데이터가 남지 않았는지 끝까지 책임지고 처리해야 합니다.
