# [실습 수업] 보안 전문가의 눈으로 코드 읽기

사이버 보안학과 신입생 여러분, 반갑습니다. 지난 이론 수업에 이어 오늘은 '컴퓨터(컴파일러)의 눈'으로 코드를 해석하는 훈련을 진행하겠습니다. 
보안 전문가에게 가장 중요한 능력은 코드를 직접 짜는 것(*writing*)을 넘어, 남이 짠 코드의 진짜 의도와 그 안에 숨겨진 취약점을 파악하는 읽기(*reading*) 능력입니다.

C 언어는 "프로그래머가 자신이 무엇을 하는지 정확히 알고 있다"고 가정하여 설계된 언어입니다. 오늘 실습을 통해 텍스트 뒤에 숨겨진 메모리의 움직임을 포착하는 눈을 길러보시기 바랍니다.

---

## [실습 1] 인간의 눈 vs 컴퓨터의 눈: 들여쓰기의 기만
**난이도**: ★☆☆☆☆ | **관련 이론**: 2.6 들여쓰기와 코드 읽기 규칙

다음 코드는 유저의 권한을 설정하는 상황을 가정한 것입니다 (읽기 4, 쓰기 2, 실행 1). 

원래 '읽기+쓰기(6)' 권한을 가진 유저에게서 '쓰기(2)' 권한을 뺏으려 합니다.

`new_permission`의 최종 값은 무엇일까요?

```c
int main(void) {
        int old_permission = 6;
        int permission_removal = 2;
        int new_permission;

        /* 권한 변경 로직 */
        new_permission = old_permission - permission_removal;
                new_permission = 7;  /* 의도적으로 들여쓰기가 되어 있음 */
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **7** (모든 권한 부여)

- **해석**: 여러분의 뇌는 들여쓰기 된 `new_permission = 7;`을 보고 "아, 이건 예외 상황에만 실행되는 코드인가 보다"라고 착각했을 수 있습니다.
  하지만 C 컴파일러는 들여쓰기를 무시합니다. 컴퓨터는 순식간에 권한을 `4`로 만들었다가, 바로 다음 줄에서 모든 권한인 `7`로 덮어써 (*overwrite*) 버립니다.

- **보안 포인트**: 해커들은 악의적인 코드를 정상적인 코드인 것처럼 보이게 하려고 고의로 들여쓰기를 조작하기도 합니다.
  '눈에 보이는 로직'과 '실제 실행되는 로직'의 차이를 이용한 공격을 논리 우회(*logic bypass*)라고 합니다.
  실제 애플의 `goto fail` 보안 사고도 단 한 줄의 잘못된 들여쓰기 때문에 인증 절차를 통과시켜 버린 사건이었습니다.
  보안 전문가는 코드를 읽을 때 자신의 추측이 아니라, 문장(*statement*)의 끝(`;`)을 기준으로 실행 흐름을 추적해야 합니다.

# [실습 2] 사라진 소수점: 정수 나눗셈의 비밀
**난이도**: ★★☆☆☆ | **관련 이론**: 5.6.2 `int`의 나눗셈

다음은 데이터를 3개의 패킷으로 나누어 전송하려는 코드입니다. `packet_size`에 저장되는 값은 얼마입니까?

```c
int main(void) {
        int total_data = 10;
        int divider = 3;
        int packet_size; 
        
        /* 데이터를 3등분 하여 크기 계산 */
        packet_size = total_data / divider;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **3** (3.333...이 아님)

- **해석**: C 언어에서 `int` 타입끼리의 나눗셈 결과는 무조건 `int`입니다.
  소수점 이하 공간이 없으므로 0을 향해 잘려나갑니다 (*truncate toward zero*, 컴파일러가 C99 이후 표준을 준수할 경우). 

- **보안 포인트**: 질문하신 대로 `3`씩 보내면 데이터 `1`이 유실됩니다.
  이처럼 정수 연산의 특성상 발생하는 데이터 유실이나 정밀도 하락은 암호화 키 생성 시 키의 강도를 약화시키거나, 금융 시스템에서 잔액 불일치를 일으키는 치명적인 보안 허점이 됩니다.
  보안 전문가는 반드시 나머지 연산자(`%`)를 통해 남은 데이터가 있는지 검증해야 합니다.

## [실습 3] 설거지 안 된 컵의 위험: 미정의 동작(*undefined behavior*)
**난이도**: ★★★☆☆ | **관련 이론**: 4.3 데이터의 순환, 5.4.3 초기화

다음 코드는 보안 키 값을 계산하는 과정입니다. `key_value`의 최종 값은 무엇일까요?

```c
int main(void) {
        int default_key = 100;
        int extra_key;          /* 초기화 하지 않음 */
        int key_value;       
        
        key_value = default_key + extra_key;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **알 수 없음 (*undefined*)**
 
- **해석**: 변수를 선언만 하고 값을 넣지 않으면 메모리 상자 안에는 이전에 다른 프로그램이 쓰다 버린 쓰레기 값 *garbage value*가 들어있습니다.
  식당에서 설거지 안 된 컵에 물을 마시는 것과 같습니다.

- **보안 포인트**: C 언어는 성능을 위해 자동으로 `0`으로 초기화해주지 않습니다.
  공격자는 이런 초기화되지 않은 변수를 이용하여 메모리에 남아있는 민감한 정보를 유출하거나 (*memory leak*), 프로그램의 동작을 예측 불가능한 상태로 만듭니다.

## [실습 4] 넘침의 미학: 오버플로우(*overflow*)
**난이도**: ★★★★☆ | **관련 이론**: 5.7.2 `int` 범위를 넘는 계산

다음은 은행 계좌 시스템의 예시입니다. 
`balance`가 일반적인 시스템에서 `int`가 표현할 수 있는 최댓값(`2147483647`)에 가까운 값인 `2000000000`을 가지고 있을 때, 잔고가 두배가 되면 어떻게 될까요?

```c
int main(void) {
        int balance = 2000000000;

        balance = balance + balance;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **미정의 동작 (*undefined behavior*)**
    - 현대적인 시스템에서의 결과값: 약 -2.9억

- **해석**: `int` 상자의 벽을 넘어서는 순간, C 언어 표준은 더 이상 이 프로그램의 안전을 보장하지 않습니다.
  대부분의 시스템에서는 숫자가 순환하여 음수로 변하는 현상이 나타나지만, 어떤 시스템에서는 프로그램이 즉시 종료될 수도 있습니다.
  이를 오버플로우(*overflow*)라고 합니다.
  
- **보안 포인트**: 이는 해커들이 가장 좋아하는 공격 루트 중 하나입니다.
  잔액을 음수로 만들어 시스템을 혼란에 빠뜨리거나, 메모리 경계를 무너뜨리는 정수 오버플로우(*integer overflow*) 공격의 원리입니다.

## [실습 5] 연산자 우선순위와 부수 효과
**난이도**: ★★★★★ | **관련 이론**: 5.5.4 컴퓨터가 숫자를 세는 방법

다음 코드는 암호화 카운터를 증가시키는 로직입니다. 최종적으로 `counter` 값은 얼마가 될까요?

```c
int main(void) {
        int counter = 10;
        
        /* 자기 자신을 갱신 */
        counter = counter + 1;
        counter = counter + 1;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **12**

- **해석**: `counter = counter + 1;`은 수학의 방정식이 아니라 명령어입니다.
	1. **읽기**: 오른쪽 `counter` 상자에서 현재 값(**10**)을 꺼냅니다.
	2. **계산**: 꺼낸 값에 **1**을 더해 **11**을 만듭니다.
	3. **쓰기**: 결과값 **11**을 왼쪽 `counter` 상자에 덮어씁니다(*overwrite*). 기존의 **10**은 사라집니다.
	이 과정을 두 번 반복하여 최종적으로 **12**가 됩니다.

- **보안 포인트**: 암호화 프로토콜에서 카운터는 '단 한 번만 사용되는 번호(*Nonce*)' 역할을 합니다.
  만약 연산 실수나 로직 오류로 인해 카운터가 제대로 증가하지 않고 같은 번호가 두 번 쓰인다면,
  해커는 암호문을 복호화하거나 이전 메시지를 다시 보내는 재전송 공격(*replay attack*)을 성공시킬 수 있습니다.
  보안 전문가는 단 1의 오차도 허용하지 않는 데이터의 정밀한 흐름을 제어할 수 있어야 합니다.

## [실습 6] 연산자 우선순위와 부수 효과
**난이도**: ★★★★★ | **관련 이론**: 5.6.7 우선순위 총정리

다음 코드는 데이터 암호화에 사용할 임시 키(`result`)를 생성하는 로직입니다. 
프로그램 실행 후 `result`에 저장될 최종 값은 무엇일까요?

```c
int main(void) {
        int counter = 12;
        int result;
        
        /* 암호화 키 생성 수식 */
        result = counter + 2 * 5; 

        return 0;
}
```

- **정답**: **22**

- **해석**: 컴퓨터는 우리가 글을 읽는 순서대로만 계산하지 않습니다.
  C 언어의 규칙에 따라 곱셈 관련 연산자(`*`, `/`, `%`)는 덧셈 관련 연산자(`+`, `-`)보다 우선순위가 높습니다.

  1. **1단계**: `2 * 5`를 먼저 계산하여 **10**을 만듭니다.
  2. **2단계**: `counter`의 값인 **12**와 **10**을 더해 **22**를 만듭니다.
  3. **3단계**: 마지막으로 대입 연산자(`=`)가 작동하여 `result` 상자에 **22**를 저장합니다.

* **보안 포인트**: 만약 프로그래머가 왼쪽부터 순서대로 계산될 것이라 착각했다면(결과값 70 예상), 암호화 키는 설계자의 의도와 완전히 다른 값이 됩니다.
  이처럼 사소한 연산 순서의 오해는 암호화 알고리즘을 무력화하거나 인증 절차를 우회하게 만드는 논리적 결함(*logic bug*)의 시발점이 됩니다.
  보안 전문가는 자신의 기억을 믿기보다, 괄호 `()`를 사용하여 계산 순서를 명시적으로 강제하는 습관을 가져야 합니다.

* **보너스 질문**: 만약 `result = (counter + 2) * 5;`라고 적었다면 결과는 어떻게 변했을까요?

---

**[과제]** 위의 6가지 코드를 눈으로 먼저 해석해 본 뒤, 실제 컴파일러를 통해 실행하여 여러분의 예상이 맞았는지 확인해 보십시오. C 언어는 "자신이 무엇을 하는지 정확히 알고 있다"고 가정하는 전문가의 언어입니다. 오늘 배운 메모리의 움직임이 여러분의 보안 지도가 되길 바랍니다.

네, 학생들의 눈빛을 보니 이제 코드의 겉모습에 속지 않고 '데이터의 진실'을 파헤칠 준비가 된 것 같군요. 아주 좋습니다.

사이버 보안 전문가에게 코드를 읽는 능력은 숨겨진 함정을 찾아내는 탐지력과 같습니다. 
1장부터 5장까지 배운 내용을 바탕으로, 데이터의 표현 방식(리터럴)과 컴파일러의 해석 규칙을 파헤치는 심화 실습 4가지를 추가로 진행하겠습니다.

이번에는 데이터의 표현 방식(리터럴)과 연산자의 숨겨진 규칙을 집중적으로 파헤쳐 봅시다.

## [실습 7] 0의 가면을 쓴 스파이: 8진수의 함정
**난이도**: ★★★★☆ | **관련 이론**: 5.3 정수 리터럴 (8진수)

다음은 서버의 포트 번호를 설정하는 상황을 가정한 코드입니다. `port_a`와 `port_b`의 차이(`diff`)는 얼마일까요?

```c
int main(void) {
        int port_a = 10;
        int port_b = 010;  /* 숫자 앞에 0이 붙어있음 */
        
        /* 두 포트 번호의 차이를 계산 */
        int diff = port_a - port_b;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **2** (`port_a`는 **10**, `port_b`는 **8**)
 
- **해석**: 우리 눈에는 둘 다 **10**처럼 보이지만, C 언어에서 숫자 `0`으로 시작하는 정수 리터럴은 8진수(*octal*)를 의미합니다.
  8진수 `010`은 10진수로 변환하면 $1 \times 8 + 0 = 8$이 됩니다.
  
- **보안 포인트**: 리눅스/유닉스 권한 설정(예: `0755`)에서 8진수는 필수입니다.
  공격자는 개발자가 10진수로 착각하게 유도하여 의도치 않은 권한을 부여하거나 엉뚱한 네트워크 포트를 개방하게 만들 수 있습니다.
  숫자 앞의 `0`은 단순한 장식이 아님을 명심하십시오.

## [실습 8] 부호의 방향: 음수 나머지 연산과 메모리 침범
**난이도**: ★★★☆☆ | **관련 이론**: 5.6.3 int의 나머지 (%)

다음은 원형 버퍼(*circular buffer*)에서 위치를 계산하는 코드입니다. 
`index`의 최종 값은 무엇입니까?

```c
int main(void) {
        int position = -10;
        int buffer_size = 3;
        
        /* 역방향으로 이동했을 때의 인덱스 계산 */
        int index = position % buffer_size;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **-1** (1이나 2가 아님)

- **해석**: C99 표준 이후 나머지 연산의 부호는 항상 왼쪽 피연산자(나누어지는 수)의 부호를 따릅니다.
  `-10`이 음수이므로 결과도 `-1`이 나옵니다.

- **보안 포인트**: 만약 이 `index` 값을 원형 버퍼에서의 데이터 위치로 사용한다면 `buffer[-1]`에 접근하게 됩니다.
  이는 허가받지 않은 메모리 영역을 건드리는 경계 외 접근(*out of bounds*) 취약점으로 이어지며, 시스템을 파괴하거나 정보를 탈취하는 통로가 됩니다.

## [실습 9] 욕심쟁이 컴파일러: 맥시멀 먼치(Maximal Munch)
**난이도**: ★★★★★ | **관련 이론**: 3.2 토큰 분리 규칙, 5.6.6 단항 연산자

다음 코드는 컴파일러가 어떻게 '토큰'으로 쪼개느냐에 따라 `result` 값이 결정됩니다.

```c
int main(void) {
        int a = 5;
        
        /* 빼기 기호 사이에 공백이 있음 */
        int result = - -a; 
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **5**

- **해석**: 
	1. **토큰 분리**: 중간에 공백이 있어 컴파일러는 이를 `--`(감소 연산자)가 아닌 두 개의 단항 마이너스 연산자(`-`)로 인식합니다. 
	2. **우선순위**: 단항 연산자는 오른쪽에서 왼쪽으로 결합하므로, `-(-5)`가 되어 결과는 `5`가 됩니다. 

- **보안 포인트**: 공백 없이 `--a`라고 썼다면 맥시멀 먼치(*maximal munch*) 원칙에 따라 컴파일러는 가능한 한 긴 기호인 `--`를 하나의 토큰으로 해석했을 것입니다.
  난독화된 악성 코드는 이처럼 공백 한 칸으로 연산의 의미를 뒤바꾸어 분석가를 혼란에 빠뜨립니다.

## [실습 10] 대입의 연쇄: 오른쪽에서 왼쪽으로의 흐름
**난이도**: ★★☆☆☆ | **관련 이론**: 5.5.2 대입 연산의 결합 방향

변수 초기화 루틴에서 흔히 볼 수 있는 패턴입니다. `x`와 `y`에는 최종적으로 어떤 값이 저장될까요?

```c
int main(void) {
        int x = 0;
        int y = 5;
        
        /* 대입 연산자가 연달아 등장 */
        x = y = 100;
        
        return 0;
}
```

### [교수의 해설]
- **정답**: **x는 100, y는 100**

- **해석**: 대입 연산자(`=`)는 다른 산술 연산과 달리 오른쪽에서 왼쪽(*right associative*)으로 진행됩니다.
	1. `y = 100`이 먼저 실행되어 `y`가 100이 됩니다.
	2. 이 수식의 결과값인 `100`이 다시 왼쪽으로 전달되어 `x = 100`이 실행됩니다.

- **보안 포인트**: 데이터의 흐름을 역추적(*reverse engineering*)할 때, 값이 언제 어느 방향으로 덮어씌워지는지(*overwrite*) 파악하는 것은 기본 중의 기본입니다.
  대입의 연쇄는 편리하지만, 코드의 명확성을 위해 보안 프로그래밍에서는 한 줄에 하나씩 명확히 적는 것을 권장하기도 합니다.
