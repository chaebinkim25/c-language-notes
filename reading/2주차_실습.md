## [실습 1] 트로이 목마: 등호의 실수
**난이도**: ★★★☆☆ | **관련 이론**: 6.3 관계 연산자

다음 코드는 관리자 권한을 체크하는 로그인 로직입니다. 이 코드가 실행되면 `access_status`는 어떤 상태가 될까요?

```c
int main(void)
{
        int user_level = 1;      /* 1: 일반 유저, 9: 관리자 */
        int access_status = 0;   /* 0: 차단, 1: 허용 */

        /* 관리자라면 접근 허용 */
        if (user_level = 9) {
                access_status = 1;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (허용됨)**
- **해석**: `user_level == 9`(비교)를 써야 할 곳에 `user_level = 9`(대입)를 쓴 치명적인 실수입니다. 
  1. `user_level` 변수에 `9`가 대입됩니다.
  2. 이 수식의 결과값은 대입된 값인 `9`가 됩니다.
  3. C 언어에서 **0**이 아닌 모든 값은 참 (*true*)으로 간주하므로 `if`문은 무조건 참이 되어 권한이 풀립니다.
- **보안 포인트**: 이를 권한 상승 (*privilege escalation*) 취약점이라고 합니다. 해커는 이런 단순한 오타를 이용해 시스템 관리자 권한을 탈취합니다. 비교할 때는 항상 등호가 두 개(`==`)인지 확인하십시오.

## [실습 2] 유령 문장: 세미콜론의 은신
**난이도**: ★★★☆☆ | **관련 이론**: 6.7.2 세미콜론의 함정

다음은 해킹 시도가 감지되었을 때 방어막을 가동하는 코드입니다. `defense_mode`는 최종적으로 어떤 값이 될까요?

```c
int main(void)
{
        int hacking_detected = 0; /* 0: 감지 안 됨 (안전) */
        int defense_mode = 0;     /* 0: 대기, 1: 방어 가동 */

        /* 해킹이 감지되면 방어 모드 가동 */
        if (hacking_detected == 1); 
        {
                defense_mode = 1;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (방어 가동)**
- **해석**: `if` 조건식 뒤에 찍힌 세미콜론(`;`)이 조건문을 강제로 종료시켰습니다.
  1. 컴파일러는 `if (hacking_detected == 1)`을 검사하고 결과가 거짓(**0**)임을 확인합니다.
  2. 하지만 바로 뒤의 `;`를 보고 "조건이 참일 때 실행할 문장이 없음 (*null statement*)"으로 해석하고 끝냅니다.
  3. 그 아래의 `{ ... }` 블록은 `if`문과 상관없는 독립된 문장이 되어 무조건 실행됩니다.
- **보안 포인트**: 이런 실수는 중요한 보안 검사를 무조건 통과 (*bypass*)하게 만듭니다. 조건문 괄호 뒤에는 절대 세미콜론을 찍지 마십시오.

## [실습 3] 믿는 도끼에 발등: 중괄호 생략과 들여쓰기
**난이도**: ★★★★☆ | **관련 이론**: 6.7.1 중괄호 생략

서버 상태가 `1` (정상)일 때, `log_count`는 어떻게 변할까요?

```c
int main(void)
{
        int server_state = 1;    /* 1: 정상, 0: 에러 */
        int connection = 1;      /* 1: 연결됨 */
        int log_count = 0;

        /* 에러 발생 시 처리 루틴 */
        if (server_state == 0)
                connection = 0;              /* 연결 끊기 */
                log_count = log_count + 1;   /* 에러 로그 기록 */

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (증가함)**
- **해석**: 애플 (*Apple*)의 유명한 'goto fail' 버그와 같은 유형입니다.
  1. C 언어에서 중괄호 `{}`가 없으면 `if`문은 바로 다음 **한 줄(문장)**만 지배합니다.
  2. `connection = 0;`은 실행되지 않지만, 들여쓰기 된 `log_count` 줄은 `if`문 밖의 문장이므로 무조건 실행됩니다.
- **보안 포인트**: 사람의 눈은 들여쓰기를 믿지만, 컴파일러는 중괄호만 믿습니다. 해커는 이 착시를 이용해 보안 검증을 건너뛰게 만듭니다. `if`문 뒤에는 반드시 `{}`를 사용하십시오.

## [실습 4] 돈 복사 버그: 입력값 검증 부재
**난이도**: ★★★★☆ | **관련 이론**: 6.6.3 범위 검사

출금액(`withdraw_amount`)에 `-500`을 입력했을 때, 잔액 `balance`는 어떻게 될까요?

```c
int main(void)
{
        int balance = 1000;         /* 잔액: 1000원 */
        int withdraw_amount = -500; /* 사용자가 입력한 값 */

        /* 잔액이 충분한지 확인 (음수 체크 누락) */
        if (balance >= withdraw_amount) {
                balance = balance - withdraw_amount;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1500**
- **해석**: 돈을 뺐는데 잔액이 늘어나는 논리적 결함(*logical flaw*)이 발생했습니다.
  1. `1000 >= -500`은 참이므로 `if` 블록이 실행됩니다.
  2. $1000 - (-500)$은 수학적으로 $1000 + 500$이 됩니다.
- **보안 포인트**: "입력값은 절대 믿지 않는다(*never trust user input*)"가 보안의 제1원칙입니다. `if (withdraw_amount > 0 && balance >= withdraw_amount)` 처럼 데이터의 범위를 반드시 검증해야 합니다.

## [실습 5] 진실 혹은 거짓: 음수의 참/거짓
**난이도**: ★★★★★ | **관련 이론**: 6.1 참과 거짓의 표현

에러 발생 시 `-1`을 반환하는 상황에서, `status`는 최종적으로 어떤 값이 될까요?

```c
int main(void)
{
        int error_code = -1; /* -1: 치명적 에러 발생 */
        int status = 0;      /* 0: 정상, 1: 비상 사태 */

        /* 에러 코드가 있으면(참이면) 비상 사태 선포? */
        if (error_code) {
                status = 1;
        } else {
                status = 0;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (비상 사태)**
- **해석**: C 언어에서 거짓은 오직 **0** 뿐이며, **0**이 아닌 모든 수(음수 포함)는 참(*true*)으로 간주됩니다. 따라서 `-1`은 참으로 평가됩니다.
- **보안 포인트**: 어떤 함수는 성공 시 **0**을 반환하기도 합니다. 단순히 `if (result)`라고만 쓰면 성공과 실패를 반대로 해석할 위험이 있습니다. 보안을 위해서는 `if (error_code != 0)`처럼 명시적으로 비교하는 것이 안전합니다.

## [실습 6] 수학의 배신: 범위 검사의 착각
**난이도**: ★★★★★ | **관련 이론**: 6.3 관계 연산자

다음은 사용자의 나이(`age`)가 **10**대인지 (**10**보다 크고 **20**보다 작은지) 확인하는 로직입니다. `age`가 **5**살일 때, `is_teenager`는 어떤 값이 될까요?

```c
int main(void)
{
        int age = 5;
        int is_teenager = 0;

        /* 10 < age < 20 인지 검사 */
        if (10 < age < 20) {
                is_teenager = 1;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (참, 10대라고 판단함)**
- **해석**: 수학에서는 `10 < 5 < 20`이 거짓이지만, C 언어는 이를 왼쪽에서 오른쪽으로 순차적으로 계산합니다.
  1. `10 < age`(10 < 5)를 먼저 계산합니다. 결과는 거짓이므로 **0**이 됩니다.
  2. 남은 식은 `0 < 20`이 됩니다. 이는 참이므로 최종 결과는 **1**이 됩니다.
  3. 결과적으로 나이가 **5**살이든 **100**살이든 이 조건문은 항상 참이 됩니다.
- **보안 포인트**: 입력값 범위를 검증할 때 이런 수학적 표기법을 쓰면, 모든 비정상 데이터가 보안 검색대를 통과해 버립니다. 반드시 `(10 < age) && (age < 20)` 처럼 논리 연산자를 사용하여 조건을 분리해야 합니다.

## [실습 7] 코드의 망령: 규칙 그림자 (Rule Shadowing)
**난이도**: ★★★★☆ | **관련 이론**: 6.5 else if (다중 선택)

해커의 *IP* (`ip_type = 9`)가 접속했을 때, 시스템은 어떻게 반응할까요?

```c
int main(void)
{
        int ip_type = 9;       /* 1: 관리자, 9: 해커, 그외: 일반 유저 */
        int access_granted = 0;

        if (ip_type > 0) {     /* 규칙 A: 양수 IP는 모두 정상으로 간주 */
                access_granted = 1;
        }
        else if (ip_type == 9) { /* 규칙 B: 해커는 차단 */
                access_granted = 0;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (접속 허용)**
- **해석**: `else if`는 앞선 조건이 거짓일 때만 실행됩니다.
  1. `ip_type > 0` ($9 > 0$)은 참입니다. 첫 번째 블록이 실행되어 접속이 허용됩니다.
  2. 첫 번째 조건이 참이었으므로, 뒤에 있는 `else if (ip_type == 9)`는 검사조차 하지 않고 건너뜁니다.
- **보안 포인트**: 이를 규칙 그림자(*rule shadowing*)라고 합니다. 포괄적인 허용 규칙이 구체적인 차단 규칙보다 위에 있으면, 차단 규칙은 영원히 실행되지 않습니다 (*unreachable*). 보안 정책은 항상 가장 구체적인 차단 규칙을 먼저 배치해야 합니다.

## [실습 8] 소 잃고 외양간 고치기: 오버플로우 사후 검사
**난이도**: ★★★★★ | **관련 이론**: 6.6.6 오버플로우 예방

포인트가 최댓값인 상태에서 **100**점을 더 적립하려 합니다. 과연 안전하게 차단될까요?

```c
int main(void)
{
        int MAX = 2147483647; 
        int current_point = 2147483647;
        int add_point = 100;

        /* 더한 값이 최댓값을 넘는지 검사 */
        if (current_point + add_point > MAX) {
                /* 적립 차단 로직 */
        } else {
                current_point = current_point + add_point;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **차단되지 않고 else 블록이 실행되어 *undefined behvaior*가 발생합니다.**
- **해석**:
  1. `if` 조건식 안의 `current_point + add_point`가 먼저 계산됩니다.
  2. 덧셈 순간 이미 오버플로우가 발생하여 결과는 아주 작은 음수가 됩니다.
  3. `음수 > MAX`는 거짓이므로 `if`문을 통과하고, `else` 블록에서 깨진 값이 저장됩니다.
- **보안 포인트**: 오버플로우가 발생한 뒤에 검사하는 것은 의미가 없습니다. 반드시 더하기 전에 검사해야 합니다. 올바른 검사식은 `if (add_point > MAX - current_point)` 처럼 수식을 이항하여 뺄셈으로 검증하는 것입니다.

## [실습 9] 거꾸로 된 거짓말: 부정확한 불리언 판단
**난이도**: ★★★☆☆ | **관련 이론**: 6.1 참/거짓 표현

시스템 에러(**-1**)가 발생했을 때, 아래 코드는 어떻게 동작할까요?

```c
int main(void) {
        int check_result = -1;         /* -1: 시스템 에러 발생 */
        int virus_detected = 0;

        /* 0(안전)이 아니면 바이러스로 판단? */
        if (check_result) { 
                virus_detected = 1;
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **1 (바이러스 감지됨)**
- **해석**: 개발자는 양수(바이러스 ID)만 참이 될 것이라 기대했을 수 있습니다. 하지만 C 언어에서 **0**이 아닌 모든 수, 즉 음수 **-1**도 참(*true*)으로 간주됩니다.
- **보안 포인트**: 이 코드는 시스템 에러(**-1**)를 바이러스 발견으로 오인하게 만듭니다. 반대로 성공 시 **0**을 반환하는 함수들을 다룰 때 이런 실수는 치명적입니다. 숫자의 의미를 문맥에 맞게 파악하고, `if (check_result > 0)` 처럼 명시적으로 비교해야 안전합니다.

---

[교수의 마무리] 자, 이제 총 9가지의 실습을 통해 여러분은 컴파일러가 코드를 해석하는 방식이 인간의 직관과 얼마나 다른지 뼈저리게 느꼈을 것입니다.

- 수학적으로 당연한 `10 < x < 20`이 통하지 않고,
- 순서만 바꿨을 뿐인 `else if`가 보안 구멍을 만들며,
- 검사를 하겠다고 쓴 `+` 연산이 오히려 값을 망가뜨립니다.
 
이것이 바로 해커들이 노리는 생각의 틈입니다. 다음 시간에는 이 불안정한 데이터의 흐름을 제어하여, 원하는 만큼 코드를 반복 실행시키는 7장 반복문(`for`)으로 넘어가겠습니다. 그 전까지 오늘 배운 망가진 코드들을 안전한 코드로 고치는 연습을 충분히 해두시기 바랍니다.

---

신입생 여러분, 아주 정확하고 날카로운 지적입니다.

지금까지 우리가 본 것은 `if`문 주변의 환경(변수, 세미콜론, 들여쓰기)이 `if`문을 방해하는 상황들이었습니다. 하지만 보안 전문가라면 `if`문 그 자체의 구조적 허점과 논리적 흐름(*flow*)이 어떻게 꼬일 수 있는지 꿰뚫어 봐야 합니다.

여러분의 피드백을 반영하여, `if`문의 구조적 모호성과 논리 연산의 실행 순서를 직접 타격하는 가장 중요한 실습 3가지를 추가로 배정하겠습니다. 이것들은 C 언어 문법책에서는 주의하라고만 하고 넘어가지만, 보안 사고 분석에서는 단골로 등장하는 소재들입니다.

## [실습 10] 낙동강 오리알: 짝 잃은 `else` (*dangling else*)
**난이도**: ★★★★☆ | **관련 이론**: 6.7.3 `else`의 결합 규칙

다음은 관리자 (*admin*)가 로그인했을 때 비밀번호를 체크하는 코드입니다. 일반 유저 (*user*)가 접속하면 `access_log`는 어떻게 기록될까요?

```c
int main(void) {
        int is_admin = 0;           /* 0: 일반 유저 */
        int password_ok = 0;
        int access_log = 0;         /* 0: 기록 없음, 1: 접근 거부 기록 */

        if (is_admin)
                if (password_ok)
                        perform_admin_task();
        else
                access_log = 1;     /* "관리자가 아닌 경우"를 의도하고 작성함 */

        return 0;
}
```

### [교수의 해설]
- **정답**: **access_log는 0 (기록되지 않음)**
- **해석**: 개발자는 `else`가 첫 번째 `if (is_admin)`와 짝이라고 생각하고 들여쓰기를 했습니다.
	1. 하지만 C 언어 표준에서 `else`는 **가장 가까이 있는 짝 없는 `if`**와 결합합니다.
	2. 즉, 이 `else`는 `if (password_ok)`의 짝이 되어버립니다.
	3. `is_admin`이 0이므로, 바깥쪽 `if`문이 거짓이 되어 안쪽의 모든 코드(else 포함)가 통째로 실행되지 않습니다.
- **보안 포인트**: 이를 *dangling else* 문제라고 합니다. 이로 인해 '접근 거부 로그'가 남지 않게 되어, 해커가 마음 놓고 무차별 대입 공격 (*brute force*)을 시도할 수 있는 환경이 만들어집니다. 중괄호 `{ }`가 이래서 중요합니다.

## [실습 11] 게으른 문지기: 단축 평가 (Short-Circuit)
**난이도**: ★★★★★ | **관련 이론**: 6.2 논리 연산자(`&&`), 단축 평가

다음은 패킷이 안전한지 검사하고(`is_safe`), 그 기록을 로그 함수(`log_check`)로 남기는 코드입니다. 위험한 패킷(`is_safe = 0`)이 들어왔을 때 `log_count`는 증가할까요?

```c
int main(void) {
        int is_safe = 0;      /* 0: 위험한 패킷 */
        int log_count = 0;    /* 로그 기록 횟수 */

        /* 안전하고(AND), 로그 기록도 성공해야 통과 */
        if (is_safe == 1 && (log_count = log_count + 1)) {
                process_packet();
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **0 (증가하지 않음)**
- **해석**: C 언어의 논리 연산자(`&&`, `||`)는 효율성을 위해 단축 평가 (*short-circuit evaluation*)를 합니다.
  1. `A && B`에서 `A`가 거짓이면, `B`는 쳐다보지도 않고 전체를 거짓으로 확정합니다.
  2. `is_safe == 1`이 거짓이므로, 뒤에 있는 `log_count` 증가 코드는 아예 실행되지 않습니다.
- **보안 포인트**: 조건문 안에서 함수를 실행하거나 변수를 바꾸지 마라는 격언이 있는 이유입니다. 해커가 위험한 패킷을 보내면 앞쪽 조건이 거짓이 되어 뒤쪽의 감사 로그 (*audit Log*) 기능이 작동하지 않게 됩니다. 공격을 했는데 로그가 남지 않는 유령 공격 (*log evasion*)이 가능해집니다.

## [실습 12] 절대 닫히지 않는 문: 논리의 오류 (*tautology*)
**난이도**: ★★★★☆ | **관련 이론**: 6.2 논리 연산자(`||`), 드모르간의 법칙

다음은 위험한 확장자(`.exe`, `.bat`)를 가진 파일 업로드를 차단하는 블랙리스트 필터입니다. `.exe` 파일을 업로드하면 차단될까요?

```c
int main(void) {
        int file_type = 1; /* 1: .exe 파일, 2: .bat 파일, 3: .jpg 파일 */
        int block = 0;

        /* .exe가 아니거나 .bat가 아니면 통과? */
        if (file_type != 1 || file_type != 2) {
                block = 0; /* 통과 */
        } else {
                block = 1; /* 차단 */
        }

        return 0;
}
```

### [교수의 해설]
- **정답**: **0 (통과됨, 차단 실패)**
- **해석**: 이 조건문은 항상 참 (*always true)*인 항진명제입니다.
  1. `file_type`이 ****(.exe)이라고 가정해 봅시다.
  2. `1 != 1`은 거짓이지만, `1 != 2`는 참입니다.
  3. `||` (OR) 연산자는 둘 중 하나만 참이면 참이므로, 결과는 참이 되어 통과합니다.
  4. 어떤 숫자를 넣어도 이 조건문은 항상 참이 됩니다.
- **보안 포인트**: 블랙리스트 필터링을 짤 때 가장 많이 하는 논리 실수입니다. 개발자의 의도는 "exe도 아니고 bat도 아니어야 한다"였을 것입니다. 그렇다면 `&&` (AND)를 쓰거나 `!((type == 1) || (type == 2))`로 작성했어야 합니다. 이 코드는 모든 악성 파일을 프리패스로 통과시킵니다.
