반갑습니다, 사이버 보안학과 신입생 여러분.

지금까지 우리는 데이터를 담는 변수 `int`, 흐름을 통제하는 조건문 `if`, 
그리고 지치지 않는 자동화 엔진인 반복문 `for`를 배웠습니다. 
오늘 다룰 8장 배열 _array_ 는 수십, 수만 개의 데이터를 한 번에 다루기 위해 같은 타입의 상자를 일렬로 늘어놓은 
거대한 선반과 같습니다. 
`for`라는 강력한 엔진이 이 선반 레일을 타고 지나가며 엄청난 양의 데이터를 순식간에 처리하게 됩니다.

하지만 보안 관점에서 C 언어의 배열은 가장 위험하고 치명적인 무기이기도 합니다. 
다른 현대 언어들과 달리, C 언어의 배열에는 끝을 알려주는 안전바 _stopper_ 나 콘크리트 벽이 없기 때문입니다. 
오늘 읽기 수업에서는 1장부터 8장까지의 지식을 총동원하여, 
이 종잇장처럼 얇은 배열의 경계를 해커들이 어떻게 무너뜨리는지 코드로 확인해 보겠습니다.

## 1부: 배열의 구조적 취약점과 초기화의 함정

### \[실습 1\] 안전바 없는 롤러코스터: 배열 경계 침범 _buffer overflow_
- **난이도:** ★★★★★
- **관련 이론:** 8.1 배열과 반복문, 8.2 배열의 인덱스, 7.3 for문의 부등호

다음은 3개의 데이터를 저장할 수 있는 배열을 만들고, for문을 통해 값을 1로 채우는 코드입니다. 
이 코드를 실행하면 어떤 끔찍한 일이 벌어질까요?

```c
#define SIZE 3

int main(void)
{
        int i;
        int data[SIZE] = {0};
        
        /* 배열에 값을 채우는 반복문 */
        for (i = 0; i <= SIZE; i = i + 1) {
                data[i] = 1;
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** 존재하지 않는 `data` 공간인 남의 메모리 영역을 침범하여 값을 덮어쓰게 됩니다 _buffer overflow_.
- **해석:** C 언어에서 배열의 인덱스는 순서가 아니라 시작점에서 떨어진 거리 _offset_ 을 뜻합니다.
  따라서 크기가 3인 배열의 방 번호는 0, 1, 2까지만 존재합니다.
  하지만 `for`문의 조건이 `i <= SIZE`로 되어 있어, `i`가 3일 때도 반복문이 실행됩니다.
- **보안 포인트:** `data[3]`은 내 배열의 끝을 넘어선 허가되지 않은 남의 땅입니다.
  엔진이 레일 끝에서 멈추지 않고 옆집 선반을 부수고 지나간 격입니다.
  해커들은 이 오프 바이 원 _off-by-one_ 실수를 이용해 다른 변수를 조작하거나 악성 코드를 실행합니다.
  배열을 다루는 `for`문에서는 습관적으로 `<=` 대신 `<`를 써야 합니다.

### \[실습 2\] 숨겨진 과거의 폭로: 초기화되지 않은 배열
* **난이도:** ★★★★☆
* **관련 이론:** 8.3 배열 선언, 8.4 배열 초기화

다음 코드는 서버가 사용자의 접속 로그를 담기 위해 임시 배열을 만든 상황입니다. 
`logs` 배열 안에는 어떤 값이 들어있을까요?

```c
int main(void)
{
        /* 5칸짜리 배열 선언 (초기화 누락) */
        int logs[5]; 
        int i;
        
        /* logs 배열의 내용을 외부로 전송하는 로직 (생략) */
        
        return 0;
}
```

#### \[해설\]
- **정답:** 이전에 다른 프로그램이나 함수가 쓰다 남긴 알 수 없는 값 _garbage value_ 들이 들어있습니다.
- **해석:** 배열을 선언해서 대여만 하고 값을 초기화해서 지우지 않았기 때문입니다.
  일반 변수 1개라면 쓰레기 값 1개가 남겠지만, 배열은 그 크기만큼 엄청난 양의 이전 데이터가 그대로 남아있게 됩니다.
- **보안 포인트:** 이것은 매우 심각한 정보 유출 _information leak_ 취약점입니다.
  만약 이전에 이 메모리 공간을 쓰던 프로그램이 '관리자 비밀번호'나 '암호화 키'를 저장했었다면,
  그 내용이 고스란히 `logs` 배열을 통해 해커에게 전송될 수 있습니다.
  반드시 `int logs[5] = {0};` 처럼 배열 전체를 `0`으로 초기화하는 습관을 들여야 합니다.

### \[실습 3\] 착각의 늪: 부분 초기화의 진실
- **난이도:** ★★★☆☆
- **관련 이론:** 8.4 배열 초기화, 4.2 리터럴

개발자가 5칸짜리 시스템 설정 배열을 모두 참 또는 활성화를 의미하는 1 상태로 꽉 채우고 싶어서 
다음과 같이 코드를 작성했습니다. 배열의 5칸은 모두 1일까요?

```c
int main(void)
{
        /* 개발자의 의도: 전부 1로 채우기 */
        int settings[5] = {1}; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 첫 번째 칸 `settings[0]`만 1이고, **나머지 4칸은 모두 0**이 됩니다.
- **해석:** C 언어의 배열 초기화 규칙에 따르면, 칸 수보다 적은 값을 적어 넣을 경우
  값을 지정하지 않은 나머지 칸은 전부 자동으로 0이 됩니다.
- **보안 포인트:** 이런 코드는 다른 프로그래머가 보았을 때 개발자의 진짜 의도가 전부 1인지, 첫 칸만 1인지를
  알 수 없게 만듭니다. 자동차 소프트웨어 표준 MISRA C에서도 이런 부분 초기화를 엄격히 금지합니다.
  전부 1로 채우고 싶다면 `{1, 1, 1, 1, 1}`처럼 모든 값을 명시적으로 적어야 안전합니다.

### \[실습 4\] 뜬금없는 세미콜론의 역습: 매크로 상수
- **난이도:** ★★★☆☆
- **관련 이론:** 8.6 매크로 상수 `#define`, 2.5 문장과 세미콜론

배열의 크기를 쉽게 관리하기 위해 매크로 상수 `#define`을 사용했습니다. 
코드를 컴파일하면 엉뚱한 곳에서 문법 에러가 발생합니다. 원인이 무엇일까요?

```c
/* 최대 접속자 수를 10으로 정의 */
#define MAX_USERS 10; 

int main(void) {
        /* 에러 발생 지점 */
        int active_users[MAX_USERS]; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** `#define` 끝에 붙인 세미콜론 `;` 때문입니다.
- **해석:** `#define`은 C 언어의 일반적인 문장이 아니라,
  컴파일 전에 글자를 그대로 바꿔치기하는 '전처리 지시문'입니다.
  따라서 위 코드는 컴퓨터의 눈에 `int active_users[10;];` 이라는 기괴한 코드로 변환됩니다.
  대괄호 `[` 안에 뜬금없이 세미콜론이 들어가서 문법이 파괴된 것입니다.
- **보안 포인트:** 보안을 위해 하드코딩된 숫자 리터럴 대신 의미 있는 매크로 상수 `MAX_USERS`를 사용하는 것은
  매우 좋은 습관입니다.
  하지만 C 언어 문장의 끝에 항상 세미콜론을 붙이던 버릇을 전처리 지시문에까지 적용하면 안 됩니다.

### \[실습 5\] 편리함의 대가: 가변 길이 배열 _VLA_ 의 시한폭탄
* **난이도:** ★★★★★
* **관련 이론:** 8.3 현대 C의 유혹 _VLA_

다음 코드는 네트워크를 통해 들어오는 데이터의 크기 `packet_size`를 먼저 읽은 뒤, 
딱 그 크기만큼만 배열을 동적으로 생성하려는 코드입니다. 
코드가 품고 있는 치명적인 위험은 무엇일까요?

```c
int main(void) {
        /* 외부(해커)가 조작할 수 있는 크기 입력값이라고 가정 */
        int packet_size = 100000000; 
        
        /* 입력된 크기만큼 배열 생성 VLA */
        int buffer[packet_size]; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 해커가 `packet_size` 값을 터무니없이 크게 보낼 경우,
  한정된 작업 메모리가 고갈 _stack overflow_ 되어 프로그램이 즉시 뻗어버립니다.
- **해석:** 변수로 배열의 크기를 정하는 것을 가변 길이 배열 _VLA_ 이라고 합니다.
  짓는 순간 방 개수를 정할 수 있어 무척 편해 보이지만, 변수 상자가 만들어지는 작업 공간은 크기가 매우 작습니다.
- **보안 포인트:** 보안 전문가들은 이 _VLA_ 를 시한폭탄 취급합니다.
  외부에서 입력된 값을 그대로 배열 크기에 사용하면 가용성 침해 _DoS_ 공격에 무방비로 노출되며,
  다른 변수의 값을 덮어쓸 위험까지 존재합니다.
  또한, _VLA_ 는 앞서 배운 `{0}` 초기화 문법조차 사용할 수 없어 쓰레기 값 문제에도 취약합니다.
  보안이 중요한 시스템 _CERT_, _MISRA C_ 에서는 _VLA_ 사용을 엄격히 금지하고 있습니다.

> **\[과제 및 당부 - 1부\]**
> 오늘은 C 언어가 제공하는 빠르고 강력한 도구인 배열의 민낯을 보았습니다.
> C 언어는 개발자에게 무한한 자유를 주지만,
> 선반의 끝이 어디인지, 그 안에 남이 쓰던 데이터가 남아있는지는 절대 대신 챙겨주지 않습니다.
> 오늘 배운 5가지 취약점 패턴 _OOB_, 정보 유출, 초기화 착각, 매크로 오타, _VLA_ 폭탄을 다시 한번 복습하고,
> 어떻게 하면 이를 막을 수 있는지 `<` 사용, `{0}` 사용 등 스스로 안전한 코드로 수정해 보길 바랍니다. 수고하셨습니다!

## 2부: 배열과 제어문의 결합이 낳는 런타임 재앙

보안의 본질을 파고드는 여러분의 끈질긴 탐구 자세, 아주 훌륭합니다. 

앞서 살펴본 5가지 예제가 배열의 문법적 허점과 생성 방식의 위험성을 경고했다면, 

이번에 추가할 4가지 실습은 배열이 조건문 `if`, 반복문 `for`와 결합할 때 발생하는 치명적인 런타임 오류에 집중하겠습니다.

특히 인덱스를 잘못 다루었을 때 보안 방어선이 어떻게 무너지는지, 컴퓨터의 눈으로 코드를 추적해 보십시오.

### \[실습 6\] 1번 방의 착각: 시작점의 오해와 경계 침범
- **난이도:** ★★★★☆
- **관련 이론:** 8.2 배열의 인덱스, 8.7 `for` + 배열 기본 패턴

초보 개발자가 3개의 데이터가 든 배열의 모든 값을 더하기 위해 코드를 짰습니다. 
코드가 실행되면 `sum`에는 정상적인 총합 60이 들어갈까요?

```c
#define SIZE 3

int main(void)
{
        int i;
        int data[SIZE] = {10, 20, 30};
        int sum = 0;
        
        /* 1번째 값부터 3번째 값까지 더하기? */
        for (i = 1; i <= SIZE; i = i + 1) {
                sum = sum + data[i];
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** 정상적인 총합이 나오지 않으며, 메모 경계를 침범 _out-of-bounds_ 하는 보안 사고가 발생합니다.
- **해석:** 많은 초보자가 일상적인 감각으로 첫 번째니까 1번 방이라고 생각하여 `i`를 1부터 시작합니다.
  하지만 C 언어에서 인덱스는 시작점에서 떨어진 거리 `offset`를 뜻하므로 첫 번째 데이터는 `data[0]`에 있습니다.
  결과적으로 이 코드는 `data[1]` 20과 `data[2]` 30을 더한 뒤,
  존재하지 않는 남의 땅인 `data[3]`에 접근하여 쓰레기 값을 더해버립니다.
- **보안 포인트:** 이런 코드는 데이터를 빼먹는 논리 오류를 넘어,
  할당되지 않은 메모리 영역을 읽어버리는 버퍼 오버리드 _buffer-over-read_ 취약점을 유발합니다.
  배열을 다룰 때는 항상 0부터 시작해서 n-1에서 끝난다는 주문을 외워야 하며,
  for문 조건에서는 습관적으로 `<=` 대신 `<`를 써야 합니다.

### \[실습 7\] 해커에게 내어준 내비게이션: 인덱스 검증 누락
- **난이도:** ★★★★★
- **관련 이론:** 8.5 배열에 값 저장하기, MISRA C 규칙 4.14

다음은 사용자가 입력한 번호 `user_index`에 해당하는 시스템 설정을 `1`로 활성화하는 코드입니다. 
코드는 해커에게 어떤 방식으로 악용될 수 있을까요?

```c
#define SIZE 5

int main(void)
{
        int settings[SIZE] = {0};
        
        /* 해커가 외부에서 조작하여 -10 또는 100을 넣었다고 가정 */
        int user_index = 100; 
        
        /* 입력받은 위치의 설정을 1로 덮어쓰기 */
        settings[user_index] = 1;
        
        return 0;
}
```

#### \[해설\]
- **정답:** 배열의 크기 `5`를 아득히 벗어난 메모리 위치에 `1`을 덮어써서 시스템을 파괴하는 버퍼 오버플로우
  _buffer overflow_ 공격이 발생합니다.
- **해석:** C 언어의 기숙사 배열에는 벽이 종잇장처럼 얇고 끝을 알려주는 안전바가 없습니다.
  인덱스로 `100`을 주든, 음수 `-10`을 주든, 엔진은 멈추지 않고 옆집 선반을 부수고 지나가서 값을 `1`로 바꿔버립니다.
- **보안 포인트:** 만약 저 `settings[100]` 위치가 시스템의 관리자 권한을 나타내는 변수나
  중요한 함수가 실행되는 위치였다면, 해커는 이 단 한 줄의 코드로 시스템 전체를 장악할 수 있습니다.
  사용자 입력 등 외부 소스에서 온 데이터가 배열의 인덱스로 쓰일 때는 반드시 `if`문을 사용하여 인덱스가 `0`보다 크거나
  같고 `SIZE`보다 작은지 상한선과 하한선 유효성을 검사해야 합니다 _MISRA C Rule 4.14_.

### \[실습 8\] 반복이 끝난 후의 유령: 루프 변수 재사용과 OOB
- **난이도:** ★★★★☆
- **관련 이론:** 7.4 반복이 끝난 뒤의 상황, 8.5 배열에 저장된 값 쓰기

다음은 배열에서 특정 비밀번호 `target`을 찾다가, 끝내 발견하지 못하면 새로운 비밀번호를 배열에 추가하려 한 코드입니다. 
어떤 끔찍한 결함이 숨어 있을까요?

```c
#define SIZE 3

int main(void)
{
        int i;
        int passwords[SIZE] = {1111, 2222, 3333};
        int target = 9999; /* 배열에 없는 값 */
        
        /* 배열 안에서 target 찾기 */
        for (i = 0; i < SIZE; i = i + 1) {
                if (passwords[i] == target) {
                        break; /* 찾으면 루프 중단 */
                }
        }
        
        /* 배열을 다 뒤졌는데도 없었다면, 그 위치에 새로 추가? */
        passwords[i] = target;
        
        return 0;
}
```

#### \[해설\]
- **정답:** 검색에 실패하여 루프가 끝났을 때 `i`의 값은 `3`이므로,
  `passwords[3] = 9999;`가 실행되어 메모리 경계를 침범 _out-of-bounds_ 합니다.
- **해석:** 7장에서 배웠듯, `for`문 반복이 조건식 `i < SIZE`을 만족하지 못하고 끝났을 때
  `i`의 값은 마지막 원소의 인덱스인 2가 아니라, 조건을 거짓으로 만든 3입니다.
  따라서 루프 밖에서 무심코 `passwords[i]`에 접근하는 것은 치명적입니다.
- **보안 포인트:** 개발자는 변수 `i`의 상태를 착각하여 논리적 버그를 만들었습니다.
  데이터를 추가하려면 기존 배열의 크기가 꽉 차지 않았는지 먼저 확인하고,
  배열의 안전한 인덱스 안에서만 값을 조작해야 합니다.
  C89처럼 변수 유효 범위 _scope_ 가 넓을 때 가장 흔히 발생하는 보안 사고 중 하나입니다.

### \[실습 9\] 편리함의 함정 2: _VLA_ 와 초기화의 충돌
- **난이도:** ★★★☆☆
- **관련 이론:** 8.3 _VLA_ 의 위험성, 8.4 배열 초기화

외부 입력 크기만큼 배열을 만드는 가변 길이 배열 _VLA_ 를 쓰면서, 
보안을 지키기 위해 모든 값을 0으로 안전하게 초기화하려고 합니다. 코드는 어떻게 될까요?

```c
int main(void)
{
        int user_size = 10;
        
        /* 가변 길이 배열을 만들면서 동시에 0으로 안전하게 초기화 시도 */
        int secure_buffer[user_size] = {0};
        
        return 0;
}
```

#### \[해설\]
- **정답:** 이 코드는 컴파일 에러가 발생하여 아예 실행조차 되지 않습니다.
  _VLA_ 는 초기화 문법 `{0}`을 사용할 수 없기 때문입니다.
- **해석:** 크기를 유연하게 정할 수 있는 _VLA_ 는 얼핏 편리해 보이지만,
  C 언어 문법상 `{0}`을 이용한 초기화가 원천적으로 차단되어 있습니다.
  에러를 없애려고 `{0}`을 지워버리면,
  배열 안에는 이전 프로그램이 쓰다 버린 중요한 정보가 고스란히 쓰레기 값으로 남아있게 됩니다.
- **보안 포인트:** 보안 전문가들이 _VLA_ 를 '시한폭탄' 취급하며 MISRA C 표준에서도
  사용을 엄격히 금지하는 이유가 바로 이것입니다.
  가용성을 해치는 스택 오버플로우 _stack overflow_ 위험뿐만 아니라,
  초기화를 못 해 발생하는 정보 유출 _information leak_ 취약점까지 감수해야 하기 때문입니다.

> 루프가 끝난 뒤의 인덱스를 쓰든, _VLA_ 를 초기화하지 않고 쓰든, 결코 브레이크를 잡아주지 않습니다.
