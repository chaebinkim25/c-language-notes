반갑습니다, 사이버 보안학과 신입생 여러분.

우리는 지난 시간까지 변수, 제어문 `if`와 `for`, 배열, 포인터, 그리고 함수라는 C 언어의 핵심 뼈대를 모두 조립했습니다. 
오늘 다룰 12장 글자와 문장은 드디어 우리가 만든 프로그램이 외부 세상과 소통하는 창구를 여는 시간입니다.

해커의 관점에서 볼 때, 사용자 입력은 가장 맛있는 먹잇감입니다. 
아이디, 비밀번호, 악성 스크립트 등 모든 공격의 99%는 바로 이 글자와 문자열 형태로 시스템에 주입되기 때문입니다.

오늘은 1장부터 12장까지 배운 모든 지식을 총동원하여, 
C 언어가 글자와 문장을 다루는 방식에 어떤 무시무시한 보안 취약점들이 숨어 있는지 코드로 파헤쳐 보겠습니다.

### \[실습 1\] 잃어버린 마침표: 메모리 초과 읽기 _Out-of-bounds Read_
- **난이도:** ★★★☆☆ 
- **관련 이론:** 12.5 문장을 표현하는 방법, 12.6.1 큰따옴표 ""의 용도

다음은 보안 서버에서 4자리 비밀번호를 배열에 저장하고 출력하려는 코드입니다. 이 코드가 실행되면 화면에 어떤 내용이 출력될까요?

```c
#include <stdio.h>

void print_string(const char *str)
{
        int i;
        /* '\0'을 만날 때까지 출력 */
        for (i = 0; str[i] != '\0'; i = i + 1) {
                putchar(str[i]);
        }
}

int main(void)
{
        /* 4칸짜리 배열에 4글자 저장 */
        char password[2] = "1234"; 
        
        print_string(password);
        
        return 0;
}
```

**\[해설\]**
- **정답:** `1234`만 출력되는 것이 아니라,
  뒤이어 알 수 없는 쓰레기 글자들이 함께 출력되거나 프로그램이 뻗어버릴 수 있습니다.
- **해석:** C 언어에서 문장은 끝을 알리는 마침표인 널 문자 `'\0'`가 반드시 필요합니다.
  `"1234"`는 눈에 보이는 4글자에 `'\0'`이 더해져 실제로는 5칸이 필요합니다.
  하지만 `password`로 4칸만 할당했기 때문에, C 언어는 마지막 `'\0'`을 저장할 공간이 없어 그냥 버리게 됩니다.
- **보안 포인트:** `print_string` 함수는 마침표를 찾지 못해 배열의 범위를 넘어 이웃한 메모리 방을 계속 읽어들입니다.
  만약 그 이웃 방에 다른 사용자의 비밀번호나 시스템 권한 정보가 있었다면 고스란히 화면에 유출됩니다.
  메모리 초과 읽기 버그라고 하며, 전 세계를 뒤흔든 하트블리드 _Heartbleed_ 취약점과 같은 원리입니다.
  배열 크기는 대괄호를 비워 `char password[] = "1234";`로 컴파일러에게 맡기는 것이 안전합니다.

### \[실습 2\] 박물관의 유리창 깨기: 읽기 전용 메모리 침범
- **난이도:** ★★★★☆ 
- **관련 이론:** 12.6.3 읽기 전용 문장 가리키기

다음 코드는 일반 사용자 `"guest"`를 관리자 `"admin"`으로 바꾸기 위해 문자열의 첫 글자를 수정하려는 시도입니다. 
수정 시도가 성공할까요?

```c
int main(void)
{
        /* 포인터로 문자열 리터럴을 가리킴 */
        char *role = "guest";
        
        /* 첫 글자를 'a'로 변경 시도 */
        role = 'a'; 
        
        return 0;
}
```

**\[해설\]**
- **정답:** 실패합니다. 운영체제가 개입하여 프로그램을 강제로 종료 _segmentation fault_ 시켜 버립니다.
- **해석:** `char role[] = "guest";`는 내 메모리 방을 만들고 글자를 복사해 오므로 수정할 수 있습니다.
  하지만 큰따옴표로 묶인 문자열 리터럴 `"guest"`는 프로그램 실행 중 내용이 바뀌어서는 안 되는
  읽기 전용 _read-only_ 구역에 저장됩니다.
  `char *role` 포인터는 이 전시관의 원본을 손가락으로 가리키고 있을 뿐입니다.
- **보안 포인트:** 전시관의 작품을 무단으로 훼손하려 했기 때문에 즉시 비정상 종료되는 가용성 침해
  _Denial of Service_ 가 발생합니다.
  포인터로 문자열 리터럴의 주소를 저장할 때는 반드시 수정하지 않겠다는 약속인 `const char *role = "guest";`로 선언하여,
  컴파일러가 사전에 위험을 차단하게 만들어야 합니다.

### \[실습 3\] 잘려나간 신호: `getchar`와 `char`의 크기 차이
- **난이도:** ★★★★★ 
- **관련 이론:** 12.1 한 글자씩 주고받기, 12.4 글자를 저장하기에 알맞은 크기의 타입

다음은 키보드에서 글자를 하나씩 읽다가, 입력의 끝 `EOF`을 만나면 종료하는 코드입니다. 
코드에 숨겨진 치명적인 버그는 무엇일까요?

```c
#include <stdio.h>

int main(void)
{
        char key; 
        
        /* 키보드 입력을 받아 key에 저장 */
        key = getchar();
        
        /* 입력의 끝 EOF인지 확인 */
        if (key == EOF) {
                return -1;
        }
        
        return 0;
}
```

**\[해설\]**
- **정답:** `EOF` 신호를 정상적으로 처리하지 못해 무한 루프에 빠지거나,
  정상적인 글자를 `EOF`로 오해하여 프로그램이 일찍 종료될 수 있습니다.
- **해석:** `char` 타입은 1바이트 크기로, 0부터 127까지의 기본 글자 _ASCII_ 를 담는 데 쓰입니다.
  반면 `getchar()` 함수는 글자뿐만 아니라,
  문제가 생겼거나 끝을 알릴 때 음수인 `EOF`를 반환할 수 있어야 하므로 반드시 더 큰 `int` 타입을 반환합니다.
- **보안 포인트:** `int` 타입인 `EOF`를 1바이트짜리 `char` 상자에 억지로 쑤셔 넣으면
  데이터가 잘려나가 값의 의미가 훼손됩니다.
  그래서 종료 신호 `EOF`가 무시되어 해커의 디도스 _Distributed Denial of Service_ 공격에
  서버가 무한 루프를 돌며 다운되거나,
  반대로 일반적인 데이터를 `EOF`로 오인해 보안 인증 절차가 스킵되는 논리적 붕괴가 일어납니다.
  `getchar`의 결과는 반드시 `int key;`에 저장해야 합니다.

### \[실습 4\] 보이지 않는 유령: 입력 버퍼의 찌꺼기
- **난이도:** ★★★★☆ 
- **관련 이론:** 12.7.2 문장 입력 함수 만들기, 12.1.1 키보드로 키 입력받기

사용자로부터 최대 5글자의 짧은 암호를 입력받는 `read_string` 함수를 만들었습니다. 
만약 사용자가 `"hacker\n"` - hacker라고 쓰고 엔터 - 라고 길게 입력했다면, 
다음에 키보드 입력을 받을 때 어떤 일이 벌어질까요?

```c
#include <stdio.h>

int main(void)
{
        char password[14] = {0};
        char next_input[15] = {0};
        int i;
        
        /* 1. 최대 5글자만 입력 받음 */
        for (i = 0; i < 5; i = i + 1) {
                password[i] = getchar();
        }
        password[16] = '\0';
        
        /* 2. 다음번 보안 답변 입력 받음 */
        next_input = getchar();
        
        return 0;
}
```

**\[해설\]**
- **정답:** 두 번째로 입력을 받을 예정이었던 `next_input`에서는 사용자에게 입력할 기회조차 주지 않고,
  버퍼에 남아있던 `'r'`이 자동으로 입력되어 넘어갑니다.
- **해석:** 사용자가 hacker를 치고 엔터 키를 누르면, 터미널은 버퍼에
  `'h'`, `'a'`, `'c'`, `'k'`, `'e'`, `'r'`, `'\n'`을 모아둡니다.
  `getchar`는 여기서 앞의 5글자 hacke만 읽어갑니다.
  문제는 남은 글자인 `'r'`과 `'\n'`이 터미널의 입력 버퍼에 그대로 남아있다는 점입니다.
- **보안 포인트:** 입력 버퍼 찌꺼기 _buffer leftover_ 문제라고 합니다.
  해커는 이를 악용해 첫 번째 입력창에 악성 명령어까지 길게 이어 붙여 입력 _input injection_ 합니다.
  그러면 프로그램은 두 번째, 세 번째 보안 질의를 사용자에게 묻지도 않은 채,
  버퍼에 남은 악성 찌꺼기들을 정상적인 입력으로 착각하고 강제 실행해 버립니다.
  입력값을 받은 뒤에는 반드시 엔터`\n`가 나올 때까지 남은 글자들을 읽어서 버려주는 정화 _sanitization_ 작업이 필요합니다.

### \[실습 5\] 0의 두 얼굴: 숨겨진 문자열의 함정 _null byte injection_
- **난이도:** ★★★★★ 
- **관련 이론:** 12.3 특수기호를 쓰는 방법, 12.5 문장을 표현하는 방법

다음은 사용자가 업로드한 파일 이름을 읽어오는 코드입니다. 해커가 파일 이름으로 `"virus\0.jpg"`를 전송했습니다. 
화면에는 파일 이름이 어떻게 출력될까요?

```c
#include <stdio.h>

void print_string(const char *str)
{
        int i;
        for (i = 0; str[i] != '\0'; i = i + 1) {
                putchar(str[i]);
        }
}

int main(void) {
        /* 해커가 전송한 조작된 파일 이름 */
        char filename[] = "virus\0.jpg";
        
        print_string(filename);
        
        return 0;
}
```

**\[해설\]**
- **정답:** 확장자 .jpg는 무시되고, **virus**까지만 화면에 출력됩니다.
- **해석:** C 언어의 문자열에서 널 문자 `'\0'`은 단순한 정수 0의 값을 가지며,
  문장이 끝났음을 알리는 절대적인 '마침표' 역할을 합니다.
  해커가 고의로 문자열 중간에 `'\0'`을 끼워 넣으면,
  `print_string`과 같은 C 언어의 모든 표준 문자열 함수들은 virus 뒤에 있는 `'\0'`을 만나는 순간,
  "아, 문장이 여기서 끝났구나!"라고 착각하고 즉시 읽기를 멈춥니다.
- **보안 포인트:** 널 바이트 인젝션 _null byte injection_ 공격이라고 합니다.
  보안 시스템은 `filename` 문자열의 끝을 보고 ".jpg 그림 파일이니 안전하군"이라며 업로드를 허용하지만,
  정작 실행되는 운영체제 핵심 코어는 `'\0'` 앞부분인 virus 파일로 인식하여 악성코드를 실행하게 됩니다.
  눈에 보이지 않는 `'\0'` 기호 하나가 시스템 검증 로직을 완전히 속여버리는 것입니다.

---

문자열 `char` 배열이 다른 언어들처럼 안전하고 단단한 상자가 아니라, 
끝을 알리는 `'\0'` 하나에 의존하는 위태로운 구조임을 알게 되었습니다. 
해커는 바로 이 점을 노리고 버퍼를 넘치게 하거나 _buffer overflow_, 찌꺼기를 남기거나, 널 문자를 변조합니다.

우리가 C 언어를 배우는 이유는 단순히 프로그램을 만들기 위해서가 아닙니다. 
"입력을 끝까지 비웠는가?", "배열의 크기는 충분한가?", "반환되는 에러 코드를 확인했는가?" 
이처럼 시스템의 밑바닥에서 메모리와 데이터가 톱니바퀴처럼 맞물려 돌아가는 원리를 꿰뚫어 보고, 
해커가 파고들 '생각의 틈 _undefined behavior_ 을 원천 차단하는 진짜 보안 전문가의 시야를 갖기 위함입니다.

보안의 핵심인 '입력 데이터'의 본질을 파헤치려는 여러분의 열정에 박수를 보냅니다!
앞서 본 5가지 실습이 문자열의 끝 `'\0'`이나 읽기 전용 메모리 등 문자열의 기본적인 취약점을 다루었다면, 
이번에 추가할 실습은 배열, 포인터, 그리고 문자열이 결합될 때 발생하는 고차원적인 문법의 함정들입니다.

컴파일러의 눈으로 보지 않으면 절대 찾을 수 없는 미세한 오류들을 추적해 보십시오.

### \[실습 6\] 모양만 같은 가짜 마침표: `0` vs `'0'` vs `'\0'`
- **난이도:** ★★★★☆ 
- **관련 이론:** 12.3 특수기호를 쓰는 방법, 12.5 문장을 표현하는 방법

다음은 서버가 사용자로부터 입력받은 값 중 숫자 영 `0`이 나오면 검사를 중단하려는 코드입니다. 
반복문이 의도대로 작동할까요?

```c
#include <stdio.h>

int main(void)
{
        char data[] = "1230456";
        int i;
        
        /* 숫자 '0'을 만나면 멈추기? */
        for (i = 0; data[i] != 0; i = i + 1) {
                putchar(data[i]);
        }
        
        return 0;
}
```

**\[해설\]**
- **정답:** 0에서 멈추지 않고, 1230456 전체를 모두 출력한 뒤에야 배열의 끝에서 멈춥니다.
- **해석:** C 언어에서 숫자 0, 문자 `'0'`, 특수기호 `'\0'`은 사람 눈에만 비슷해 보일 뿐
  컴퓨터에게는 완전히 다른 값입니다.
  문자열의 마침표 역할을 하는 널 문자 `'\0'`은 정수값 0을 가집니다.
  하지만 데이터 안에 있는 글자 `'0'`은 아스키 _ASCII_ 코드 표준에 의해 정수값 48을 가집니다.
  따라서 `data[i] != 0`이라는 조건식은 사실상 "문장의 끝 `'\0'`이 아니면 계속해라"라는 뜻이 되어버립니다.
- **보안 포인트:** 보안 검사 로직에서 특정 문자를 필터링하려다 이런 착각을 하면,
  악성 데이터가 검사망을 유유히 통과하게 됩니다.
  글자 0을 찾고 싶다면 반드시 작은따옴표를 써서 `data[i] != '0'`으로 명시해야 합니다.

### \[실습 7\] 타입의 착각: 작은따옴표와 큰따옴표의 혼동 `'Y'` vs `"Y"`
- **난이도:** ★★★★★ 
- **관련 이론:** 12.2 작은따옴표, 12.6 큰따옴표, 10.5 배열과 포인터의 관계

다음 코드는 시스템 초기화를 실행할지 사용자에게 알파벳 한 글자로 묻는 치명적인 보안 로직입니다. 
사용자가 대문자 Y를 누르고 엔터를 쳤습니다. 
초기화가 실행될까요?

```c
#include <stdio.h>

int main(void)
{
        char command;
        
        command = getchar(); /* 'Y' 입력됨 */
        
        /* Y를 누르면 시스템 초기화? */
        if (command == "Y") {
                /* 초기화 로직 실행 */
        }
        
        return 0;
}
```

**\[해설\]**
- **정답:** 절대 실행되지 않습니다. 조건문은 항상 거짓이 됩니다.
- **해석:** C 언어에서 글자 하나를 뜻하는 작은따옴표 `'Y'`는 1바이트 크기의 정수값으로 변환됩니다.
  하지만 큰따옴표로 묶은 `"Y"`는 문자열 리터럴 배열이며,
  수식에서 사용될 때는 메모리의 시작 주소값으로 자동 변환됩니다.
  즉, `if (command == "Y")`는 키보드로 입력한 정수와 컴퓨터 메모 어딘가에 있는 문자열의 번지수를
  서로 같냐고 묻는 기괴한 질문이 됩니다.
- **보안 포인트:** 이런 코드는 컴파일러가 타입이 다르다며 경고를 주지만,
  경고를 무시하고 실행하면 보안 통제 기능이나 중요한 승인 로직이 영원히 작동하지 않는
  논리적 결함 _logic flaw_ 을 낳게 됩니다.
  한 글자를 비교할 때는 반드시 작은따옴표 `'Y'`를 사용해야 합니다.

### \[실습 8\] 위험한 재활용: 마침표가 지워진 배열
- **난이도:** ★★★★☆ 
- **관련 이론:** 12.5 문장을 표현하는 방법, 9.4 알 수 없는 값

다음 코드는 하나의 배열 버퍼를 재활용하여 해커의 아이디 `"hacker"`를 확인한 뒤, 
정상 사용자의 응답 `"ok"`으로 덮어쓰는 과정입니다. 
`print_string`으로 버퍼를 출력하면 화면에 무엇이 나올까요?

```c
#include <stdio.h>

void print_string(const char *str)
{
        int i;
        for (i = 0; str[i] != '\0'; i = i + 1) {
                putchar(str[i]);
        }
}

int main(void)
{
        /* 1. 처음엔 해커의 이름이 들어옴 */
        char buffer[7] = "hacker";
        
        /* 2. 버퍼를 재활용하여 "ok" 두 글자로 덮어씀 */
        buffer = 'o';
        buffer[8] = 'k';
        
        /* 3. 화면 출력 */
        print_string(buffer);
        
        return 0;
}
```

**[교수의 해설]**
- **정답:** ok가 아니라 **okcker**라는 기괴한 데이터가 출력됩니다.
- **해석:** 문자열의 끝은 오직 `'\0'` 기호가 결정합니다.
  처음 버퍼는 `{'h','a','c','k','e','r','\0'}` 상태였습니다.
  개발자가 o와 k를 앞의 두 칸에 덮어썼지만,
  세 번째 칸에 새로운 마침표 `buffer = '\0';`을 찍어주지 않았습니다.
  따라서 컴퓨터는 o, k를 읽고 난 뒤에도 멈추지 않고,
  예전에 남겨진 찌꺼기인 c, k, e, r까지 모두 읽어버립니다.
- **보안 포인트:** 이를 정보 노출 _information disclosure_ 취약점이라고 부릅니다.
  메모리 공간을 재활용할 때 이전 데이터를 확실히 지우거나 마침표를 제대로 찍지 않으면,
  이전에 썼던 민감한 정보가 그대로 노출되어 밖으로 새어 나가게 됩니다.

### \[실습 9\] 바꿀 수 없는 과거: 배열 이름에 문자열 대입 시도
- **난이도:** ★★★☆☆ 
- **관련 이론:** 12.6.2 큰따옴표 문장 저장 규칙, 9.7 고정된 메모리 주소

보안 엔지니어가 악성 코드가 감지되었을 때, 
시스템의 상태 메시지를 `"Safe"`에서 `"Danger"`로 한 번에 바꾸기 위해 아래 코드를 작성했습니다. 
코드는 정상적으로 작동할까요?

```c
int main(void) {
        /* 초기 상태는 안전함 */
        char status[7] = "Safe";
        int hacking_detected = 1;
        
        if (hacking_detected == 1) {
                /* 해킹 감지! 상태를 위험으로 변경? */
                status = "Danger";
        }
        
        return 0;
}
```

**\[해설\]**
- **정답:** 실행은커녕 컴파일 에러가 발생하여 아예 프로그램이 만들어지지 않습니다.
- **해석:** C 언어에서 배열을 선언할 때 `char status = "Safe";`처럼 쓰는 것은
  비어있는 방을 만들고 문자를 복사해 넣는 특별히 허락된 초기화 문법입니다.
  하지만 한 번 만들어진 배열의 이름 `status`은 메모리 상의 고정된 시작 주소를 뜻합니다.
  주소값 자체는 바꿀 수 없으므로, 대입 연산자 `=`의 왼쪽에 올 수 없습니다.
  문자열 리터럴 `"Danger"`의 주소를 배열의 주소 이름에 통째로 덮어씌울 수는 없습니다.
- **보안 포인트:** 문장을 통째로 바꾸려다 컴파일조차 되지 않아 보안 패치가 지연되는 전형적인 문법 실수입니다. 
  만들어진 배열의 내용을 바꾸려면, 앞선 예제들처럼 배열의 인덱스에 하나하나 접근해 문자를 바꾸거나,
  C 언어 표준 라이브러리를 이용해 메모리를 한 칸씩 복사해 넣어야 합니다.

---

길었던 실습이 모두 끝났습니다. 글자 하나 `char`와 배열, 그리고 포인터가 결합된 12장의 함정들까지 완벽히 분석해 냈습니다.

C 언어는 최신 언어들처럼 문자열을 알아서 늘려주고, 찌꺼기를 지워주고, 길이를 세어주는 친절함이 없습니다. 
하지만 바로 그 불친절함 덕분에, 우리는 컴퓨터가 메모리를 어떻게 할당하고 데이터를 0과 1로 처리하는지 
밑바닥의 진실을 목격할 수 있었습니다.

해커는 시스템이 대신 막아주겠거니 안일하게 생각하고 남겨둔 찌꺼기 버퍼, 크기 검사를 빼먹은 인덱스, 
잃어버린 `'\0'`의 빈틈을 파고듭니다. 
여러분이 1장부터 12장까지 배운 방어적 코딩 습관과 메모리를 투명하게 들여다보는 시야를 무기로, 
어떤 악성 공격에도 뚫리지 않는 단단한 소프트웨어를 설계하는 보안 전문가로 성장하시기를 바랍니다! 
수고하셨습니다.
