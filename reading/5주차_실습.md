반갑습니다, 사이버 보안학과 신입생 여러분.

지난 9장까지 우리는 데이터가 공중에 떠 있는 것이 아니라 메모리라는 거대한 아파트의 
특정 호수 주소에 저장된다는 사실을 배웠습니다. 
오늘 다룰 10장 포인터 _pointer_ 는 C 언어의 꽃이자, 해커들이 가장 사랑하는 궁극의 무기입니다.

지금까지는 변수의 이름 `x`, `y`를 불러서 값을 다뤘지만, 
포인터를 장착하면 메모리의 정확한 주소를 콕 집어서 직접 조작할 수 있게 됩니다. 
오늘은 1장부터 10장까지 배운 지식을 총동원하여, 
이 막강한 포인터가 어떻게 메모리를 제어하고 때로는 시스템을 파괴하는지 코드를 통해 읽어보겠습니다.

## 1부: 포인터의 기본 원리와 치명적 실수들

### \[실습 1\] 원격 제어의 마법: 역참조 연산자 `*`
- **난이도:** ★★☆☆☆
- **관련 이론:** 10.1 주소 연산자 `&`, 10.3 역참조 연산자 `*`

다음 코드는 시스템의 핵심 보안 설정 `security_level`을 포인터를 이용해 다루는 장면입니다. 
프로그램이 끝날 때 `security_level`의 최종 값은 얼마일까요?

```c
int main(void)
{
        int security_level = 1;      /* 1: 일반 보안 (100번지에 있다고 가정) */
        int *p = &security_level;    /* p에 100번지 주소 저장 */
        
        *p = 99;                     /* p가 가리키는 곳의 값을 99로 변경 */
        
        return 0;
}
```

#### \[해설\]
- **정답:** `security_level`은 **99**가 됩니다.
- **해석:** `int *p = &security_level;`에서 `&`는 변수가 살고 있는 메모리 주소를 빼내는 연산자입니다.
  `p`라는 포인터 변수 안에는 `security_level`의 주소가 저장됩니다.
  `*p = 99;`에서 `*`는 역참조 _dereference_ 연산자입니다.
  `p`에 적힌 주소로 직접 찾아가 그 안의 값을 99로 바꿉니다.
- **보안 포인트:** 비유하자면 `p`는 TV 리모컨이고, `*p`는 리모컨으로 조작하는 TV 화면입니다.
  변수의 진짜 이름을 한 번도 부르지 않고도 원격으로 값을 바꿀 수 있다는 것, 이것이 포인터의 막강한 힘입니다.

### \[실습 2\] 눈 가리고 총 쏘기: 와일드 포인터 _wild pointer_
- **난이도:** ★★★★☆
- **관련 이론:** 10.4.1 초기화하지 않은 포인터, 9.4 초기화 안 한 변수의 위험성

다음 코드는 해커의 공격을 막기 위해 방어 코드를 작성한 것입니다. 코드를 실행하면 어떤 일이 발생할까요?

```c
int main(void)
{
        int *defense_ptr; /* 포인터 선언만 하고 초기화하지 않음 */
        
        /* 어딘지 모르겠지만 방어력 1000을 덮어씌움 */
        *defense_ptr = 1000; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 시스템이 강제 종료되거나, 엉뚱한 메모리가 파괴되는 대형 사고인 메모리 오염이 발생합니다.
- **해석:** 포인터를 선언만 하고 초기화하지 않으면, 메모리에 원래 남아있던 알 수 없는 쓰레기 값이 주소로 취급됩니다.
  와일드 포인터 _wild pointer_ 라고 부릅니다.
- **보안 포인트:** 내가 뭘 건드린 것 같긴 한데 어딘지 모르고 아무 데나 낙서한 것과 같습니다.
  만약 그 우연한 주소가 운영체제의 핵심 영역이거나 다른 중요한 변수라면 시스템 전체가 마비됩니다.
  포인터는 반드시 `int *p = &x;`처럼 확실한 주소나, 가리킬 곳이 없다면 `NULL`로 초기화해야 합니다.

### \[실습 3\] 철거된 집의 열쇠: _use-after-free_ 취약점
- **난이도:** ★★★★★
- **관련 이론:** 10.4.3 댕글링 포인터, 9.5 블록과 함수

다음은 중괄호 `{}` 블록, 작업대를 사용하여 임시 암호화 키를 다루는 로직입니다. 
코드가 품고 있는 무서운 취약점은 무엇일까요?

```c
int main(void)
{
        int *p = NULL;
        
        {
                int temp_key = 777; /* 블록 안에서 임시 변수 생성 */
                p = &temp_key;      /* p가 임시 변수의 주소를 가리킴 */
        } /* 블록 종료. temp_key는 사라짐 */
        
        /* 블록 밖에서 p를 다시 사용 */
        *p = 888; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 이미 사라진 변수의 주소에 값을 쓰는 댕글링 포인터 _dangling pointer_ 버그가 발생합니다.
- **해석:** 9장에서 배웠듯, 중괄호 `{}` 블록 안에서 만든 변수는 블록이 끝나는 순간 작업대와 함께 메모리에서
  반납됩니다. 하지만 바깥에 있던 `p`는 여전히 그 철거된 집의 주소를 들고 있습니다.
  철거된 집에 몰래 들어가 가구를 부수는 격입니다.
- **보안 포인트:** 보안 업계에서는 이를 _use-after-free_ 취약점이라고 부릅니다.
  해커가 철거된 그 자리에 악성코드를 몰래 입주시켜 두면,
  프로그램은 정상적인 데이터인 줄 알고 예전 열쇠로 문을 열었다가 악성코드를 실행해 버립니다.
  수많은 스마트폰과 브라우저가 이 취약점 하나로 장악당했습니다.

### \[실습 4\] 포인터의 보폭: 포인터 산술 _pointer arithmetic_
- **난이도:** ★★★☆☆
- **관련 이론:** 10.6 포인터 산술, 10.5 배열과 포인터의 관계

배열과 포인터가 만나는 지점입니다. 3칸짜리 배열 `data`가 있습니다. `result`에 저장되는 값은 무엇일까요?

```c
int main(void)
{
        int data[3] = {10, 20, 30};
        int *p = data; /* 배열 이름은 첫 번째 칸의 주소로 자동 변환됨 */
        
        /* 주소에 1을 더하고 그 안의 값을 읽는다 */
        int result = *(p + 1); 
        
        return 0;
}
```

#### \[해설\]
- **정답:** `result`는 **20**이 됩니다.
- **해석:** C 언어에서 배열의 이름 `data`는 첫 번째 칸 `data[0]`의 시작 주소값처럼 행동합니다.
  가장 중요한 것은 `p + 1`의 의미입니다.
  포인터에게 `+1`은 단순한 숫자 1을 더하는 것이 아니라,
  자신이 가리키는 타입 `int`의 상자 한 개 크기만큼 다음으로 점프하라는 뜻입니다.
  따라서 `p + 1`은 두 번째 상자인 `data[1]`의 주소가 되고, 거기에 `*`를 붙였으므로 20을 읽어옵니다.
- **보안 포인트:** 이 보폭의 규칙을 잘못 계산하거나 `*(p + 5)`처럼 배열 크기를 넘어서 점프하게 되면,
  8장과 9장에서 배운 치명적인 버퍼 오버플로우 _buffer overflow_ 공격이 똑같이 발생하게 됩니다.

### \[실습 5\] 포장지의 비밀: 대괄호 `[]`의 진짜 모습
- **난이도:** ★★★★☆
- **관련 이론:** 10.7 `a[i]`를 포인터 연산으로 분해하기

초보 개발자가 배열을 다루다가 재미있는 실험을 했습니다. 아래 코드에서 `val1`과 `val2`는 같은 값을 가질까요?

```c
int main(void)
{
        int secret_codes[3] = {123, 456, 789};
        
        int val1 = secret_codes[2];
        int val2 = *(secret_codes + 2);
        
        return 0;
}
```

#### \[해설\]
- **정답:** 네, 두 값은 정확히 똑같이 789입니다.
- **해석:** 우리가 배열을 쓸 때 당연하게 썼던 대괄호 `[]`는 사실 C 언어 표준에서 포인터 덧셈과
  역참조 연산 `*(E1 + E2)`을 보기 좋게 포장해 놓은 문법에 불과합니다.
  `secret_codes[2]`는 내부적으로 `*(secret_codes + 2)`로 완벽히 동일하게 계산됩니다.
  시작 주소에서 2칸 이동한 뒤, 그곳의 값을 가져오라는 뜻입니다.
- **보안 포인트:** C 언어의 배열이 왜 인덱스 범위를 검사해주지 않는지 9장에서 궁금했을 것입니다.
  배열 대괄호의 진짜 정체가 단순한 '포인터 덧셈'이기 때문입니다.
  컴파일러는 여러분이 `+ 2`를 하든 `+ 1000`을 하든 그저 수학 계산을 한 뒤 메모리 값을 읽어올 뿐,
  그것이 남의 영토인지 _out-of-bounds_ 감시해주지 않습니다.

> **\[마무리 당부 - 1부\]**
> 자, 신입생 여러분. 1장부터 10장까지의 긴 여정이 끝났습니다.
> 우리는 C 언어가 프로그래머를 믿는다는 1장의 철학 아래, 메모리의 주소를 낱낱이 공개하고 `&`,
> 그곳을 직접 타격 `*`할 수 있게 해주는 포인터라는 무기를 손에 쥐었습니다. 
> 여러분이 작성한 코드 한 줄 `*p = 10;`이 안전한 시스템을 만들 수도 있고,
> UAF나 메모리 오염을 일으켜 해커에게 서버의 통제권을 통째로 넘겨주는 _hijacking_ 백도어가 될 수도 있습니다.
> 오늘 배운 메모리와 주소의 감각을 절대 잊지 말고,
> 언제나 이 포인터가 가리키는 곳이 내 땅이 맞는가를 의심하는 훌륭한 보안 전문가로 성장하길 바랍니다. 
> 이것으로 10장 읽기 수업을 마칩니다! 질문 있습니까?

## 2부: 메모리 구조와 포인터의 치명적 결합

보안의 심연으로 한 걸음 더 들어가려는 여러분의 열정, 아주 훌륭합니다. 
앞서 살펴본 5가지 예제가 포인터 자체의 문법과 기본적인 위험성을 다루었다면, 
이번에 추가할 4가지 실습은 메모리의 구조와 포인터가 결합하여 만들어내는 고차원적인 보안 취약점들입니다. 

해커의 시선으로, 메모리 아파트의 번지수가 어떻게 조작되고 파괴되는지 추적해 보십시오.

### \[실습 6\] 가짜 복사본과 진짜 리모컨: 값 복사 vs 주소 공유
- **난이도:** ★★★☆☆
- **관련 이론:** 9.3 변수의 정체, 10.3 역참조 연산자

다음 코드는 원본 비밀번호 `original_pw`를 보호하기 위해 사본을 만들어 다루려는 의도로 작성되었습니다. 
코드가 모두 끝난 후 `original_pw`의 값은 얼마가 될까요?

```c
int main(void)
{
        int original_pw = 1234;
        
        /* 1. 값을 복사한 변수 */
        int fake_copy = original_pw;
        /* 2. 주소를 가진 포인터 */
        int *real_controller = &original_pw;
        
        /* 해커의 공격 */
        fake_copy = 0000;      
        *real_controller = 9999; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** `original_pw`는 **9999**가 됩니다.
- **해석:** `fake_copy = original_pw;`는 단순히 `original_pw` 상자 안의 내용물 1234만 새로운 상자에 복사하는 것입니다.
  따라서 `fake_copy`를 0000으로 바꿔도 원본에는 전혀 영향을 주지 않습니다.
  하지만 `real_controller`는 원본 변수가 살고 있는 실제 메모리 번지수 `&`를 저장하고 있습니다.
  `*real_controller = 9999;`는 복사본이 아니라 그 주소로 직접 찾아가 원본의 값을 9999로 덮어써 버립니다.
- **보안 포인트:** 보안 시스템에서 민감한 데이터를 다룰 때,
  자신이 다루는 것이 안전하게 격리된 복사본인지 아니면 원본을 직접 타격할 수 있는 포인터인지 헷갈리면
  치명적인 데이터 변조 사고가 발생합니다.

### \[실습 7\] 변장한 해커: 배열 인덱스와 포인터 덧셈의 진실
- **난이도:** ★★★★☆
- **관련 이론:** 10.5 배열과 포인터의 관계, 10.7 `a[i]`를 포인터 연산으로 분해하기

다음 코드는 3칸짜리 배열을 포인터로 다루는 모습입니다. 
해커가 작성한 `*(p + 3) = 99;`는 메모리에서 어떤 일을 벌일까요?

```c
#define SIZE 3

int main(void) {
        int secure_data[SIZE] = {10, 20, 30};
        
        /* 배열의 이름은 첫 번째 칸의 주소다 */
        int *p = secure_data; 
        
        /* 해커의 공격: 포인터 덧셈 연산 */
        *(p + 3) = 99; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 배열의 크기를 벗어나, 존재하지 않는 4번째 칸의 메모리를 99로 파괴합니다.
- **해석:** C 언어에서 배열의 이름은 첫 번째 칸의 시작 주소값으로 변환됩니다.
  `p + 3`은 시작 주소에서 `int` 상자 3개 크기만큼 옆으로 점프하라는 뜻입니다.
  가장 충격적인 진실은, 우리가 8장에서 배운 `secure_data[3]`이라는 대괄호 문법이 내부적으로는
  정확히 `*(secure_data + 3)`으로 포장된 포인터 덧셈 연산에 불과하다는 점입니다.
- **보안 포인트:** C 언어에 배열의 경계를 지켜주는 안전바가 없는 진짜 이유가 바로 이것입니다.
  배열 접근은 그저 시작 주소에서 N칸 이동한 뒤 값을 써라는 단순한 수학 계산일 뿐이므로,
  경계를 넘어서는 버퍼 오버플로우 _buffer overflow_ 를 시스템이 막아주지 못합니다.

### \[실습 8\] 끝을 모르는 폭주: 끝 포인터 _end pointer_ 패턴과 부등호
- **난이도:** ★★★★★
- **관련 이론:** 10.6.3 끝 포인터 패턴, 7.3 `for`문의 부등호 실수

보안 엔지니어가 배열의 모든 데이터를 0으로 안전하게 지우기 위해 '끝 포인터'를 활용했습니다. 
코드의 치명적인 문제점은 무엇일까요?

```c
#define SIZE 3

int main(void)
{
        int data[SIZE] = {11, 22, 33};
        int *p;
        
        /* 배열의 끝을 나타내는 표지판 계산: 시작 주소 + 3칸 */
        int *end = data + SIZE; 
        
        /* 포인터 p가 end와 같아질 때까지 <= 반복? */
        for (p = data; p <= end; p = p + 1) {
                *p = 0; /* 데이터를 0으로 지움 */
        }
        
        return 0;
}
```

#### \[해설\]
- **정답:** 마지막 반복에서 `*end = 0;`이 실행되어 허가되지 않은 메모리를 0으로 덮어쓰는 오류가 발생합니다.
- **해석:** `data + SIZE`로 계산된 `end` 포인터는 배열의 마지막 원소 다음 칸을 가리키는 '경계선 표지판'입니다.
  포인터가 여기까지 도달했는지 주소를 비교 `p < end` 하는 것은 안전하지만,
  그 주소에 역참조 `*end`를 하여 값을 쓰거나 읽는 것은 절대 금지되어 있습니다.
- **보안 포인트:** for문의 가운데 조건에 `<` 대신 `<=`를 쓴 오프 바이 원 _off-by-one_ 실수입니다.
  단 한 번 더 반복했을 뿐이지만,
  이 실수로 인해 내 배열 바깥의 시스템 메모리나 다른 변수의 값이 0으로 파괴되어 프로그램 제어권이 넘어갈 수 있습니다.

### \[실습 9\] 돌아온 열쇠: 지역 변수 주소의 유출
- **난이도:** ★★★★★
- **관련 이론:** 9.5 블록과 함수 저장 기간, 10.4.3 댕글링 포인터

다음은 특정한 중괄호 `{}` 블록 안에서만 임시 비밀번호를 처리하고 버리려는 보안 로직입니다. 
코드는 해커에게 어떤 정보를 넘겨주게 될까요?

```c
int main(void)
{
        int *secret_ptr;
        
        {   /* 보안 작업대 (블록) 시작 */
                int temp_pw = 1234;
                secret_ptr = &temp_pw; /* 임시 비밀번호의 주소를 밖으로 유출! */
        }   /* 블록 종료. temp_pw는 공간을 반납함 */
        
        /* 해커가 유출된 주소로 다시 찾아가서 값을 훔쳐봄 */
        int stolen_pw = *secret_ptr; 
        
        return 0;
}
```

#### \[해설\]
- **정답:** 철거된 메모리 영역에 남아있는 1234라는 비밀번호 찌꺼기가 그대로 유출되거나
  예측 불가능한 쓰레기 값을 읽게 됩니다.
- **해석:** C 언어에서 중괄호 `{}`가 끝나면 그 안에서 만든 변수 `temp_pw`는 메모리에서 반납되며,
  컴퓨터는 이 공간을 다른 용도로 쓸 수 있게 만듭니다.
  하지만 바깥에 있던 `secret_ptr` 포인터는 여전히 그 폐기된 공간의 주소를 쥐고 있습니다.
- **보안 포인트:** 메모리가 반납되었다고 해서 컴퓨터가 친절하게 그 안에 든 값을 지우개로 지워주지 않습니다.
  해커는 이 철거된 주소를 열어보며 중요 정보를 빼내는 정보 노출 공격을 하거나,
  나아가 이 영역에 악성 코드가 덮어씌워졌을 때 그 코드를 강제 실행하게 만드는 공격을 가할 수 있습니다.
