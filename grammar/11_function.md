# 11: 일을 나누는 방법: 함수 _function_

우리는 지금까지

- `int`로 값을 만들고 계산하기
- `if`로 갈림길을 만들기
- `for`로 같은 일을 여러번 반복하기
- 배열로 많은 값을 한꺼번에 다루기
- 포인터로 주소 저장하기

를 해봤다. 

그런데 코드가 길어지고 복잡해지면, 필연적으로 다음과 같은 보안 위험을 마주하게 된다.

- **가독성 저하 _unauditable_:** 코드가 길면 논리적 실수를 찾아내기 어렵다. 
- **유지보수 문제 _patch management failure_:** 비슷한 코드를 복사해서 쓰면 일일이 찾아서 수정하기 어렵다.
 
여기서 필요한 프로그래밍 핵심 도구가 함수 _function_ 이다. 

**함수란 무엇인가?**

함수는 코드를 나누는 단위다. 큰 문제를 작은 실행 단위로 분해한다. 

## 11.0 이 장의 목표: 함수를 만들고 쓰기

함수와 관련된 코드는 크게 함수를 만드는 코드와 함수를 실행하는 코드로 분류된다. 
함수를 만드는 것은, 비유하자면 기계를 조립하는 과정으로, 프로그램에서 한 번만 가능하다.
함수를 실행하는 것은, 비유하자면 기계 버튼을 누르는 과정으로, 여러 번 하는 것이 가능하다.

한편, 함수는 크게 출력, 입력, 내부 동작이라는 세 구성요소로 이루어져 있으며, 이 장에서는 함수를 만들고 실행하는 문법과 구성요소 각각에 대해서 알아본다.

- **함수 만들기 _definition_:** 함수의 입력, 출력, 내부 동작을 정의하고, 이름을 붙인다. 
- **함수 실행하기 _call_:** 함수에 적절한 입력값을 제공하고, 맡은 작업을 수행하게 한 다음, 출력값을 받는다.
- **함수 출력 _return_:** `return`문의 역할을 이해한다. 
- **함수 입력 _parameter vs argument_:** 함수의 입력을 _definition_ 과 _call_ 맥락에서 이해한다.
- **함수 내부 동작 _body_:** 함수가 실행될 때 독립적인 메모리 공간을 쓰는 것을 이해한다.

## 11.1 왜 함수가 필요한가: 코드에서 숲을 볼 수 있는 방법

배열을 배웠으니, 이제 이런 코드를 쓸 수 있다.

```c
int main(void)
{
        int i;
        int a[5] = { 10, 20, 30, 40, 50 };
        int sum = 0;
        
        /* 배열 a의 합계 구하기 */
        for (i = 0; i < 5; i = i + 1) {
                sum = sum + a[i];
        }
        
        return 0;
}
```

문제는 이 다음이다. 프로그램이 커져서 배열 `b`의 합계도 필요해진 상황을 생각해보자. 

초보자는 보통 이렇게 해결한다.

**코드를 복사하는 나쁜 예**

```c
/* a의 합계 */
for (i = 0; i < 5; i = i + 1) {
        sum_a = sum_a + a[i];
}

/* b의 합계를 구하기 위해 코드를 복사해서 변수 이름만 바꿈 */
for (i = 0; i < 5; i = i + 1) {
        sum_b = sum_b + b[i];
}
```

복사해서 코드를 늘리면 세 가지 문제가 발생한다.

- **보안 패치의 악몽 _security nightmare_:** 만약 `for`문에서 보안 이슈가 발견되었다고 가정하자. 코드를 100군데 복사했다면, 100군데를 모두 찾아 고쳐야한다. 실수로 한 곳이라도 놓친다면? 해커는 정확히 그 수정되지 않은 한 곳을 뚫고 들어온다.
- **실수 유도 _error-prone_:** `b`의 합계를 구하는 코드를 복사 붙여넣기 하다가 `sum_a`에 더하는 오타를 내기 쉽다.
- **가독성 저하 _code bloat_:** 코드가 길어질수록 어디부터 어디까지가 합계를 구하는 부분인지 한눈에 파악하기 어렵다.

**해결책: 함수**

함수는 이 복잡한 과정을 블랙박스 안에 숨기는 도구다. 

우리는 복잡한 계산 과정 `for`, `i`, `+` 등은 상자 안에 넣어두고, 겉에서는 배열을 줄테니, 합계를 달라고 요청만 하면 된다.

코드는 이렇게 바뀐다. 

**함수를 쓰는 좋은 예**

```c
/* sum_array 함수의 코드를 한번만 만든다 */

/* 배열 합계를 구해야 할 때마다 sum_array 함수를 실행한다 */
sum_a = sum_array(a, 5);
sum_b = sum_array(b, 5);
```

이제 합계 계산 로직은 단 한 곳 `sum_array`에만 존재한다.

- **유지보수성 _maintainability_:** 고칠 일이 생겨도 한 곳만 고치면 되고,
- **가독성 _readability_:** 이름 `sum_array`만 봐도 무엇을 하는 코드인지 명확하다.

> [!NOTE]
> **`main`함수의 역할 변화: 실무자에서 관리자로**
>
> 함수를 쓰기 시작하면 `main` 함수의 역할이 완전히 달라진다.
> 
> - **지금까지의 `main` 실무자:** 직접 덧셈하고, 반복문 돌리고, 결과 저장까지 혼자 다 했다.
> - **앞으로의 `main` 관리자:** `sum_array` 같은 함수에게 데이터를 던져주고, 결과만 받아서 전체 흐름을 지휘한다.
> 
> 함수는 중복을 없애고 실수를 줄일 뿐만 아니라, 프로그래머가 '큰 그림'을 보게 해주는 가장 강력한 도구다. 

## 11.2 함수를 구성하는 세가지 요소: 입력, 동작, 출력

함수를 일상 생활에 비유하자면, 작업 하나를 수행하는 보안 검색대나 작은 기계와 같다.

자판기나 계산기 같은 기계를 생각해 보자. 기계에는 세 가지 요소가 있다.

- **입력 _input_:** 동전이나 재료를 넣는 곳.
- **동작 _processing_:** 기계 안에서 재료를 가공하는 일.
- **출력 _output_:** 결과물이나 거스름돈이 나오는 곳.

C 언어의 함수도 이 구조와 똑같지만, 보안 전문가는 이를 조금 더 엄격하게 바라본다.

- **입력 - 매개변수 _parameter_:** 공격의 통로
  - **정의:** 함수가 일을 하기 위해 외부에서 받아오는 재료다.
  - **보안 관점:** 이곳은 외부 데이터가 내부로 들어오는 유일한 통로 _input gate_ 다. 해커는 이 통로를 통해 악성 코드나 비정상적인 값인 독이 든 재료를 주입하려 한다.
  - **원칙:** 들어오는 모든 값을 의심하라. 함수 시작 부분에서 이 재료가 상했는지 검사하는 입력값 검증 _input validation_ 이 필수다. 

- **동작 - 함수 본문 _body_:** 격리된 공정
  - **정의:** `{ }` 중괄호 안에서 실제로 수행되는 코드들이다.
  - **보안 관점:** 이곳은 외부와 차단된 밀실이다. 여기서 선언한 변수들은 함수가 끝나면 모두 파기되므로, 원칙적으로는 정보가 밖으로 새지 않는다 _information hiding_.

- **출력 - 반환값 _return value_:** 결과 및 상태 보고
  - **정의:** 일이 끝나고 `return`문으로 돌려주는 최종 결과이다.
  - **보안 관점:** 이것은 단순한 결과물이 아니라 임무 성공 여부를 알리는 보고서다. 함수가 실패했다면 에러 코드  **-1** 등을 반환해야 하며, 함수를 실행한 쪽에서는 이를 반드시 확인해야 한다.

> [!NOTE]
> **보안 관점: 실패를 숨기지 말 것 _fail fast_**
>
> 함수가 실패했는데도 정상처럼 행동하는 것은 가장 위험한 설계다.
> 잘못된 상태가 프로그램 전체로 전파되며 _error propagation_,
> 나중에는 어디서 문제가 시작되었는지 추적하기 어려워진다.
>
> 안전한 설계 원칙은 다음과 같다.
>
> - 함수는 실패를 감추지 말고 즉시 보고하라.
> - 함수를 실행했으면 반환값을 반드시 검사하라.
>   반환값을 무시하고 버리는 것은 _MISRA C rule 17.7_ 위반이며, 에러 정보를 확인하지 않는 것은 Dir 4.7 위반이다. 반환값 무시는 절대 금물이다.
> - 오류가 발생하면 가능한 한 빨리 중단하라 _fail fast_.
>
> 실패를 조기에 감지하면, 피해는 함수 하나로 국한된다.
> 실패를 방치하면, 프로그램 전체가 오염된다.

**`main`함수 다시 보기: 운영체제와의 계약**

지금까지 무심코 썼던 `main`도 사실은 운영체제 _windows, macOS, linux_ 가 호출하는 함수다. 

```c
int main(void)
{
        /* ... 코드들 ... */
        return 0;
}
```

이 코드를 보안 기계 모델로 해석하면 이렇다. 

- **매개변수 `void`:** 지금은 아무 재료도 받지 않고 시작한다.
- **내부 동작: `/* ... 코드들 ... */`:** `{ }` 안의 코드들이 실제로 수행된다.
- **반환값 `return 0;`:** 모든 작업이 정상적으로 끝나서 `0`을 반환한다. 보통 **0**은 성공, **0**이 아닌 값은 실패를 의미한다. 

우리는 이미 함수를 만들고 쓰고 있었다. 단지 지금까지는 `main`이라는 '사장님 함수' 하나만 썼을 뿐이다.

다음으로, `main`이 혼자 짊어졌던 책임을 쪼개서, 각자의 보안 구역을 담당하는 직원 함수들을 직접 만들어 볼 차례다. 

## 11.3 함수를 조립하는 방법: 함수 정의 _definition_

가장 작은 예제로 시작해 보자. 

**예제: 두 수를 더하는 함수**

```c
/*
 * add 함수는 두 정수를 더한다.
 * 단순화를 위해 오버플로우 검사는 생략한다. 
 *
 * 매개변수:
 * - a: 더할 정수
 * - b: 더할 또다른 정수
 *
 * 반환값: a + b
 */
int add(int a, int b)
{
        return a + b;
}
```

코드를 해부해 보면 다음과 같다. 

- **반환 타입 `int`:** 이 기계는 최종적으로 정수 `int` 하나를 뱉어낸다는 약속이다.
- **함수 이름 `add`:** 우리가 붙인 이름으로, 앞으로 이 이름으로 함수를 불러다 쓴다.
- **매개변수 `(int a, int b)`:** 이 작업을 하려면 정수 두 개가 필요하다는 입력 조건이다. 
- **함수 본문 `{ ... }`:** 실제로 수행할 작업 내용이다.
- **반환 `return a + b;`:** 함수를 즉시 종료하고, 계산된 값 `a + b`을 호출한 곳으로 돌려준다.

정식으로 써보면, C 언어 표준에서는 다음과 같이 함수 정의 문법을 정해놓았다.

_반환_타입_ _함수_이름_ **(** _매개변수_목록_ **)** **{** _문장들_ **}**

> [!NOTE]
> **함수 헤더 _header_ 와 본문 _body_**
>
> 함수 정의는 크게 두 부분으로 나뉜다.
> 사람의 머리와 몸통이 하는 일이 다르듯, 함수도 역할이 명확히 구분된다.
>
> - **함수 헤더 _header_:**
>   함수의 첫 번째 줄로, 외부와 소통하는 접점 _interface_ 다.
>   "나는 이런 이름이고, 이런 재료를 받아서, 이런 결과를 줄 거야"라고 알리는 계약서와 같다.
>   - **구성:** 반환 타입 + 함수 이름 + 매개변수
>   - **주의:** 함수를 정의할 때, 헤더 뒤에 본문이 이어지기 때문에 세미콜론 `;`을 붙이지 않는다.
>     
> - **함수 본문 _body_:**
>   중괄호 `{}`로 감싸진 부분으로, 실제로 작업할 것이 구현 _implementation_ 되어 있다.
>   변수를 가공하는 로직이 들어있지만, 밖에서는 안을 들여다볼 수 없는 구조 _information hiding_ 다.
>
> **코드 해부:**
> ```c
> int add(int a, int b)        /* 헤더: 반환 타입 int, 함수 이름 add, 매개변수 int a, int b */
> {                            /* 본문 시작 */
>         return a + b;        
> }                            /* 본문 끝 */
> ```

> [!NOTE]
> **함수 이름을 짓는 규칙 _naming convention_**
>
> 변수가 데이터를 담는 명사라면, 함수는 데이터를 처리하는 동사다.
> 좋은 함수 이름은 주석 없이도 코드를 읽을 수 있게 해주는 최고의 문서다 _self-documenting code_.
> 함수 이름은 컴파일러를 위한 문법 규칙 _syntax rules_ 뿐만 아니라, 동료를 위한 작명 관례 _naming convention_ 도 지켜야 한다.
>
> - **동사로 시작하라 _verb-noun_:**
>   함수는 무언가를 수행하는 동작이다. 무엇을 하는지 명확히 드러내라.
>   - 좋은 예: `calculate_sum` 합계를 계산해라, `print_report` 보고서를 출력해라, `is_valid` 유효한지 확인해라
>   - 나쁜 예: `data` 무엇을 하는지 알 수 없다, `process` 너무 모호하다
>
> - **스네이크 표기법 _snake_case_:**
>   C언어 표준 라이브러리와 리눅스 커널은 전통적으로 소문자와 언더스코어 `_`를 사용하는 스네이크 표기법을 따른다.
>   - 예: `get_user_input`, `system_init`
>
> - **예약된 식별자 _reserved identifiers_:**
>   언더바 `_`로 시작하는 이름은 피하라. `_func`, `__init` 등은 컴파일러나 표준 라이브러리가 내부적으로 사용하기 위해 미리 찜해둔 예약된 식별자다.
>   함부로 쓰면 이름 충돌 _name collision_ 로 인해 알 수 없는 에러가 발생할 수 있다.

## 11.4 함수의 버튼을 누르는 방법: 함수 호출 _call_

함수 정의가 기계를 설계하고 조립해두는 것이라면, 함수 호출 _call_ 은 그 기계의 버튼을 눌러 실제로 작동시키는 것이다. 

**예제: `add`를 호출해서 결과 받기**

```c
/* 함수 호출: 3과 5를 던져주고 결과를 받는다 */
int x = add(3, 5);
```

여기서 `add(3, 5)`는 계산하면 값 하나로 변하는 식 _expression_ 이다. 

6장에서 `a > b`는 계산하면 **0** 또는 **1**로 변하는 식이라고 배웠던 것을 기억하는가? 함수 호출도 똑같다. 
컴퓨터 안에서 다음과 같은 과정이 일어나서 `add(3, 5)`가 값으로 변한다. 

1. `add(3, 5)`를 실행하기 위해 `3`과 `5`를 들고 `add` 함수 본문으로 이동한다.
2. 함수 호출 계산 결과로 반환받은 **8**을 들고 돌아온다.
3. 원래 자리에 있던 `add(3, 5)`는 사라지고, 그 자리를 **8**이 대체한다.

> [!NOTE]
> **매개변수 _parameter_ 와 인자 _argument_**
>
> 이 두 용어는 혼용되어 쓰이곤 하지만, 엄밀히 말하면 사용되는 맥락에 따라 다르다.
>
> - **매개변수 _parameter_:**
>   
>   함수를 정의 _definition_ 할 때 사용하는 변수의 이름이다.
>   "이 함수는 이런 형태의 값이 필요해"라고
>   자리를 맡아두는 비어있는 공간 _placeholder_ 와 같다.
>
>   함수가 실행되면, 여기에 전달된 값이 복사되어 들어온다.
>   
>   자판기에 비유하면 동전 투입구와 같다.
>   
>   - 예: `int add(int a, int b)` 에서 `a`, `b`
>
> - **인자 _argument_:**
>   
>   함수를 실행할 때 실제로 던져주는 구체적인 값이다.
>   함수가 실행되면 이 인자의 값이 매개변수에 복사된다.
>   
>   자판기에 비유하면 동전 투입구에 집어넣는 500원짜리 동전과 같다.
>   
>   - 예: `add(3, 5);` 에서 `3`, `5`

> [!WARNING]
> **인자의 개수는 매개변수와 같아야 한다**
> 
> 함수를 호출하는 것은 계약을 이행하는 것이다.
> 계약서에 "정수 2개를 달라"고 적혀있다면, 반드시 정수 2개를 줘야 한다.
>
> C 언어 표준에는 매개변수의 개수가 정해져 있는데 인자의 개수가 이와 다를 경우 미정의 동작 _undefined behavior_ 이라고 정해져 있다.
> 대개의 경우 컴파일 단계에서 에러가 발생한다. 

### 11.4.1 호출식의 피연산자는?

`3 + 5`라는 식에서 `+`가 연산자이고 `3`, `5`가 피연산자인 것처럼, `add(3, 5)`에서도 `()`가 연산자다.
그렇다면 이 연산자의 피연산자 _operand_ 는 무엇일까?

1. **함수 이름:** `add`와 같이 어떤 기계를 돌릴지 정하는 이름 _postfix-expression followed by parentheses `()`_.
2. **인자 목록:** `3`, `5`와 같이 기계에 넣을 재료들 _possibly empty, comma-separated list of expressions_.

여기서 초보자가 가장 많이 오해하는 것이 있다.
"인자 자리에는 꼭 숫자나 변수만 넣어야 하나요?"

정답은 "아니오"다.
결과적으로 값이 되는 식 _expression_ 이라면 무엇이든 넣을 수 있다.

**예제: 수식을 인자로 넣기**

```c
/* 1 * 2와 3 + 4를 계산해서 넣는다 */
int result = add(1 * 2, 3 + 4);
```

이 코드는 어떻게 실행될까? 요리사가 냄비에 재료를 넣기 전에 손질부터 하듯이, 컴퓨터도 함수를 부르기 전에 인자의 계산 _evaluation_ 을 먼저 끝낸다.

1. **인자 계산 - 재료 손질:**
   - 첫 번째 인자 `1 * 2`를 계산하여 **2**로 만든다.
   - 두 번째 인자 `3 + 4`를 계산하여 **7**로 만든다.

2. **함수 호출 - 냄비에 투입:**
   - 손질된 값 **2**과 **7**를 들고 `add` 함수로 간다.
   - `add(2, 7)`이 실행된다.

**심화: 함수의 입력값으로 함수 호출 결과 넣기 _nested call_**

함수 호출도 결국 값으로 변한다고 했다. 그렇다면 인자 자리에 또 다른 함수 호출을 넣을 수도 있을까?
물론이다.

```c
/* add(3, 5)가 먼저 8로 변한다 -> 그 다음 add(8, 1)이 실행된다 */
int final = add(add(3, 5), 1);
```

이처럼 함수는 레고 블록처럼 안쪽에서부터 계산되어 바깥쪽으로 결과가 전달된다.

> [!WARNING]
> **주의: 인자의 계산 순서는 보장되지 않는다**
>
> 만약 `add(A(), B())` 처럼 함수 두 개를 인자로 넣었다면, `A`가 먼저 실행될까 `B`가 먼저 실행될까?
> 정답은 "컴파일러 마음대로" _unspecified behavior_ 다.
> C 언어 표준은 이 순서를 정해두지 않았다.
>
> 따라서 순서가 중요한 작업이라면 인자 안에서 함수를 동시에 호출하지 말고, 따로 변수에 담아서 순서를 명확히 해야 한다.

### 11.4.2 호출식의 우선순위는: 높다.

함수 호출은 값을 만들어내는 식이기 때문에, 다른 연산자와 섞어 쓸 수도 있다.

```c
int result;
/* 함수 호출과 곱셈, 덧셈이 섞여 있다 */
result = 2 * add(3, 5) + 10;
```

이 코드는 어떻게 계산될까? 수학 시간의 기억을 떠올려보자. 곱셈과 덧셈이 섞여 있으면 곱셈을 먼저 한다. 그렇다면 함수는?

"함수 호출 `()` 연산자는 C 언어 연산자 우선순위에서 2순위 그룹에 속한다."

괄호로 묶은 수식을 제외하면 우선순위가 제일 높고, 지금까지 배운 연산자 중에서 같은 우선순위는 배열 첨자 `[]` 연산이 있다.

곱셈 `*`이나 덧셈 `+`보다 우선순위가 높기 때문에, 예제에서는 함수가 먼저 실행되어 값으로 변해야 한다.

컴퓨터가 이 수식을 처리하는 순서는 다음과 같다.

1. **함수 호출:** `add(3, 5)`가 가장 먼저 실행되어 **8**로 변한다.
   - 수식의 상태: `2 * 8 + 10`
2. **곱셈:** `2 * 8`을 계산하여 **16**이 된다.
   - 수식의 상태: `16 + 10`
3. **덧셈:** `16 + 10`을 계산하여 **26**이 된다.
   - 수식의 상태: `26`
4. **대입:** 최종 결과 **26**이 변수 `result`에 저장된다.

수식 안에 함수가 섞여 있다면, "함수라는 기계가 다 돌아가서 숫자로 변한 뒤에야" 더하기나 곱하기를 할 수 있다.

> [!NOTE]
> **치환 모델 _substitution model_ 로 이해하기**
>
> 복잡한 수식이 나와도 당황하지 말자. 함수 호출 코드를 지우개로 지우고, 그 자리에 반환값 _return value_ 를 적어 넣는다고 상상하면 된다.
>
> `x = add(1, 2) * add(3, 4);`
> 
> `x = 3 * 7;`
> 
> `x = 21;`

## 11.5 `return`: 함수의 종료와 결과 전달

함수에서 `return`은 두 가지 일을 동시에 한다.

- **함수를 즉시 끝낸다:** 실행 종료
- **값을 호출한 쪽으로 돌려준다:** 값 반환

### 11.5.1 `return`은 즉시 탈출이다

`return` 문은 단순히 값을 돌려주는 배달부가 아니라, 함수의 실행을 즉시 종료하는 마침표다.
`return` 문을 만나는 순간, 뒤에 코드가 아무리 많이 남아있어도 함수는 즉시 멈추고 함수를 실행했던 곳으로 간다.

> [!WARNING]
> **보안 관점: 도달할 수 없는 코드 _unreachable code_**
>
> `return` 문 뒤에 작성된 코드는 실행 흐름이 영원히 닿을 수 없다 _unreachable_.
> 보안 표준에서는 이를 엄격히 금지한다.
>
> _CWE-561_ 에 따르면, 실행되지 않는 코드는 단순한 낭비가 아니라 개발자의 논리적 착각을 의미한다.
> 예를 들어, 해킹 방어 코드를 실수로 `return` 뒤에 적었다면?
> 코드는 존재하지만 실제로는 아무런 방어막이 없는 상태가 된다.
>
> _CERT MSC12-C_ 와 _MISRA C:2025 rule 2.1_ 에서도 효과가 없거나 도달할 수 없는 코드는 탐지하고 제거하라고 권고한다. 
> 도달할 수 없는 코드는 소스 코드 리뷰를 방해하고 _unauditable_, 유지보수하는 사람에게 이 코드가 언젠가 실행될 것이라는 헛된 믿음을 준다.

`return`의 특성을 이용해서 함수 중간에서라도 에러가 발견되거나 할 일이 끝나면 지체 없이 `return` 하는 기법인 조기 종료 _early return_ 기법이 있다.
조기 종료는 중첩을 줄이고, 조건을 위에서 걸러내는 구조 _guard clause_ 를 만들어 가독성을 높인다.

**피라미드 코드 - 7.7.1절의 예제**

```c
int i, j;

for (i = 0; i < 5; i++) {
        for (j = 0; j < 5; j++) {
                if (target[i][j] == 7) {
                        return 1;  /* 찾았음을 의미하는 1을 반환값으로 가지고 함수를 종료한다. */
                }
        }
}
```

### 11.5.2 `return` 문의 정식 문법

`return` 문의 정식 문법은 다음과 같다.

**return** _수식_ **;**

수식의 결과 값이 계산된 후에, 반환된다. 

> [!NOTE]
> **보안 관점: 반환 값이 없는 함수는 정보 유출 가능성이 있다**
>
> 반환 타입이 `void`가 아닌 함수에서, 함수가 값을 반환하지 않고 종료되는 것은 단순한 실수가 아니라 심각한 보안 취약점이다 _CERT C MSC37-C_.
>
> - **제어 흐름의 붕괴 _control-flow corruption_:**
>   정의되지 않은 반환값은, 메모리에 남아 있던 값이 사용될 가능성을 만든다.
>   그 값은 예측할 수 없으며, 상태 확인 로직을 왜곡시킬 수 있다.
> 
> - **미정의 동작 _undefined behavior_:**
>   엉뚱한 반환값을 쓰는 것은 C 표준에서 미정의 동작 _undefined behavior_ 으로 정의되어 있다.
>   그 결과 보안 검사 코드가 제거되거나, 분기 로직이 왜곡될 수 있다.
>    
> - **`main`함수의 예외와 원칙:**
>   C99 표준부터 `main` 함수는 명시적으로 `return`을 하지 않으면 자동으로 `0`을 반환한다고 되어 있다. 
>   하지만 보안 전문가는 컴파일러의 자동 기능에 기대지 않는다.
>   어떤 환경에서도 항상 명시적으로 `return 0;`을 작성하여 의도를 분명히 하라.

### 11.5.3 반환값이 없는 `return`과 `void` 함수

모든 함수가 계산 결과를 내놓아야 하는 것은 아니다.
화면에 글자를 출력하거나, 경고음을 울리거나, 로그 파일에 기록을 남기는 것처럼 동작 _side effect_ 자체가 목적인 함수도 있다.

반환할 값이 없는 함수는, 정의할 때 반환 타입 자리에 `void` 비어있음이라고 적는다.

```c
/*
 * 경고음을 count번 출력하는 함수
 *
 * 매개변수: count 울릴 횟수
 * 반환값: void 없음
 */
void beep(int count)
{
        int i;
        for (i = 0; i < count; i = i + 1) {
                /* 경고음을 울린다 */
        }

        /* 반환할 값이 없으므로 return 생략 가능. 함수 끝. */
}
```

`void` 함수는 끝 `}`에 도달하면 자동으로 종료된다. 조기 종료를 위해 `return`문을 쓸 때는 반환값 없이 `return;`이라고 쓰면 된다. 

> [!CAUTION]
> **`void` 함수에서의 `return`은 반환값이 없어야 한다.**
>
> C 표준에서는 반환 타입이 `void`인 함수에서는 반환값이 있는 `return`이 나타나서는 안 된다고 명시되어 있다.
> 제약 위반이기 때문에 에러 메시지가 나온다 _must issue a diagnostic_.

## 11.6 매개변수: 함수가 받는 입력값 상자

함수를 정의할 때, 괄호 안에 있는 `int a, int b` 같은 것들은 함수 안에서만 쓰이는 변수로, 매개변수 _parameter_ 라고 부른다.
쉽게 말해, 바깥에서 던져주는 값을 받기 위해 함수가 미리 준비해 둔 입력값 상자다.

```c
int add(int x, int y)
{
        /* x와 y는 add 안에서만 존재하는 상자다 */
        return x + y;
}
int main(void)
{
        /* 3과 5라는 실제 값 인자를 함수 add의 상자 x, y에 던져넣는다! */
        int result = add(3, 5);
        return 0;
}
```

함수 `add`의 매개변수 상자 (`a`, `b`)는 `add` 함수 안에서만 존재한다. 

### 11.6.1 변수 이름이 유효한 범위 _block scope_

다음 코드를 보자. 많은 초보자가 헷갈려 하는 부분이다. 

```c
int add(int x, int y)
{
        return x + y;
}

int main(void)
{
        int x = 3;
        int y = 5;
        int result;
        
        result = add(x, y);
        
        return 0;
}
```

`main` 함수에서는 지역변수 `x`, `y`, `r`이 쓰인다. `x`에는 초기값이 `3`이 저장되고, `y`에는 초기값이 `5`가 저장된다. 
`r`에는 `add` 함수의 호출 결과가 저장되는데, 호출할 때 인자로 `x`와 `y`에 저장된 값이 쓰였다.

`add` 함수에서는 매개변수 `x`, `y`가 쓰인다. 두 매개변수에 저장된 값을 더해서 반환한다. 

`main`에서의 `x`와 `add`에서의 `x`는 다른 상자를 가리킨다. 비유하자면, 동명이인과 같다. 

그럼 어느 범위 _scope_ 까지 우리가 만든 변수의 이름이 하나의 상자를 가리키는 것일까? 

매개변수의 이름은 함수 본문에 해당하는 블록 _block_ 안에서 유효하다. 

> [!NOTE]
> **블록 `{}` 안에서 선언한 변수가 유효한 범위는?**
>
> - **일반 블록 `{}`:** 블록 _block_ 안에서 선언된 일반 변수들도 블록 유효범위 _block scope_ 를 갖는다. 선언된 다음부터 해당 블록이 끝나는 `}` 지점까지 유효하다.
> - **`for` - C99 이후:** `for`의 괄호 안에서 선언된 변수도 블록 유효범위 _block scope_ 를 갖는다. 선언된 다음부터 해당 `for`가 끝나는 지점까지 유효하다. 

> [!CAUTION]
> **보안 관점: 이름 가려짐 현상 _shadowing_ 주의**
>
> 동일한 블록 `{}` 안에서 같은 이름을 다시 선언하면 컴파일 에러가 발생한다.
>
> 하지만 안쪽 블록에서는 바깥쪽 블록에 존재하는 이름과 같은 이름을 쓰면, 에러 메시지가 나오지 않는다.
>
> ```c
> int x = 10;
> {                           /* 안쪽 블록 시작 */
>         int x = 20;         /* 이름 가림 현상 shadowing 발생 */
> }                           /* 안쪽 블록 끝 */
> ```
> 
> 안쪽 블록이 끝날 때까지는 바깥쪽 이름이 가려져서 접근할 수 없고 _shadowing_, 안쪽 블록이 끝난 뒤에는 바깥쪽 이름이 다시 유효해진다.
>
> _MISRA C Rule 5.3, CERT DCL01-C_ 에서는 이름 가림 _shadowing_ 을 금지한다.
>
> 의도치 않은 변수에 값을 잘못 저장하는 실수가 발생할 수 있기 때문이다 _CWE-563_.
>
> _CWE-1109_ 에서는 같은 이름을 여러 목적으로 쓰는 것 자체를 보안 취약점으로 분류한다. 

> [!NOTE]
> **함수 이름이 유효한 범위는?**
>
> 함수를 정의하면, 파일 유효범위 _file scope_를 갖는다. 정의한 후부터 해당 소스 파일의 끝까지는 함수 이름을 사용할 수 있다.


### 11.6.2 함수 호출시 값은 복사된다 _call by value_

함수를 호출 _call_ 할 때 인자 _argument_ 의 값은 매개변수 _parameter_ 에 어떻게 전달되는 것일까?

함수 본문을 실행하기 전에 인자의 값 _value_ 가 해당하는 매개변수에 복사되어 저장된다 _call by value_. 

**함수 본문에서 매개변수의 값을 바꾸면 호출한 곳의 지역 변수에 영향이 있을까?**
```c
void try_change(int x)
{
        /* 함수가 호출되어 본문이 실행될 때 x는 인자의 값으로 먼저 초기화된다. */

        x = 0; /* 매개변수 x를 바꿔보자 */
}

int main(void)
{
        int x = 1;

        try_change(x);

        /* main의 x는 여전히 1이다 */
        return 0;
}
```

이 코드에서 프로그램이 시작점인 `main` 함수가 실행되는 과정을 보자. 

1. **`main` 함수 호출됨:** `main` 함수에서 쓸 메모리 공간이 마련된다.
2. **`int x = 1;`:** `main` 함수를 위한 메모리 공간에 상자를 만들고 `x`라는 이름을 붙이고, 값 `1`을 저장한다.
3. **`try_change(x)`:**
   - **1) 호출 연산의 피연산자:** `try_change`가 함수라는 것이 확인되고, `x`의 값이 **1**로 평가된다.
   - **2) 작업 공간:** 함수에서 쓸 메모리 공간이 마련된다.
   - **3) 매개변수:** 매개변수 `x`에 해당하는 상자가 만들어지고, 초기값이 **1**로 복사해서 대입된다.
   - **4) `x = 0`:** 매개변수 상자에 저장된 값이 **0**으로 바뀐다.
   - **5) `}`:** 함수가 종료된다. 작업 공간은 반납된다.
4. **`/* ... */`:** `main` 함수를 위한 메모리 공간에는 아무 변화도 없다.
5. **`return 0;`:** 시스템에 정상 종료 코드를 반환하고 함수를 종료한다. 

여기서 핵심은, `main` 함수의 지역 변수 `x`가 `try_change` 함수를 호출하기 전과 후에 동일한 값을 유지한다는 것이다. 

> [!NOTE]
> **보안 관점: 격리의 원칙 _compartmentalization_**
>
> 거대한 배나 잠수함은 내부가 하나의 통으로 되어 있지 않고, 여러 개의 격벽 _bulkhead_ 로 나뉘어 있다.
> 만약 배의 앞부분에 구멍이 나서 물이 들어와도, 격벽 문을 닫으면 그 구역만 침수되고 배 전체는 가라앉지 않기 때문이다.
>
> 프로그래밍에서는 함수 _function_ 가 이 격벽 역할을 한다.
>
> 프로그래밍에서 함수는 기본적으로 자신만의 메모리 공간을 사용한다.
> 따라서 단순히 값을 복사해서 전달하는 경우에는 다른 함수의 지역 변수에 직접 영향을 주지 않는다.
> 
> 그래서 해커가 한 함수를 공격해 장악하더라도, 다른 함수의 메모리나 데이터에는 접근하지 못하게 피해를 최소화한다 _damage containment_.

## 11.7 함수에 포인터 넘기기: 원본을 바꾸는 기술

C는 항상 '값을 복사해서 전달'하지만, 그 값이 주소일 때, 함수가 원본을 바꾸는 효과가 생긴다.

**주소를 넘기면, 함수가 내 방에 들어와서 가구를 바꿀 수 있다.**

### 11.7.1 진짜 swap: `swap_int(int *a, int *b)`

주소를 받아서 원본을 바꾸는 제대로 된 `swap` 함수를 만들어보자.

```c
/* 매개변수가 포인터 int* 다. 즉, 주소를 받겠다는 뜻이다. */
void swap_int(int *a, int *b)
{
        int tmp;
        
        tmp = *a;   /* 1. a가 가리키는 곳인 x의 값을 읽어온다 */
        *a = *b;    /* 2. b가 가리키는 값 y를 a가 가리키는 곳 x에 쓴다 */
        *b = tmp;   /* 3. tmp 값을 b가 가리키는 곳 y에 쓴다 */
}

int main(void)
{
        int x = 1;
        int y = 2;
        
        /* 값(1, 2)이 아니라 주소(&x, &y)를 넘긴다 */
        swap_int(&x, &y);
        
        /* 성공! x=2, y=1 */
        return 0;
}
```

**성공의 핵심:**
- `swap_int`는 `x`, `y`라는 변수 이름은 모른다.
- 하지만 `&x` 주소를 `a`라는 리모컨에 담아왔다.
- `*a`를 건드리는 순간, 메인 함수에 있는 `x`가 바뀐다.

### [값 변화 추적]

| 단계 | main x | main y | a가 가리키는 곳) | b가 가리키는 곳 | 동작 |
| :--- | :---: | :---: | :---: | :---: | :--- |
| **호출 직후** | 1 | 2 | `&x` 1000번지 | `&y` 1004번지 | 주소 전달 완료 |
| **함수 내부** | **2** | **1** | `&x` 1000번지 | `&y` 1004번지 | `*a`, `*b`를 통해 원본 교체 |
| **리턴 후** | **2** | **1** | `a`가 사라짐 | `b`가 사라짐 | **원본이 바뀌어 있음** |


### 11.7.2 출력 인자 _output parameter_: 결과를 담아오세요

함수는 `return`으로 값을 하나만 반환할 수 있다는 한계가 있다.
하지만 포인터를 쓰면 여러 개의 결과를 얻어낼 수 있다.

> **"그릇의 주소를 줄 테니, 여기에 결과를 담아줘."** 패턴이다.

#### 예제: 몫과 나머지를 한 번에 구하기 `divmod`

```c
#include <stddef.h>    

/*
 * 반환값 int: 에러 코드 역할. 성공하면 0, 문제가 발생하면 -1. 
 * 결과값: 포인터 q가 가리키는 곳에 몫을, r이 가리키는 곳에 나머지를 저장
 */
int divmod(int a, int b, int *q, int *r)
{
        if (b == 0) return -1;            /* 0으로 나누기 방어, 에러 코드 -1 */
        
        if (q != NULL) *q = a / b;        /* q라는 주소를 줬다면 몫을 쓴다 */
        if (r != NULL) *r = a % b;        /* r이라는 주소를 줬다면 나머지를 쓴다 */
        
        return 0;                         /* 성공 */
}

int main(void)
{
        int quotient, remainder, result;
        
        result = divmod(10, 3, &quotient, &remainder);
        
        /*
        * remainder에는 나누기 성공을 의미하는 0이 저장되어 있다.
        * quotient에는 10 / 3의 결과인 3이 저장되어 있다.
        * remainder에는 10 % 3의 결과인 1이 저장되어 있다.
        */
        
        return 0;
}
```

### 11.7.3 `NULL` 체크의 진짜 매력: "필요 없는 건 무시하기"

위의 `divmod` 코드에서 `if (q != NULL)` 체크를 넣은 것은 단순히 오류를 방지하기 위함만이 아니다.
그 덕분에 함수를 훨씬 유연하게 쓸 수 있다.

만약 나는 몫만 필요하고 나머지는 필요 없다면?

```c
int remainder;
/* 나머지가 들어갈 자리에 NULL을 넣어버린다 */
divmod(10, 3, &remainder, NULL);
```

함수 내부에서 `r`이 `NULL`이므로 `*r = ...` 부분은 실행되지 않고 안전하게 넘어간다.
이처럼 포인터 인자는 옵션을 처리할 때도 매우 유용하다.

## 11.8 배열을 함수로 넘기기: 포인터와 동일

8.7절에서 본 것처럼 C 언어에서는 배열 이름을 수식 _expression_ 에 쓰면 제일 앞 원소의 시작 주소로 자동 변환 _array decay_ 되도록 설계되어 있다[^1].
그래서 배열 타입의 매개변수는 일반 `int` 타입의 매개변수보다 조금 더 특별하게 다룬다.

### 11.8.1 배열을 매개변수에 쓰는 법: 포인터로 쓴다

배열 이름이 함수 호출식 _function call_ 에서 인자로 평가될 때, 어차피 제일 앞 원소의 시작 주소로 자동 변환되어 넘어온다.

따라서 함수를 정의할 때도 매개변수가 배열의 시작 주소 값만 전달받게 된다. C 언어에서는 이 주소를 받기 위해 매개변수 자리에 `int arr[]`처럼 적는다.

그러면 컴파일러는 `arr`을 진짜 배열로 새로 만드는 것이 아니라, 단순히 시작 위치를 저장하는 변수로 취급한다. 

그래서 함수의 매개변수 선언 부분에서만큼은 다음 세 표현이 완전히 똑같다.

```c
void func(int a[]); /* "나는 배열을 받을 거야" - 사람을 위한 힌트. 가장 흔한 표기. */
void func(int a[5]); /* "크기가 5인 배열?" - 매개변수가 배열일 때 배열의 크기는 무시됨. */
void func(int *a);  /* "나는 주소를 받을 거야" - 실제 컴파일러의 해석. */
```

> [!NOTE]
> **C 표준의 숨겨진 규칙: 매개변수 조정 _parameter adjustment_**
> 
> 매개변수 자리에 `int arr[]`라고 적었을 때, 정확하게 어떻게 되는 것일까?
> C 표준에서는 "매개변수에서 배열 타입은 주소를 담는 타입으로 조정된다 _array of type shall be adjusted to pointer to type_"라고 규정하고 있다.
> 즉, 코드에 적혀 있기는 배열 같지만, 컴파일러가 알아서 변수의 타입을 적절하게 바꾸는 것이다.
>
> 추가로 배열 타입의 매개변수에 대해 자주 나오는 질문을 정리해 본다.
> 
> **Q. 함수 매개변수에 배열을 쓸 때 `int arr[5]`처럼 크기를 적지 않는 이유는?**
>
> 어차피 컴파일러에 의해 주소만 담는 변수로 변환되기 때문에, 괄호 안에 적어놓은 크기 정보는 컴파일러에게 완전히 무시된다.
>
> **Q. 주소만 담는 변수로 바뀌었는데, `[]` 연산자는 어떻게 쓰나?**
>
> C 언어는 개발자의 편의를 위해, 대괄호 `[]` 연산자는 시작 위치가 저장된 변수에도 사용될 수 있도록 설계해 두었다.
>
> **Q. 그럼 배열의 경우는 어떻게 값을 전달한다 *call by value*라고 할 수 있나?**
>
> C 언어는 항상 인자의 값이 매개변수에 복사된다 _call by value_. 배열도 예외가 아니다. 단지 복사되는 값이 주소 값일 뿐이다.

```c
/*
 * 매개변수로 배열을 받는 함수
 *
 * 매개변수
 * arr: 배열
 * n: 배열 arr의 크기
 */
void function_takes_array(int arr[], int n)
{
        /* 배열 arr과 크기 n을 가지고 작업을 한다 */ 
}
```

여기서 반드시 기억해야 할 점은,
배열을 함수로 전달할 때 반드시 배열의 크기 정보를 함께 전달해야 한다는 것이다.

함수 안에서는 `arr`가 가리키는 시작 위치만 알 수 있을 뿐,
이 배열이 실제로 몇 개의 원소를 가지고 있는지 자동으로 알 방법이 없다.

> [!NOTE]
> **보안 관점: _buffer overflow_ 와 _Aleph1_ 의 _smash the stack_**
>
> 함수 안에서는 배열의 시작 위치만 알 수 있고, 배열의 실제 크기는 자동으로 보호되지 않는다.
>
> 만약 잘못된 길이를 전달하거나, 반복문의 종료 조건을 실수로 잘못 작성하면,
> 배열의 범위를 벗어난 메모리에 접근하게 된다 _out-of-bounds_.
>
> 정상적인 경우에서 봤을 때 큰 실수가 없었더라도, 해커가 악의적인 데이터를 가공해서 함수에 전달하면, 
> 작업 공간 _stack_ 에 저장된 함수의 돌아갈 목적지 주소 _return address_ 를 덮어쓸 수도 있다.
> 그 결과 `return`을 했을 때 함수를 호출한 곳이 아니고 해커의 코드로 점프하게 된다.
>
> 1996년 _Aleph1_ 은 해커 잡지 _Phrack_ 에 실린 _smashing the stack for fun and profit_ 이라는
> 유명한 글에서 이러한 취약점을 이용해 어떻게 시스템을 무너뜨릴 수 있는지를 공개했다.
>
> 배열의 크기를 함께 전달하고, 경계를 철저히 검사하는 습관은
> 단순한 코딩 규칙이 아니라 보안의 기본 원칙이다 _CWE-119, CWE-787_.

### 11.8.2 매개변수로 받은 원본 배열을 바꾸기 (*side effect*)

배열이 함수로 전달될 때 시작 주소 값이 복사되기 때문에, 함수 안에서도 원본 배열에 직접 접근할 수 있다.
함수 내부의 작업이 함수 외부의 상태(원본 배열의 데이터 등)를 변경하는 것을 프로그래밍에서는 부수 효과 (*side effect*)라고 부른다.

```c
/*
 * 원본 배열을 변경하는 함수
 *
 * 매개변수
 * arr: 배열
 * n: 배열 arr의 크기
 */
void modify_array(int arr[], int n)
{
        if (n > 0) {
                arr[0] = -1;         /* 인자에 해당하는 원본 배열에 저장된 값이 바뀐다 */ 
        }
}
```

> [!CAUTION]
> **보안 관점: 배열의 경계 준수 MISRA C Rule 17.5**
>
> C 언어는 문법적으로 잘못된 크기의 배열을 넘기는 것을 허용하므로, 함수 내부에서 예상치 못한 경계 초과 오류가 발생하기 쉽다. 따라서 산업계에서는 이를 엄격히 규제한다.
> MISRA C Rule 17.5에서는 배열 크기에 해당하는 인자 값은, 실제 배열의 크기보다 반드시 작거나 같아야 한다고 규정한다.

## 11.9 함수 존재를 미리 알려주기: 함수 선언

C 언어의 컴파일러는 소스 코드를 순서대로 해석한다.

함수 호출식이 있을 때, 컴파일러가 문법에 문제가 없는지 확인할 수 있으려면 호출 시점에 대상 함수에 대한 정보를 알고 있어야 한다.

- 함수 이름
- 매개변수 타입 및 개수
- 반환값의 타입

컴파일러에게 이 정보들을 알려주는 방법은 두 가지다.

1. 함수 자체를 호출하기 전에 정의한다. 코드가 짧은 프로그램에서 주로 사용하다.
2. 이런 함수가 나중에 정의될 거라고 머리말만 미리 알려준다. 표준적인 방법이다.

머리말만 미리 알려주는 것을 함수 선언 _declaration_ 또는 함수 원형 _prototype_ 이라고 부른다.

> [!NOTE]
> **실무 코딩 표준: 정의할 때도 선언이 보여야 한다 _MISRA C Rule 8.4_**
>
> "어차피 정의가 선언을 겸한다면, 그냥 함수를 위에 떡하니 정의해 두고 선언은 안 써도 되는 것 아닐까?"
> 실무를 할 때는 여러명의 소프트웨어 엔지니어가 협업을 하기 때문에, 함수를 정의하는 입장과 함수를 호출하는 입장에서 같은 함수 선언을 공유한다는 것을 확인할 수 있어야 한다. 
>
> - **_MISRA C Rule 8.4_ 호환되는 선언의 가시성:** 일반 함수를 정의할 때는, 반드시 그에 맞는 호환되는 선언 _declaration_ 이 보여야한다 _visible_.
>
> 그래야 함수를 갖다 쓸 사람과 함수를 만드는 사람이 서로 똑같은 명세서를 보고 있는지 컴파일러를 통해 교차 검증할 수 있다.

### 11.9.1 선언과 정의의 문법 차이: 본문의 존재 유무

선언은 일종의 목차고, 정의는 본문이다. 선언과 정의의 가장 쉬운 구분법은 중괄호 `{}`가 있느냐 없느냐다. 

- **선언 _declaration_ - 이런 함수가 있다:** _함수_헤더_ **;**
- **정의 _definition_ - 이런 함수가 있고, 이렇게 동작한다:** _함수_헤더_ **{** _실행할_문장들_ **}**

코드로 보면 다음과 같다.

```c
/* 선언 */
void function(void);

/* 정의 */
void function(void)
{
        /* ... 실행할 내용 ... */
}
```

정의는 선언과 더불어 구현을 더한 것이다. 함수를 정의한다는 것은 컴파일러에게 "이 함수는 이름이 뭐고, 재료는 뭘 받는다"는 정보를 다 주는 셈이다. 
그래서 전체 코드가 소스 파일 하나에 다 들어가는 작은 프로젝트에서는, 함수를 호출식보다 위에 정의하면, 컴파일러가 이미 읽고 알게 되므로 굳이 선언을 추가로 쓸 필요가 없다. 

> [!WARNING]
>
> **함수 선언과 정의에서 헤더가 다르면?**
>
> 함수 선언은 여러번 할 수 있다. 하지만 같은 함수에 대한 선언은 헤더가 호환 가능 _compatible_ 해야 한다.
>
> 만약 선언에서는 반환 타입을 `int`로 적어두고 실제 정의에서는 `void`로 만들거나, 매개변수의 타입이나 개수가 조금이라도 다르면 어떻게 될까?
> 컴파일러는 타입 충돌 _conflicting types_ 에러를 발생시킨다.
>
> 컴파일러가 호환성을 따지는 핵심은 타입 _type_ 이지만, 가독성을 위해 반환 타입, 함수 이름, 매개변수의 타입과 이름까지 똑같이 맞추자.


[^1]: 배열 이름이 제일 앞 원소의 주소로 자동 변환되지 않는 예외는 `sizeof` 연산자나 `&` 연산자의 피연산자일 때, 그리고 문자열 상수인 배열의 경우에는 문자 배열의 초기값으로 쓰일 때이다.
