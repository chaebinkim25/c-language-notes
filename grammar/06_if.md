# 6. `if`: 조건에 따라 실행 흐름 바꾸기

5장까지의 우리는 변수를 만들고 `int x`, 값을 대입하고 `x = 10`, 계산하는 `x + 1` 작업을 했다.
실행 흐름은 모든 코드가 순서대로 실행되었다.

하지만 보안의 세계에서 무조건적인 신뢰는 가장 큰 취약점이다. 
외부에서 들어오는 데이터, 사용자의 입력, 시스템의 상태는 언제나 예상 범위를 벗어날 수 있다. 

- **기밀성 유지 _confidentiality_:** 사용자의 비밀번호가 일치하지 않는다면, 데이터를 보여주지 않는다.
- **무결성 검증 _integrity_:** 네트워크로 전송받은 파일이 원본과 다르다면 처리를 중단한다.
- **가용성 보장 _availability_:** 나누는 값이 0이라면, 나눗셈을 수행하지 않는다.
- **경계 검사 _boundary check_:** 준비된 메모리 공간보다 더 긴 입력은 거부한다.

프로그램에 판단력을 부여할 시간이 되었다. 무조건 직진만 하던 코드에 만약에 `if`라는 질문을 던져보자.

> [!NOTE]
> 
> **보안 관점: 해커의 목표**
>
> 해커는 입력 검증이 빠진 코드 경로를 노린다.
> 길이 검사를 하지 않았거나, 음수 입력을 막지 않은 부분을 찾아 공격한다.
> 꼼꼼하게 작성된 `if`는 보안 코딩의 가장 기본적인 방어막이다.

## 6.0 이 장의 목표: 선택의 기술

이 장에서는 가장 기본적인 판단 기술에 집중한다.

- **참과 거짓 이해하기 _boolean_:** C 언어는 `0`을 거짓 _false_ 로, 그 외 모든 값을 참 _true_ 로 본다.
- **흐름 제어하기 _branching_:** 위협을 감지했다면 차단하고, 안전하다면 허용하라. `if`, `else`
- **관계 파악하기 _relational_:** 데이터의 경계를 확인하고 검증한다. `>`, `<`, `==`

이 세 가지만 확실히 익혀도, 프로그램은 조건에 따라 서로 다른 실행 경로를 선택할 수 있게 된다.

## 6.1 C에서 참/거짓은 어떻게 표현될까: `1`과 `0`

우리는 일상에서 "그건 참 _true_ 다", "그건 거짓 _false_ 다"라고 명확하게 말한다. 
C89 표준에는 `true`나 `false`라는 키워드가 아예 없었다.

대신 C 언어는 숫자로 모든 것을 판단한다. 
마치 전구에서 전기가 흐르면 불이 들어오고, 전기가 흐르지 않으면 불이 들어오지 않는 것과 같다.

**C 언어의 판단 규칙: 전구 이론**

- **`0` _zero_:** 거짓 _false_. 전기가 흐르지 않아서 불이 들어오지 않는다.
- **`0`이 아닌 모든 값 _non-zero_:** 참 _true_. 전기가 흘러서 불이 들어온다. 

C 언어에서 판단이란 값이 `0`인지 아닌지로 평가하는 것이다.

```c
int is_valid;        /* 유효한 상태인지를 저장하는 변수 */

is_valid = 0;        /* 거짓 false */
is_valid = 1;        /* 참 true */
is_valid = -3;       /* 참 true - 0이 아니면 무조건 참이다. */
```

컴퓨터 과학에서는 이렇게 참과 거짓을 나타내는 값을 보통 불리언 _boolean_ 값이라고 부른다.

> [!NOTE]
> 
> **C 언어 역사: 불리언 _boolean_ 의 진화**
>
> - **C89:** 불리언 타입이 없었다. 정수 `0`과 `1`을 직접 사용했다.
> - **C99:** `0` 또는 `1`로 저장되는 `_Bool` 타입이 도입되었고, `<stdbool.h>`를 포함하면 `bool`, `true`, `false`를 사용할 수 있게 되었다.
> - **C23:** `bool`, `true`, `false`가 정식 키워드로 승격되었다. 헤더 파일 없이도 바로 사용할 수 있다.

> [!CAUTION]
> **보안 관점: `0`의 두 얼굴**
>
> 앞으로 프로그래밍을 배우게 되면 `0`의 의미가 문맥에 따라 달라지는 상황을 마주하게 된다.
>
> - **논리 판단 `if`:** `0`은 거짓 _false_ 다.
> - **에러 코드 `return`되는 값:** `0`은 성공 _success_ 를 의미한다. 에러가 0개라는 뜻이다.
>
> 만약 개발자가 프로그램 종료 코드를 착각한다면,
> 정상 종료를 오류로 처리하거나,
> 오류 상태를 정상으로 처리하는 문제가 발생할 수 있다.
>
> 보안 프로그래밍에서는 숫자의 의미를 항상 문맥 속에서 해석해야 한다.

## 6.2 조건에 따라 실행 과정을 바꾸는 간단한 형태: 참이면 실행, 거짓이면 건너뜀

이제 문법을 보자. C 언어에서 `if`라는 키워드를 써서 만드는 문장인 `if`문은 보안 게이트와 같은 구조를 가진다.
조건을 통과해야만 안쪽 코드가 실행된다. 

**문법 _syntax_**

```c
if (수식) {
        /* 조건이 참일 때만 실행되는 내용 */
}
```

`if`문 구조는 크게 세 부분으로 나뉜다. 

- **`if`:** 출입을 허가할지 검사하겠다고 컴퓨터에게 신호를 주는 키워드다.
- **`(수식)`:** 괄호 안에는 검사할 내용이 들어간다. 6.1에서 본 것처럼 값이 `0`이면 거짓 - 출입 금지, `0`이 아니면 참 - 출입 허가로 해석된다.
- **`{ ... }`:** 블록 _block_ 내부는 보안 구역이다. 조건이 참일 때만 실행할 문장들을 묶어둔 것이다.

다음은 사용자가 관리자 _admin_ 일 때만 시스템 설정을 초기화하는 코드다.

```c
int main(void)
{
        int is_admin = 1;        /* 1: 관리자, 0: 일반 사용자 */

        if (is_admin) {          /* is_admin이 0이 아니므로 (참), 중괄호 안으로 진입한다. */

                /* 관리자 권한으로 시스템 초기화 */

        }

        return 0;
}
```

만약 `is_admin`이 `0`인 일반 사용자였다면 어땠을까?

1. **조건식 평가:** 결과인 `0`은 거짓 _false_ 다.
2. **분기:** 결과가 `0`이므로 블록을 실행하지 않는다.
3. **블록 다음 문장 실행:** `return 0;`이 실행된다. 시스템은 초기화되지 않았다.
 
여기서 핵심은, `if`문이 코드의 실행 흐름을 갈라놓는다는 것이다.
조건이 거짓이라면 해당 블록은 실행되지 않는다 _skip_. 
일직선으로 흐르던 실행 흐름이 조건에 따라 달라지게 되는데, 이를 컴퓨터 과학에서는 분기 _branch_ 라고 한다. 

> [!NOTE]
> 
> **보안 관점: CPU의 조건 분기**
>
> CPU는 조건의 결과에 따라 특정 위치로 점프하거나 다음 명령을 계속 실행하며, 이를 조건 분기 _conditional branch_ 라고 한다.
>
> 메모리 취약점 등을 이용해 분기 대상 주소를 바꾸는 공격을 실행 흐름 조작 _control flow hijacking_ 이라고 한다.
>
> 조건을 올바르게 작성하는 것은 입력 검증과 논리 오류를 막는데 중요하며, 이는 보안 프로그래밍의 기본이 된다.

> [!NOTE]
> 
> **보안 관점: 해커와의 숨박꼭질**
>
> 우리가 `if`문으로 올바른 길을 안내하려 노력하는 동안, 해커들은 그 길을 비틀어 샛길로 빠기게 만들려고 한다.
> 개발자가 작성하는 C 코드가 엉성해서 취약점이 있으면,
> 운영체제는 _data execution prevention_, _address space layout randomization_,
> _control flow integrity_ 라는 기술들로 프로그램을 필사적으로 지키려고 노력한다.
> 하지만 이 방어막들도 우회 기법이 존재하므로,
> 애초에 보안 프로그래밍 _secure coding_ 으로 취약점을 만들지 않는 것이 최고다. 

## 6.3 조건을 만드는 기본 재료: 비교 기호 6개

판단을 내리려면 기준이 있어야 하고, 그 기준을 확인하려면 비교를 해야 한다. 
C 언어에서는 다음 6가지 관계 연산자 _relational operator_ 를 사용해 두 값을 비교한다.

|기호|뜻|보안 예시|참이 되는 경우|
|:---:|:---|:---|:---|
|`==`|같다|`pw == input`|비밀번호가 일치할 때|
|`!=`|다르다|`hash != origin`|파일이 변조되었을 때|
|`<`|작다|`user_level < 3`|권한이 부족할 때|
|`>`|크다|`len > buffer_len`|버퍼 크기 초과|
|`<=`|작거나 같다|`try_count <= 5`|시도 횟수 제한 내|
|`>=`|크거나 같다|`age >= 19`|성인 인증 통과|

> [!CAUTION]
> 
> **`==` vs `=`:** 수학과 다른 위험한 함정
>
> 초보 개발자가 자주 저지르는 실수가 이 기호를 혼동하는 것이다.
> 
> - **수학:** `a = b` (같다)
> - **C 언어:**
>     - `a = b` (대입: `b`의 값을 `a`에 저장한다)
>     - `a == b` (비교: 같은지 확인한다)

> [!NOTE]
> 
> **보안 관점: 셀프 백도어 _privilege escalation_**
> 
> 만약 이 실수가 로그인 시스템에서 발생하면 어떻게 될까?
>
> ```c
> /* 관리자인지 확인하려고 했으나... */
> if (is_admin = 1) {
>         /* 큰일 났다! is_admin에 1이 대입되면서 관리자 권한을 획득해버렸다. */
>         /* 관리자 전용 코드 */
> }
> ```
> 
> 1. `is_admin` 변수에 `1`이 저장된다.
> 2. C 언어는 이 수식의 결과가 `1`이므로 참이라고 판단한다.
> 3. 그 결과 관리자 권한이 무조건 허용된다.
>
> 결과적으로 단순한 오타가 아니라, 시스템 전체를 위험에 빠뜨리는 권한 상승 버그가 된다.

> [!TIP]
> 
> **보안 관점: 요다 표기법 _yoda conditions_**
>
> `==`와 `=`를 혼동하는 치명적인 실수를 막기 위해, 일부 보안 개발자들은 상수를 왼쪽에 쓰는 습관을 들인다.
> - **`if (x = 5)`:** 컴파일러가 에러를 내지 않아서 실수하기 쉽다
> - **`if (5 = x)`:** 실수로 `==` 대신 `=`라고 쓰면, `5`에 `x`를 대입할 수 없으므로 컴파일 에러가 발생한다. `==`를 써서 `if (5 == x)`라고 한 경우는 에러도 없고 잘 동작한다.
> 
> 마치 스타워즈의 요다처럼 순서를 바꿔서 말한다고 해서 요다 표기법이라고 부른다.
> 실수는 방지할 수 있지만 가독성이 떨어져서 팀 스타일에 따라 사용 여부를 결정한다.
 
**비교 연산의 결과**

비교식은 수식 _expression_ 의 일종이기 때문에, 결과값이 있다.

- **비교 결과가 참일 때:** 결과 값은 `1`이 된다.
- **비교 결과가 거짓일 때:** 결과 값은 `0`이 된다.

비교 연산자의 결과는 항상 `int` 타입이며, 값은 `0` 또는 `1`이다. 
6.1에서 본 C 언어의 참과 거짓 규칙과도 연결된다.

```c
if (a > b)
```

와 같은 코드가 있으면, 컴퓨터는 `a > b`를 계산해서 `0`또는 `1`로 평가한 뒤에 `if`문을 실행한다.

**코드에서 확인하기**

```c
int main(void)
{
        int a = 10;
        int b = 3;
    
        int t1, t2;
    
        t1 = (a > b);       /* 결과: 1 */
        t2 = (a < b);       /* 결과: 0 */

        return 0;
}
```

## 6.4 둘 중 하나만 실행하는 법: `if - else`

앞서 배운 `if` 문은 조건이 맞으면 실행하고, 아니면 무시하는 옵션과 같았다. 

하지만 보안의 현실에서는 허용이냐 차단이냐 중에 하나를 반드시 선택해야 하는 경우가 더 많다.

- **악성 코드:** 백신에 의해 코드가 분석되고 있으면 활동을 하지 않고, 백신이 없으면 컴퓨터를 공격한다.
- **보안솔루션:** 해킹 시도면 차단하고, 정상 유저면 통과시킨다. 

그럴 때 쓰는 것이 `else`, 조건이 만족하지 않으면 실행하는 문법이다.

**문법 _syntax_**

```c
if (조건) {
        /* 조건이 참일 때 실행되는 내용 */
} else {
        /* 조건이 거짓일 때 실행되는 내용 */
}
```

`if-else`는 두 갈래 길이다. 조건식은 항상 `0` 또는 `0`이 아닌 값으로 평가되므로, 두 블록 중 정확히 하나만 실행된다.

> [!CAUTION]
>
> **보안 관점: 중복 코드 사용 금지 _don't repeat yourself, DRY_**
>
> 불필요하게 동일한 로직이 `if`와 `else`에 동일하게 존재하면, 한 곳에서 버그나 보안 취약점을 수정했을 때, 복사된 다른 곳의 코드에 패치 _fix_ 가 누락될 수 있다 _CWE-1041_. 복제된 코드는 코드 품질 저하의 강력한 지표로 간주된다 _CWE-398_.
>
> **나쁜 예: 중복된 코드로 인해 패치가 누락될 위험이 높다**
> ```c
> int is_admin = 0;
> int user_id = 0;
> /* 유저 아이디를 user_id에 저장하고, 관리자인지를 is_admin에 저장한다 */
>
> if (is_admin) {
>         /* user_id를 로그에 기록 - 중복된 코드 */
>         /* 관리자 화면을 띄운다 */
> } else {
>         /* user_id를 로그에 기록 - 중복된 코드: 로그 패치가 실수로 여기만 적용될 수도 있다 */
>         /* 일반 유저 화면을 띄운다 */
> }
> ```
>
> **좋은 예: 공통 로직을 밖으로 빼서 _DRY_ 원칙을 준수한다**
> ```c
> int is_admin = 0;
> int user_id = 0;
> /* 유저 아이디를 user_id에 저장하고, 관리자인지를 is_admin에 저장한다 */
>
> /* user_id를 로그에 기록 - 단일 지점에서 관리 */
>
> if (is_admin) {
>         /* 관리자 화면을 띄운다 */
> } else {
>         /* 일반 유저 화면을 띄운다 */
> }
> ```

**예제: 앱 무결성 보호 _anti-debugging_**

보안팀은 앱이 디버거에 의해 분석되고 있는지 감지해, 의심 환경에서는 동작을 제한하거나 변경한다 _anti-debugging_. 

- **상황:** 유저가 앱을 실행한다.
- **보안 정책:** 정상적인 실행이라면 로그인 화면을 보여주고, 해킹 도구가 감지되면 앱을 종료하거나 민감 기능을 차단한다.

```c
int is_being_debugged = 0;
/* 운영체제에게 내 프로그램을 분석하는 디버거가 있는지 물어보고, 결과값을 is_being_debugged에 저장한다 */

if (is_being_debugged) {
        /* 참: 디버거가 감지되었다. */
        /* 현재 상황을 기록한다. */
        /* 앱을 종료하거나 민감 기능을 차단한다. */
} else {
        /* 거짓: 정상 환경이다. 서비스를 제공한다. */
        /* 정상 서비스를 실행한다. */
}
```

보안 관리자에게 `if-else`는 보안 정책을 강제하는 기본 수단이다.
환경 검증, 권한 검증, 입력 검증은 결국 `if-else`로 구현된다. 

- **`if` 위협 감지:** 즉시 서비스를 중단하고 방어 태세로 전환한다.
- **`else` 안전함:** 안전이 확인되면 정상적인 서비스를 제공한다.

보안이 중요한 금융 앱에서 이런 코드가 없다면, 공격자가 권한 검사를 우회하거나 비정상 입력으로 로직을 망가뜨릴 여지가 생긴다.

> [!NOTE]
> 
> **윈도우의 `IsDebuggerPresent` 함수**
>
> 실제로 윈도우 운영체제는 비슷한 기능 _application programming interface_ 를 제공한다.
> `IsDebuggerPresent`라는 이름의 함수는 현재 내 프로그램이 디버거에 의해 분석당하고 있는지를
> `0` - 분석되고 있지 않음 - 또는 `0`이 아닌 값 - 분석되고 있음 - 으로 알려준다.

## 6.5 다중 선택지를 만드는 방법: `else if`

세상 일은 모 아니면 도처럼 두 가지로만 나뉘지 않는다. 

보안 등급을 예로 들면, 관리자, 유저, 게스트처럼 세 개 이상의 선택지가 필요할 때가 훨씬 많다.

보안 장비는 수만 가지의 네트워크 신호를 구분해서 처리해야 한다.

그럴 때는 `if`와 `else` 사이에 `else if`를 끼워 넣어 보안 필터의 단계를 계속 늘릴 수 있다. 

**문법 _syntax_**

```c
if (조건1) {
        /* 조건1이 참일 때 실행되는 내용 */
} else if (조건2) {
        /* 조건1은 거짓이고, 조건2가 참일 때 실행되는 내용 */
} else {
        /* 위의 모든 조건이 거짓일 때 실행되는 내용 */
}
```
마치 여러 층으로 된 보안 검색대와 같다. 
위에서부터 차례대로 조건을 검사하다가, 하나라도 참이면 해당 조취를 취하고 전체 검문을 종료한다.

**예제: 방화벽의 포트 필터링 _port filtering_**

방화벽은 들어오는 네트워크 패킷의 포트 번호 _port_ 를 보고 문을 열어줄지 결정한다.

- **22번:** 관리자용 원격 접속 SSH는 무조건 허용
- **80번:** 웹사이트 접속 HTTP는 기록 후 허용
- **그 외:** 정체불명의 접속 시도는 차단

```c
int main(void)
{
        int port = 3389;   /* 해커가 원격 데스크톱으로 접속 시도 중! */
        
        /* Rule 1: 관리자 전용 포트 SSH */
        if (port == 22) {
                /* [ALLOWED] 관리자 원격 접속을 승인합니다. */
        }
        /* Rule 2: 일반 웹 서비스 포트 HTTP */
        else if (port == 80) {
                /* [LOGGING] 웹 접속 기록을 남기고 승인합니다. */
        }
        /* Rule 3: 그 외 모든 포트 Default Deny */
        else {
                /* [BLOCKED] 허용되지 않은 포트 접근 차단!*/
        }
    
        return 0;
}
```

> [!NOTE]
>
> **보안 관점: 기본 차단 원칙 _default deny_**
>
> 알지 못하는 새로운 공격이나 예상치 못한 예외 상황에 대비하기 위해,
> 마지막 `else`는 항상 기본 차단으로 닫아두어야 한다.
> 명시적으로 허용된 조건 `if`, `else if`를 제외한 모든 접근을 거부하는 방식을
> 허용 목록 _whitelist_ 보안이라고 부른다.
> 문법적으로 `else`는 생략할 수 있지만,
> 보안 코딩에서는 이를 안전 실패 _fail-safe defaults_ 원칙을 구현하는 필수적인 방어막으로 여긴다.

**핵심 규칙: 선착순 1명과 보안의 함정**

`else if`를 쓸 때 가장 중요한 규칙은 위에서부터 검사해서 처음 참이 되는 블록 하나만 실행한다는 점이다.

만약 22번 포트로 접속했다면? 첫 번째 `if` 문에서 접속이 승인되고, 뒤에 있는 `else if`나 `else` 블록은 실행되지 않는다.

> [!CAUTION]
> 
> **보안 관점: 규칙 그림자 _rule shadowing_**
>
> 방화벽 _firewall_ 설정을 할 때 이 순서가 매우 중요하다.
> 만약 모든 접속 허용 규칙을 맨 위 `if`에 두고, 해커 IP 차단 규칙을 그 아래 `else if`에 두면 어떻게 될까?
> 해커가 들어와도 첫 번째 규칙인 모든 접속 허용에서 통과되고, 차단 규칙은 영원히 실행되지 않는다.
> 상위 규칙이 하위 규칙을 가리는 이런 현상을 _rule shadowing_ 이라고 한다.
> 보안 정책은 항상 구체적인 차단/허용 규칙을 먼저 `if`로, 포괄적인 규칙을 나중에 `else`로 배치해야 한다.
> _rule shadowing_ 은 도달할 수 없는 코드 _unreachable code_ 를 만들 수 있다.
> 도달할 수 없는 코드는 심각한 보안 위협으로 간주되어 엄격히 금지된다 _CWE-561, MSC12-C, MISRA C:2025 Rule 2.1 & 2.2_
>
> /* 나쁜 예: 포괄적인 규칙이 구체적인 규칙을 가려버림 Rule Shadowing */
> ```c
> if (port > 0) {
>         /* [ALLOW] 모든 포트 허용 - 치명적 보안 결함! */
> } else if (port == 22) {
>         /* [BLOCK] 관리자 포트 22 차단 시도 */
>         /* 영원히 실행되지 않는 unreachable code - 해커가 22번으로 들어와도 첫 번째 if에서 통과됨 */
> }
> ```

## 6.6 자주 쓰는 조건 패턴 6가지: 보안 관점에서 

조건문 문법 공부는 끝났다. 여기부터는 실전이다. 
나중에 수만 줄짜리 코드를 짜게 되어도 계속해서 등장하는 기본기다.

### 6.6.1 0인지 검사하기: 시스템 붕괴를 막는 방어적 프로그래밍

5장에서 봤듯이 `0`으로 나누는 것은 수학적으로 불가능하며, 컴퓨터에서도 금지된다. 
사용자가 실수로 입력했든, 해커가 일부러 `0`을 넣었든, 프로그램은 위험한 상황을 스스로 막아야 한다.

```c
int main(void)
{
        int a = 10;
        int b = 0;
        int q;

        /* 안전장치: 나누는 수 b가 0이 아닐 때만 계산한다 */
        if (b != 0) {
                q = a / b;      
        } else {
                /* 0이라면 계산하지 않고 0으로 처리하거나, 에러 메시지를 띄운다 */
                q = 0;          
        }

        return 0;
}
```

`if` 문은 단순히 분기하는 것을 넘어, 프로그램이 뻗지 _crash_ 않도록 막아주는 에어백 역할도 할 수 있다. 
현장에서는 이를 방어적 프로그래밍 _defensive programming_ 이라고 부른다.

> [!NOTE]
> 
> **보안 관점: _denial of service, DoS_ 공격과 가용성**
>
> 만약 서버 프로그램에 이런 방어 코드가 없다면?
> 해커는 고의로 0을 입력값으로 보내 서버를 강제 종료 _crash_ 시킬 것이다.
> 서비스 거부 공격 _denial of service, DoS_ 라고 한는데, 서비스가 중단되면 회사는 막대한 손해를 입는다. 
> `if (b != 0)` 한 줄은 수천만 원짜리 서버를 지키는 가장 저렴하고 확실한 보험이다. 

### 6.6.2 짝수/홀수 판별하기: `%` 연산자의 패턴화

나머지 연산 `%`는 단순히 나머지를 구하는 계산기가 아니다. 
프로그래밍에서 `%`는 규칙적인 패턴을 만들어내는 강력한 도구다.

```c
if (n % 2 == 0) {
        /* 짝수: 2로 나눈 나머지가 0 */
} else {
        /* 홀수: 2로 나눈 나머지가 1 */
}
```

보안 분야에서는 이 간단한 원리를 응용해 엄청난 양의 디도스 _distributed denial of service, DDoS_ 공격 트래픽을 분산시키거나, 암호화 키를 교체하는 규칙을 만든다. 

**보안 예제: 방화벽 부하 분산 _load balancing_**

해커가 초당 10만 번의 공격을 퍼부어 방화벽을 다운시키려 한다고 가정해보자. 방화벽 한 대로는 버틸 수 없다.

하지만 `% 2`를 사용하면 트래픽을 정확히 반반으로 나누어 두 대의 장비에 분산시킬 수 있다. 라운드 로빈 _round robin_ 방식이라고 한다.

```c
int main(void)
{
        int packet_id = 1004;  /* 들어온 패킷의 고유 번호 */

        /* 패킷 번호를 2로 나눈 나머지로 홀짝을 판별한다 */
        if (packet_id % 2 == 0) {
                /* 짝수 패킷 -> 1번 방화벽으로 보낸다 */
        } 
        else {
                /* 홀수 패킷 -> 2번 방화벽으로 보낸다 */
        }
        return 0;
}
```

이처럼 `%` 연산자는 데이터를 분산처리하기 위해 분류하는 가장 빠르고 효율적인 방법이다. 장비가 더 많아지면 `%` 하는 숫자를 늘리면 된다. 

### 6.6.3 범위 검사하기: 음수 출금 공격 _money copy bug_

어떤 값이 정상적인 범위 안에 있는지 검사하는 것을 보안 용어로 입력값 검증 _input validation_ 이라고 한다. 
해커가 시스템의 허점을 찌르는 것을 막는 제1 방어선이다.

은행 시스템에서 돈을 출금하는 기능을 만든다고 가정해보자.
출금을 하면, $잔액 = 잔액 - 출금액$이라는 수식을 써서 잔액을 업데이트 한다. 만약 개발자가 출금액이 0보다 커야 한다는 검사를 깜빡하면 어떤 일이 벌어질까?

해커는 -50,000원을 출금하겠다고 입력한다.

1. **수식:** $잔액 - (-50,000)$
2. **결과:** 마이너스와 마이너스가 만나면 플러스가 된다. $잔액 + 50,000$
3. **오동작:** 돈을 뺐는데 오히려 잔액이 늘어나는 버그가 발생한다.

```c
int main(void)
{
        int balance = 10000;    /* 현재 잔액 10,000원 */
        int withdraw = -50000;  /* 해커의 공격: 음수 금액 출금 시도! */

        /* [보안 패치] 입력값 검증: 0 이하의 값은 즉시 차단한다. */
        /* 이 코드가 없으면 해커는 부자가 된다. */
        if (withdraw <= 0) {
                /* [경고] 잘못된 금액입니다. (해킹 시도 탐지) */
        } 
        else if (withdraw > balance) {
                /* [오류] 잔액이 부족합니다. */
        }
        else {
                /* 모든 검증을 통과한 안전한 경우에만 계산한다 */
                balance = balance - withdraw;

                /* 출금 완료. 남은 잔액: balance */
        }

        return 0;
}
```
이처럼 `if` 문을 사용한 범위 검사는 단순한 문법이 아니라, 시스템의 논리적 결함 _logical flaw_ 를 방어하는 가장 강력한 방패다.

> [!NOTE]
> 
> **보안 관점:** 모든 입력을 의심하라
>
> 보안 관리자는 유저가 입력한 데이터를 절대 믿지 않는다 _never trust user input_.
> 유저가 실수로, 혹은 악의적으로 엉뚱한 값을 넣을 것이라고 항상 가정하고 방어 코드를 작성해야 한다.

### 6.6.4 값 잘라내기: 범위를 벗어나면 강제로 고정하기 _clamping_
값이 너무 작거나 너무 클 때, 범위를 벗어났다고 에러를 내는 대신 강제로 범위 안으로 끌고 들어오는 방법도 있다. 
전문 용어로 클램핑 _clamping_ 이라고 한다.

```c
int main(void)
{
        /* 유저에게 데이터를 전송받은 후 같은 데이터를 보내서 네트워크 상태를 확인한다. */
        /* 현재 유저가 다시 전송받을 데이터를 claim_size로 요청한 상태다. */

        int real_size = 10;                /* 실제로 전송받은 데이터 크기 */
        int claim_size = 65535;            /* 유저가 요청한 다시 전송받을 데이터 크기 */
        
        /* [보안 패치] 유저가 요청한 양이 전송받은 양보다 크면, 실제 양으로 강제 고정한다. */
        if (claim_size > real_size) {
                claim_size = real_size; 
        }

        /* claim size 만큼의 데이터를 유저에게 다시 전송한다 */
                
        return 0;
}
```

> [!NOTE]
> **보안 역사: 범위를 벗어남 _out-of-bounds, OOB_ 와 하트블리드 _heartbleed_ 버그**
>
> 2014년, 전 세계 서버의 비밀번호를 유출시켰던 전설적인 버그 하트블리드 _heartbleed, CVE-2014-0160_ 는 클램핑 _clamping_ 을 하지 않아서 발생했다.
>
> 1. **해커:** "자, 데이터 1바이트 보낼게. 근데 이거 길이는 64KB라고 생각해줘. 그리고 그만큼 데이터를 돌려줘."
> 2. **서버 - 버그 있음 - :** 길이를 의심하지 않는다. "어? 64KB? 알았어."
> 3. **결과:** 해커가 보낸 1바이트 뒤에 있는 서버의 메모리 64KB를 복사해서 해커에게 전송해버린다. 계속 복사해서 보내다보면 비밀번호, 개인 정보 등도 유출된다..
>
> 이 버그는 메모리 범위를 벗어나는 _out-of-bounds, OOB_ 중에서도 할당된 메모리보다 더 많이 읽어버리는 _buffer-over-read_ 취약점이었다.
> `if (요청길이 > 실제길이)` 한 줄만 있었어도 막을 수 있었다.

### 6.6.5 가드 클로즈: 입구에서 컷하기 _guard clause_

조건이 많아지면 코드는 점점 오른쪽으로 밀려난다. 흔히 피라미드 코드 _pyramid code_ 라고 부른다.

**나쁜 예: 피라미드 코드**
```c
if (is_logged_in) {
        if (is_admin) {
                if (has_permission) {
                        /* 실제 중요한 로직이 저 안쪽에 숨어 있음 */
                        execute_critical_system();
                } else {
                        return ERROR_PERMISSION;
                }
        } else {
                return ERROR_NOT_ADMIN;
        }
} else {
        return ERROR_NOT_LOGGED_IN;
}
```

보안 관점에서 이런 코드는 치명적이다. 
가장 중요한 핵심 로직이 여러 겹의 중괄호 속에 파묻혀 있어, 개발자가 실수로 조건을 빼먹거나 로직을 잘못 수정할 확률이 높아진다.

**좋은 예: 가드 클로즈 _guard clause_ 패턴**

가드 클로즈는 예외 상황을 입구에서 즉시 되돌려 보내는 `return` 방식이다.
본 게임인 핵심 로직에 들어가기 전 보안 검색대에서 부적격자를 먼저 쫓아내는 것과 같다.

```c
/* 1. 로그인이 안 되어 있으면 즉시 차단 */
if (not_logged_in) return ERROR_NOT_LOGGED_IN;

/* 2. 관리자가 아니면 즉시 차단 */
if (not_admin) return ERROR_NOT_ADMIN;

/* 3. 권한이 없으면 즉시 차단 */
if (no_permission) return ERROR_PERMISSION;

/* [핵심 로직] 여기까지 왔다면 보안 검사를 모두 통과한 안전한 상태다! */
```

피라미드 코드와 비교했을 때, 가드 클로즈는 다음과 같은 이유에서 보안에 더 좋다.

- **가독성:** 핵심 로직이 들여쓰기 없이 명확하게 보인다.
- **실수 방지:** `else`의 짝을 맞추느라 고생할 필요가 없다.
- **안정성:** 검증 후 실행 단계가 선형적으로 진행되어 논리의 허점을 찾기 쉽다.

### 6.6.6 오버플로우 예방: 사고가 나기 전에 멈추기

5장에서 우리는 `int` 그릇에 너무 많은 값을 담으면 넘쳐흐르는 오버플로우 _overflow_ 현상을 배웠다.
많은 입문자가 "더한 뒤에 검사하면 되지 않을까?"라고 생각한다.

```c
int sum = a + b; /* a와 b가 양수라고 가정한다 */
if (sum < 0) { /* 오버플로우 감지? 이미 늦었다! */ }
```

하지만 보안 프로그래밍에서 이미 엎질러진 물은 주워 담을 수 없다. 오버플로우가 발생한 순간 값은 이미 망가졌고 *undefined behavior*, 해커는 그 찰나를 노린다.
따라서 더하기 전에 덧셈이 안전할지 미리 확인해봐야 한다.

**안전한 덧셈 검사 공식**

수학적으로 $a + b > MAX$라면 오버플로우다. 
좌변에 $b$를 이항하면, $a > MAX - b$가 된다.
$b$가 양수라면, $MAX - b$를 할 때는 오버플로우가 나지 않기 때문에, 
이 원리를 이용해서 덧셈이 오버플로우가 될지를 안전하게 확인할 수 있다.

```c
int main(void)
{
        int INT_MAX = 2147483647;         /* int가 32비트인 경우를 가정했을 때 표현 가능한 최대값 */
        int current = 2000000000;
        int add = 1000000000;

        /* 보안 패턴: 더하면 INT_MAX를 넘을지 미리 확인한다 pre-check */
        /* a > MAX - b */
        if (current > INT_MAX - add) {
                /* 차단. 더하면 오버플로우가 발생하므로 연산을 수행하지 않는다. */
        } else {
                /* 안전. 검사를 통과했으므로 더해도 안전하다. */
                current = current + add;
        }
    
        return 0;
}
```

> [!NOTE]
> 
> **보안 관점: 방어적 프로그래밍 _defensive programming_**
> 
> 자동차에 에어백이 있더라도, 가장 좋은 것은 사고가 나지 않게 방어 운전을 하는 것이다.
> 오버플로우 검사는 프로그램이 충돌하기 전에 위험을 감지하고 회피하는 대표적인 방어 운전 코드다.

## 6.7 조건을 조립하는 논리 연산자: `&&`, `||`, `!`

조건을 만들다 보면 "A도 옳아야 하고 B도 옳아야 한다", "A나 B나 다 안 된다", "A가 틀리거나 B가 틀리거나 둘 다 틀려야 한다" 같은 말을 코드로 옮겨야 한다.

C언어는 이를 위해 다음과 같은 논리 연산자들을 제공한다.

- **`!` 논리 부정 _not_:** 참을 거짓으로, 거짓을 참으로 바꾼다.
- **`&&` 논리 곱 _and_:** 두 조건이 모두 참일 때만 참이 된다.
- **`||` 논리 합 _or_:** 두 조건 중 하나라도 참이면 참이 된다. 

비교 연산 `>`, `<`를 한 결과는 조건이 맞으면 무조건 `1`, 틀리면 무조건 `0`이었다.

```c
10 > 3;   /* 비교 식의 결과값은 참에 해당하는 1이다. */
3 > 10;   /* 비교 식의 결과값은 거짓에 해당하는 0이다. */
```

논리 연산자 `!`, `&&`, `||`의 결과도 마찬가지로 항상 `1` 또는 `0`으로 나온다. 

### 6.7.1 `!` : 조건 뒤집기 _not_

`!` 연산자는 참을 거짓으로, 거짓을 참으로 바꾼다.

- **`!0`은 `1`:** 거짓을 참으로
- **`!123`은 `0`:** 참을 거짓으로

입력이 무엇이든 결과는 무조건 0 또는 1이 된다. 

한글 문장으로 비유하면 "...이 아니면"이고, 영어 문장으로 비유하면 "not ..." 이다. 

```c
/* found 변수에는 찾았을 때 1, 못찾았을 때 0이 저장되어 있다고 가정한다. */
if (!found) {
        /* 찾지 못했다면 if not found */
}
```

> [!NOTE] 
> 
> **괄호는 선택이 아니라 필수**
>
> `!`는 연산자 우선순위가 매우 높다. 
> 따라서 복잡한 조건을 뒤집을 때는 반드시 괄호로 감싸야 한다.
>
> * `!A && B` : `A`만 뒤집은 결과를 `B`와 _and_ 연산. 
> * `!(A && B)` : `A`와 `B`의 결과를 먼저 구해서 _and_ 연산을 하고 전체를 뒤집기.

### 6.7.2 `&&` : 둘 다 참 _and_

`&&` 연산자는 두 조건이 모두 참 _true_ 일 때만 결과가 `1`이 된다. 하나라도 거짓이면 결과는 `0`이다.

- **`참 && 참`은:** `1`
- **`참 && 거짓`은:** `0`
- **`거짓 && 거짓`은:** `0`

가장 대표적인 사용처는 범위 검사다.

```c
if (x >= 0 && x <= 9) {
        /* x가 0 이상이고, 그리고 9 이하일 때 */
}

```
> [!CAUTION]
> **수학식처럼 썼다간 큰일 난다**
>
> C 언어 문법을 처음 익힐 때 많이 하는 실수가 범위를 수학식처럼 한 줄에 쓰는 것이다.
>
> ```c
> /* 문법 에러는 안 나지만, 논리 에러 발생 */
> if (0 <= x <= 9) { ... }
> ```
>
> 비교 연산을 연달아 쓴 코드가 위험한 이유는 컴파일 에러가 뜨지 않고 실행되기 때문이다. 
> 컴퓨터는 이 식을 앞에서부터 순서대로 계산해버린다.
>
> 1.  `0 <= x`를 먼저 계산한다. 결과는 무조건 `0` 또는 `1`이다.
> 2.  그 결과인 `0` 또는 `1`을 가지고 다시 `9`와 비교한다 `<= 9`.
> 3.  그런데 `0`도 `9`보다 작고, `1`도 `9`보다 작다.**
>
> 결국 `x`가 100이든 -50이든 이 조건식은 언제나 참이 되어 버린다. 
> 따라서 C언어에서는 반드시 `&&`를 이용해 두 조건을 접착제로 붙여줘야 한다.

### 6.7.3 `&&`로 중첩 `if`문을 한줄로

지금까지 배운 방식으로는 범위를 검사할 때 `if`를 중첩해서 쌓아야 했다. 10대인지 확인하기 위해 첫번째 `if`에서 나이가 `10` 이상인지를 확인하고, 두번째 `if`에서 나이가 `19` 이하인지를 확인했다.

```c
/* 중첩 if는 깊고 복잡하다 */
if (age >= 10) {
        if (age <= 19) {
                /* 여기서 age는 10대 */
        }
}
```

우리에게는 이제 `&&`라는 강력한 접착제가 생겼다. 두 개의 `if`를 굳이 따로 만들 필요 없이 한 줄로 합칠 수 있다.

```c
/* &&를 쓰면 깔끔하게 한 줄이다 */
if (age >= 10 && age <= 19) {
        /* 여기서 age는 10대 */
}
```

무엇이 좋아졌을까?

1.  **깊이가 얕아짐:** 불필요한 들여쓰기가 사라져 코드가 평평해졌다.
2.  **가독성:** "10세 이상 그리고 19세 이하"라는 의도가 한눈에 들어온다.

프로그래밍에서는 이렇게 결과는 같지만 내부 구조를 더 효율적이고 깔끔하게 고치는 과정을 리팩터링 _refactoring_ 이라고 한다. 문법을 하나씩 익힐 때마다, 여러분의 코드는 더 짧고 강력해질 것이다.

### 6.7.4 `||` : 하나라도 참이면 참 _or_

`||` 연산자는 `&&`보다 훨씬 관대하다. 두 조건 중 어느 한쪽이라도 참이면 결과는 `1`로 참이 된다. 둘 다 거짓일 때만 `0`인 거짓이 된다.

- **`참 || 거짓`은:** `1`
- **`거짓 || 참`은:** `1`
- **`거짓 || 거짓`은:** `0`

주로 양자 선택 _or_ 의 상황을 표현할 때 사용한다.

**범위 밖을 검사할 때**

`&&`가 "0에서 100 사이"를 검사했다면, `||`는 그 반대인 "범위를 벗어난 아웃라이어"를 잡아낼 때 유용하다.

```c
/* 0보다 작거나, 100보다 크면 잘못된 점수 */
if (score < 0 || score > 100) {
        /* 에러 처리하기 */
}
```

**여러 선택지를 검사할 때**

사용자의 입력이 소문자 'q'일 수도 있고, 대문자 'Q'일 수도 있다. 둘 중 하나만 맞아도 종료해야 한다면 이렇게 쓴다.

```c
/* 'q'를 눌렀거나, 'Q'를 눌렀다면 */
if (input == 'q' || input == 'Q') {
        return 0; /* 종료 */
}
```

> [!NOTE]
> **컴퓨터는 게으르다**
>
> `&&`와 `||`는 결과가 이미 정해지면 오른쪽을 보지 않을 수 있다.
> 단축 _short-circuit_ 평가라고 부르며, 6.7에서 안전 규칙으로 정리한다.

### 6.7.5 `||`로 반복되는 `else if` 정리하기

`&&`가 중첩 `if`로 인해 깊어지는 들여쓰기를 해결해 줬다면, `||`는 같은 목적의 `else if`로 인해 반복되는 코드를 해결해 준다.

지금까지의 방식에서는 잘못된 값을 골라낼 때, 에러 처리 코드를 여러 번 복사해서 붙여넣어야 했다.

```c
/* [Before] 중복되는 코드 */
if (score < 0) {
        /* 잘못된 값에 대한 에러 처리 */
}
else if (score > 100) {
        /* 잘못된 값에 대한 똑같은 에러 처리 */
}
```
"0보다 작거나" 또는 "100보다 크면" 에러라는 목적은 같은데, 
에러를 처리하는 코드를 반복해서 쓰는 것은 비효율적이다. 
`||` 연산자를 쓰면 하나의 `if`에서 에러를 처리할 수 있다.

```c
/* || 연산자를 쓰면 하나의 if로 통합된다 */
if (score < 0 || score > 100) {
    /* 잘못된 값에 대한 에러 처리 한번만 */
}
```

무엇이 좋아졌을까?

1.  **중복 제거:** 똑같은 에러 메시지를 두 번 쓸 필요가 없다. 나중에 에러 메시지를 고칠 때도 한곳만 고치면 된다.
2.  **논리 통합:** "정상 범위를 벗어난 경우"를 하나의 블록으로 묶어서 관리할 수 있다.

`||` 연산자는 6.4장에서 봤던 `don't repeat yourself` 원칙을 지키는 가장 쉬운 방법이다.

### 6.7.6 우선순위와 괄호: 명확성

논리 연산자 사이에도 우선순위가 존재한다.

1. `!`는 우선순위가 높다.
2. `&&` 논리 곱은 곱셈과 비슷하다.
3. `||` 논리 합은 덧셈과 비슷하다.

수학에서 덧셈보다 곱셈을 먼저 계산하듯, C언어도 `&&`를 `||`보다 먼저 처리한다. 
하지만 이 순서를 억지로 외우려 애쓸 필요는 없다. 
가장 좋은 규칙은 `&&`와 `||`가 섞이면 무조건 괄호로 묶는 것이다.

예를 들어, 다음 코드를 보자.

```c
if (a && b || c) { ... }
```

컴퓨터는 우선순위 규칙에 따라 `(a && b)`를 먼저 하고 나중에 `c`를 본다. 
하지만 코드를 읽는 사람은 "뭐가 먼저지?" 하고 멈칫하게 된다.

그래서 의도를 명확히 드러내기 위해 괄호를 쓴다.

```c
/* 의도: (a와 b가 참)이거나, 혹은 c가 참이면 */
if ((a && b) || c) { ... }

/* 의도: a는 무조건 참이고, (b나 c 중 하나)가 참이면 */
if (a && (b || c)) { ... }
```

괄호 몇 글자를 더 적는 수고가, 나중에 코드를 읽을 때 생길 오해를 막아준다.

`x >= 0 && x <= 9`에서는 비교 연산자(`>=`, `<=`)가 우선순위에 따라 먼저 계산되고, 
그래서 괄호 없이도 동작한다. 
하지만 우리는 사람이 읽기 쉽게 괄호를 쓰는 것을 선택할 수도 있다. 

## 6.8 단축 평가 _short-circuit evaluation_: 결과가 뻔하면 뒤는 안 본다

논리 _and_, _or_ 연산자는 조건식 안에서 어디까지 검사할지 결정하는 작은 흐름 제어 기능도 한다.

`&&`와 `||`는 왼쪽부터 평가하며, 왼쪽에서 결과가 결정되면 오른쪽은 실행하지 않는다.

### 6.8.1 `&&`의 단축 평가: 보호 문법 _guard idiom_

`&&` 연산자는 왼쪽 피연산자의 계산 결과가 `0`으로 거짓이면, 오른쪽 피연산자의 값과 관계 없이 무조건 결과가 `0`이므로, 오른쪽 피연산자는 계산하지 않는다. 오른쪽에서 할 위험한 연산을 보호하기 위해서 왼쪽에서 안전 검사를 하는데 쓰인다. 

```c
/* 안전한 코드 패턴 */
if (y != 0 && x % y == 0) {
        /* 1. y가 0이 아닌지 먼저 확인하고, 0이면 중단) */
        /* 2. 0이 아닐 때만 x % y를 계산해서 x가 y의 배수인지 확인 */
}
```

`x % y`를 하기 전에, `y`가 `0`이라면 나누기를 하지 않고 결과값을 `0`으로 낸다. 

### 6.8.2 `||`의 단축 평가: 정답을 찾으면 즉시 중단

반대로 `||` 연산자는 왼쪽 조건만 참이면 결과값이 참이 된다. 따라서 왼쪽 조건이 참이면, 오른쪽은 검사하지 않는다. 무거운 작업을 건너뛰는 최적화에 자주 쓰인다. 

```c
/* 예시: VIP 회원이면 입장료 결제 함수를 실행하지 않음 */
if (is_vip || make_payment_and_check()) {
    enter_gate();
}
```

1. `is_vip`가 참(1)이라면:
   이미 입장이 확정되었다.
   뒤에 있는 `make_payment_and_check()` 함수는 호출되지 않는다.
2. `is_vip`가 거짓(0)이라면:
   그제야 뒤쪽의 `make_payment_and_check()`를 실행해서 결제를 시도한다.
3. `make_payment_and_check()`를 실행한 결과 결제가 완료되어 값이 1이면:
   입장한다.

`||`는 앞에서 해결되면 뒤는 신경 끄는 효율적인 문지기 역할을 한다.

### 6.8.3 `&&`와 `||`를 쓸 때는 변수를 지켜주자

단축 평가 때문에 생기는 가장 흔하고 치명적인 실수가 있다. 어떤 경우에도 실행해야 하는 코드를 오른쪽에 넣으면 단축 평가로 인해 실행되지 않을 수 있다.

앞서 본 `is_vip` 예제에서, 누가 입장하든 방문 횟수를 1 증가시키고 싶다고 가정해 보자.
 
```c
int visits = 0;
int is_vip = 1; /* VIP 회원 */

/* 의도: VIP거나 일반인이거나, 일단 방문자 수를 늘리고 입장 */
if (is_vip || (visits = visits + 1) > 10) {
        /* is_vip인 경우 (visits = visits + 1) > 10 이 실행되지 않는다. */
}
```

vip가 입장했을 때 `visits`는 어떻게 될까?

* **기대:** vip가 입장했으니 `visits`의 값이 `1` 증가.
* **실제:** `visits`는 여전히 변하지 않음.

이유는 단순하다. `is_vip`가 `1`로 참이므로, 컴퓨터는 오른쪽 식 `(visits = visits + 1) > 10`을 아예 쳐다보지도 않고 건너뛰었기 때문이다. 결국 vip 회원들은 통계에서 누락되는 유령 회원이 되어버린다.

가장 안전한 것은, 값이 변해야 하는 작업을 논리 계산 바깥으로 꺼내는 것이다.

```c

/* vip거나 일반인이거나, 일단 방문자 수를 늘리기 */
visits = visits + 1;

/* 조건 처리에는 그 다음에 */
if (is_vip || visits > 10) {
        /* visits에는 현재 누적 방문자 수가 저장되어 있다. */
}
```

## 6.9 초보자가 가장 많이 하는 실수 2가지: 중괄호 생략과 비어있는 문장

C 언어를 처음 배울 때, 코드가 에러 없이 컴파일은 되는데 프로그램이 이상하게 동작하는 경험을 하게 된다. 
문법을 몰라서가 아니라, C 언어의 유연함이 독이 되는 순간이다.

가장 치명적인 2가지 함정을 미리 파악하고 피하는 법을 배워보자.

### 6.9.1 중괄호 `{}` 생략하기: 눈을 속이는 착시 현상

C 언어 문법에는 `if` 뒤에 실행할 문장이 딱 한 줄이면 중괄호를 생략해도 된다는 규칙이 있다. 하지만 우리는 이 규칙을 머릿속에서 지워버리는 것이 좋다.

코드를 간결해게 만들어주는 장점보다, 착시 현상을 일으켜 버그를 만드는 단점이 훨씬 크기 때문이다.

```c
if (x > 0)
        x = x + 1;     /* if에서 처리됨 */
        x = x + 10;    /* 들여쓰기를 했지만, if와 관계 없음. 항상 실행됨 */
```

사람 눈에는 들여쓰기 때문에 두 줄 모두 `if`에 묶인 것처럼 보인다. 
하지만 C 컴파일러 그리고 CPU는 들여쓰기나 의도는 보지 않고 오직 문법만 본다. 

```c
if (x > 0)
        x = x + 1;     /* if에서 처리됨 */

x = x + 10;    /* 관계 없이 항상 실행되는 문장 */
```

들여쓰기가 다른 코드들은 사람 눈으로 보기에는 차이가 나지만, 컴파일러가 보기에는 똑같다.
그래서 눈에 잘 띄지 않는 심각한 버그가 발생할 소지가 있다.

C 언어에서 들여쓰기는 문법이 아니라 가독성을 위한 관습일 뿐이다.

> [!NOTE]
> **보안 역사: 애플의 '`goto fail` 버그**
>
> 2014년, 애플의 아이폰과 맥 운영체제에서 치명적인 보안 취약점이 발견되었다 _CVE-2014-1266_.
> 개발자가 `if`로 에러 확인을 한 후에 실패 위치로 점프하는 코드를 두 번 붙여넣은 것이다.
>
> ```c
> int err = 0;                    /* 보안 문제가 없으면 값이 0이고, 문제가 있으면 0이 아닌 변수 */
> 
> if (보안_검사를_할_필요가_없으면) 
>         종료_위치로_이동;
>         종료_위치로_이동;        /* 문제의 코드: if 문 밖이라서 무조건 실행됨! */
>
> /* 여기서 최종 보안 검사를 해야 하는데... 건너뜀! */
> err = 중요한_보안_검사_코드();
>
> /* 종료_위치: */
> return err;
> ```
>
> **왜 위험한가?**
>
> 두번째 `종료_위치로_이동` 코드는 `if (보안_검사를_할_필요가_없으면)`과 관계없이 항상 실행되는 코드다.
> 그래서 바로 다음에 있는 `중요한_보안_검사_코드()`가 실행되지 않고 _skip_, 프로그램은 종료 위치로 이동해버린다.
> 문제는 이때 `err` 변수에 문제 없음을 의미하는 `0`이 들어있었다는 점이다.
> 시스템은 "어? 경고가 0개네? 보안 검사 통과!"라고 착각하게 되었고, 해커가 만든 가짜 사이트에도 문을 활짝 열어주는 대형 사고로 이어졌다.
>
> 클램핑을 하지 않은 _heartbleed_와 마찬가지로, 보안 사고는 복잡한 해킹 기술보다 단순한 조건문 실수에서 시작된다.

### 6.9.2 무조건 통과의 함정: `if (조건) ;`

C 언어에서는 문장이 끝날 때마다 세미콜론 `;`을 쓰는 것이 철칙이다. `a = 10;`와 같이 쓴다. 
코드를 짜다 보면 습관적으로 문장 끝에 세미콜론 `;`을 붙이게 된다. 
손에 익은 이 습관 때문에 초보자들은 절대 찍어서는 안 될 곳에도 무의식적으로 세미콜론을 쓰곤 한다. 
바로 `if` 문의 조건식 뒤다.

```c
if (is_admin == 1) ;      /* 관리자인 경우 null 문장인 ;가 실행된다. */
{
        /* 관리자 전용 코드 - 항상 실행된다. */
}
```

신기한 점은, 이 코드는 컴파일 에러가 나지 않는다. 컴파일러는 이 상황을 다음과 같이 해석한다.

1. **`if (is_admin == 1)`:** 관리자인지 검사한다.
2. **`;`:** 조건이 참이면 아무것도 하지 않는다 _null statement_.
3. **`{ /* 관리자 전용 코드 */ }`:** 관리자 전용 코드를 실행하라.

동작이 더 명확하게 보이게 들여쓰기를 해보면 다음과 같다.

```c
if (is_admin == 1)
        ; /* 아무 일도 하지 않는 문장 */

/* 이 블록은 if와 무관하다 */
{
        /* 관리자 전용 코드 */       
}
```

세미콜론 `;` 하나도 C 언어에서는 완전한 문장 _statement_이다.
결국 `if`문은 허공에 발차기를 하고 끝나버리고, 바로 다음에 있는 중요 코드는 보호막 없이 무조건 실행된다. 

> [!NOTE]
> 
> **`if`의 정식 문법**
>
> `else`가 없는 `if`문의 문법을 정식으로 써보면 다음과 같다.
>
> `if` `(` _수식_ `)` _문장_
>
> 여기서 문장의 종류로 세미콜론 `;`만 있는 null 문장 _null statement_, 중괄호 `{}`로 묶인 복합 문장 _block_ 등이 올 수 있다.
> 컴파일러 입장에서 `;`는 아무것도 하지마라는 유효한 명령이다.

> [!TIP]
> 
> **컴파일러의 경고를 켜자**
>
> 이 실수는 눈으로 찾기 어렵기 때문에, 컴파일러의 잔소리를 들어야 한다.
> 사용 중인 환경에 맞춰 경고 레벨을 최대로 높여보자.
>
> - gcc, clang 사용자: 컴파일 명령어에 `-Wextra` 옵션을 추가한다.
>   - 예: `gcc main.c -Wall -Wextra`
>   - 비어있는 문장이 발견되면 `warning: if statement has empty body`라고 알려준다.
> - visual studio 사용자: 기본적으로 경고 C4390을 띄워준다.
>   - 더 확실하게 하려면, '프로젝트 속성 메뉴'에서 'C/C++'을 선택하고, '일반'을 선택하고, '경고 수준'을 수준 4인 /W4로 설정한다.
