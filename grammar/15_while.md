# 15. 흐름 제어의 완성: 반복을 유연하게

우리는 6장에서 `if`문을 배우며 프로그램에 갈림길을 만들었다. 덕분에 무조건 직진만 하던 코드가 상황에 따라 다른 길을 선택할 수 있게 되었다. 
7장에서는 `for`문을 통해 "횟수가 보이는" 반복을 구현했다. 1부터 10까지 더하거나, 배열을 순회하는 일은 이제 익숙해졌다.

그런데 현실의 문제는 조금 더 복잡하다. 코드를 짜다 보면 다음 두 가지가 부족해 코드가 지저분해지는 순간이 온다.

**언제 끝날지 모르는 반복**: 사용자가 '종료' 버튼을 누를 때까지, 혹은 파일의 끝에 닿을 때까지 반복해야 한다면, 횟수를 정해두는 `for`문은 어울리지 않는다.

또 하나의 문제가 있다. 반복문 안에서 `i = i + 1` 같은 코드를 무심코 섞어 썼다가, **부수 효과**라는 함정에 빠져 원인을 알 수 없는 버그를 만들기도 한다.

그동안 미뤄두었던 흐름 제어의 나머지 퍼즐 조각을 맞추는 시간이 왔다. 새로운 도구들을 익히면 여러분의 논리는 더 단단해지고, 코드는 더 안전해질 것이다.

## 15.0 목표

15장의 목표는 다음과 같다. 

- **유연한 반복 제어:** 횟수가 정해지지 않은 상화에서 `while`과 `do`-`while`을 적재적소에 활용하여 프로그램을 제어할 수 있다.

- **부수 효과 _sidef effect_ 의 이해와 안전한 코딩:** 수식의 계산 과정에서 발생하는 부수 효과를 이해하고, `++i`와 `i++`이 섞인 코드에서 발생할 수 있는 미정의 동작을 피하는 안전한 코딩 습관을 기른다.

## 15.1 횟수가 아니라 '조건'일 때: `while`

`for`는 "시작; 조건; 갱신"이 한 줄에 모여 있어서 횟수가 정해진 반복에 아주 강하다. 마치 "운동장 10바퀴 뛰세요"라는 명령과 같다.

하지만 현실의 반복에는 언제 끝날지 모르는 경우도 있다.

- "사용자가 '종료' 버튼을 누를 때까지"
- "파일의 끝이 나올 때까지"
- "정답을 맞힐 때까지"

마치 "비가 올 때까지 운동장을 뛰세요"와 같다. 
몇 바퀴를 돌지는 중요하지 않고, 오직 조건인 비가 오냐 안 오냐만 중요하다.

### 15.1.1 `for`가 어색해지는 순간

특정 상태가 유지되는 동안 반복을 할 때는 `for`문으로 짜면 코드가 어색해진다.

예를 들어, 집에 갈 때가 아닌 동안 `!should_go`, 계속 쇼핑하기를 코드로 짜보자. 

```c
/* for문: 앞뒤가 텅 비어서 어색함 */
for (;!should_go;) {
        if (should_buy()) {
                shopping();
                extra_money -= price();
                update_status_after_buy();
        }
        else {
                update_status_no_buy();
        }
}
```

문법적으로는 틀리지 않았지만, 읽기가 불편하다. 여기서 `for` 대신 사용하는 것이 바로 `while`이다.

```c
/* while문: 문장처럼 자연스럽다 */
while (!should_go) {
        if (should_buy()) {
                shopping();
                extra_money -= price();
                update_status_after_buy();
        }
        else {
                update_status_no_buy();
        }
}
```

코드를 읽어보면 집에 갈 때가 아닌 동안 `while` ..., 계속한다라고 영어 문장처럼 자연스럽다.

**`while`은 반복되는 `if`문이다**

구조를 뜯어보면 `while`문은 `if`문과 형제자매다. 

- **`if`:** 조건이 참이면 한 번 실행하고 끝.
- **`while`:** 조건이 참이면 실행하고, 다시 조건 검사로 되돌아가기. 

`while`은 어렵게 생각할 것 없이 조건이 맞으면 계속 뱅글뱅글 도는 `if`문이라고 보면 된다.

### 15.1.2 `for`문과 `while`문의 변환 공식

`for`문은 마법이 아니다. 내부적으로는 `while`문과 거의 같은 구조로 동작한다.

```c
/* for 문의 구조 */
for (초기수식; 조건수식; 업데이트수식) {
        반복할_코드;
}
```

`while`문으로 풀어서 쓰면 다음과 같다. 실행 순서를 눈여겨보자.

```c
/* while 문으로 변환 */
초기수식;

while (조건수식) {
        반복할_코드;
        업데이트수식;      /* 코드가 끝나면 업데이트 */
}
```

예를 들어 `0`부터 `2`까지 출력하는 코드를 비교해 보자.

```c
/* for는 한 줄에 전부 있어서 깔끔하다 */
int i;
for (i = 0; i < 3; i = i + 1) {
        /* 반복 작업 */
}

/* while은 흩어져있다 */
int i;
i = 0;                  /* 초기수식 */
while (i < 3) {         /* 조건수식 */
        /* 반복 작업 */
        i = i + 1;      /* 업데이트수식 */
}
```

반복 횟수가 명확할 때는 `for`가 훨씬 안전하고, 조건에 따라 흐름이 복잡할 때는 `while`로 펼쳐 쓰는 것이 좋다.

> [!WARNING]
> **`for`와 `while`의 가장 큰 차이점**
>
> `for`를 `while`로 바꿀 때 딱 하나 조심해야 할 것이 있다.
> 바로 `continue`다.
>
> - **`for`:** `continue`를 만나면 업데이트수식 `i = i + 1`로 점프해서 다음 순서를 준비한다.
> - **`while`:** `continue`를 만나면 업데이트수식 `i = i + 1`을 건너뛰고 바로 조건수식으로 간다.
>
> `continue`를 사용하는 `while`문을 짤 때는 업데이트수식의 위치를 각별히 신경 써야 한다.

### 15.1.3 끝을 모르는 반복 패턴 2종: 선검사와 무한루프

`for`문이 횟수를 정해놓고 뛰는 것이라면, `while`은 상황에 따라 유연하게 대처하는 반복이다. 실무에서는 크게 두 가지 패턴을 쓴다.

**조건 기반 선 검사**

가장 정석적인 방법이다. 시작하기 전에 조건부터 본다는 특징이 있다.

```c
int money = 1000;

/* 돈이 없으면 아예 시작도 안 한다 */
while (money > 0) {
        /* 쇼핑을 한다 */
        money = money - 100;
}
```

**무한 루프 + `break` 중간 검사**

먼저 돌리고 본다. 그러다 특정 조건이 되면 탈출 `break`한다.
작업을 해봐야 끝낼지 말지 알 수 있을 때 유용하다.

```c
int input = 0;

/* 1은 참이므로 무한히 반복 */
while (1) {
        /* 숫자를 입력받아서 input에 저장한다 */

        if (input == 0) {
                /* 종료한다 */
                break; /* 반복문을 박차고 나간다 */
        }

        /* 입력값을 처리한다 */
}
```

왜 무한루프 + `break` 패턴을 쓸까? 

같은 프로그램을 선 검사 패턴으로 짤 경우, 코드가 중복되기 때문이다. 

```c
int input = 0;

/* 숫자를 입력받아서 input에 저장한다 */

while (input != 0) {

        /* 입력값을 처리한다 */

        /* 숫자를 입력받아서 input에 저장한다 */

}
```

`input`을 검사하기에 앞서 입력을 받아야 하기 때문에 `input`에 입력을 받는 코드가 초기수식 자리와 업데이트수식 자리에 중복해서 들어가게 된다. 입력 안내문을 수정할 일이 생기면 두 곳을 모두 고쳐야 하니 번거롭고 실수하기 쉽다.

무한루프와 `break`를 섞어 쓴 경우, 입력을 받는 코드가 깔끔하게 하나로 통합된다. 먼저 입력을 받고, 검사해서 멈출지 결정하고, 멈출 일이 아니면 처리한다는 인간의 자연스러운 사고방식과도 잘 맞아떨어진다.

반복의 중간에서 조건을 검사하고 빠져나오는 구조를 루프 앤드 어 하프 _loop-and-a-half_ 패턴이라고 부르며, 실무에서 입출력을 다룰 때 흔하게 사용한다.

> [!NOTE]
> **`while (1)`과 `while (true)`** 
>
> C언어에서는 `0`이 아닌 모든 숫자를 참으로 간주하므로, 
> C 프로그래머들은 관습적으로 가장 짧은 `while(1)`을 즐겨 쓴다.
> 만약 C99 이상의 최신 스타일을 선호한다면` <stdbool.h>`를 `include`하고, 
> `while (true)`라고 써도 된다. 
> 컴퓨터 입장에서는 완전히 똑같이 동작한다.

### 15.1.4 `while`에서 `continue` 쓰기

7장에서 배운 `break`와 `continue`는 `while`에서도 똑같이 쓸 수 있다. 
하지만 `continue`를 쓸 때는 각별한 주의가 필요하다. 
점프하는 목적지가 `for`와 미묘하게 다르기 때문이다.

- **`for`의 `continue`:** 증감식으로 점프해서 안전하다.
- **`while`의 `continue`:** 조건식으로 바로 점프해서 위험하다.

`while` 안에서 `continue`를 잘못 쓰면 업데이트하는 코드를 건너뛰어 버리는 사고가 발생한다.

**흔한 반복문:**
```c
/* i를 증가하며 반복하고, 묶음 변수의 상자에 저장된 값이 0이 아니면 데이터 처리를 한다. 묶음 변수 안에 0이 저장된 상자가 반드시 존재한다고 가정한다. */
int i = 0;
while (i < n) {
        if (a[i] != 0) {
                /* 데이터 처리 */
        }
        i = i + 1;
}
```

**흔한 실수: 제자리걸음 무한 루프**

다음 코드를 보자. 데이터가 0인 경우를 건너뛰고 싶었지만, 결과는 끔찍하다.

```c
/* 나쁜 예: i 증가가 맨 아래에만 있는 경우 */
int i = 0;
while (i < n) {
        if (a[i] == 0) {
                /* 여기서 조건확인으로 바로 점프! i는 그대로 0인 상태 */
                continue;
        }
        /* 0이 아닌 값을 처리하는 코드 */
        i = i + 1;   /* 이 코드가 실행되지 않아서, i는 영원히 0 */
}
```

`a[i] == 0`인 상자에 도달했을 때, `i`가 증가하지 않은 채로 다시 검사하러 가므로, 그 다음부터 컴퓨터는 제자리걸음만 하며 영원히 `a[i]`만 쳐다보게 된다.

**해결책: 수동으로 밀어주기**

이를 고치려면 `continue`를 하기 직전에 수동으로 진행 `i = i + 1`을 시켜줘야 한다.

```c
/* 수정: continue 전에도 발걸음을 떼게 해준다 */
int i = 0;
while (i < n) {
        if (a[i] == 0) {
                i= i + 1;    /* continue 전에 잊지 말고 i 업데이트 */
                continue;
        }
        /* 0이 아닌 값을 처리하는 코드 */
        i = i + 1;           /* 평소의 업데이트 */
}
```

> [!NOTE]
> **`while`에서 `continue`를 할 때 코드가 지저분해 보인다면?**
>
> `while`에서 `continue`를 할 때마다 그전에 `i = i + 1`로 업데이트를 하면 중복 코드로 깔끔하지 않다.
> 인덱스 `i`를 규칙적으로 증가시키는 경우라면, 실수할 여지가 없는 `for`문을 쓰는 것이 가장 안전하다.

### 15.1.5 `while`로 하는 스캔: "찾으면 멈추기"

`while`은 데이터를 처음부터 끝까지 훑는 스캔 작업에 특히 강하다. `for`문이 정해진 횟수를 채우는 느낌이라면, `while`은 조건을 만족할 때까지 걷는 느낌이다.

대표적인 사용처는 두 가지다.

- **배열 검색:** 원하는 값을 찾으면 즉시 멈춤
- **문자열 처리:** 끝 `\0`을 만날 때까지 걷기

**배열에서 값 찾기**

여기서 앞서 배운 `&&`의 단축 평가가 빛을 발한다. 범위를 넘지 않았는지와 값을 찾았는지를 한 줄에 검사할 수 있다.

```c
int i = 0;
/* 범위 안에 있고, 아직 못 찾았다면, 계속 가라 */
while (i < size && arr[i] != target) {
        i = i + 1;
}
```

`while`이 종료되는 경우는 딱 두 가지뿐이다.

- **찾았을 때:** `arr[i] == target`이 되어 멈추고, `i`는 찾은 위치다.
- **못 찾았을 때:** `i == size`가 되어 멈추고, `i`는 배열 끝 경계다.

따라서 반복문이 끝난 뒤에 `i`가 어디에 멈췄는지만 보면 된다.

```c
if (i < size) {
        /* arr[i]에서 찾았다 */
} else {
        /* 못찾았다 */
}
```

**문자열 걷기 _sentinel_**

12장에서 배운 문자열도 `while`의 주무대다. 문자열의 길이를 몰라도, 센티널 `\0`이 나올 때까지 계속 가면 되기 때문이다.

```c
const char *p = str;

/* null 문자 '\0'가 아닐 때까지 전진 */
while (*p != '\0') {
        /* 문자 *p 방문중 */
        p = p + 1;
}
```

## 15.2 일단 저지르고 고민할 때: `do-while`

`while`은 조건을 먼저 확인하는 방식이다. 
조건이 처음부터 거짓이면 본문이 단 한 번도 실행되지 않는다. 
마치 놀이공원 입장권 검사를 할 때 표가 없으면 아예 들어갈 수 없는 것과 같다.

그런데 어떤 상황에서는 일단 한 번은 실행해야 한다.

- 사용자에게 메뉴를 일단 보여주고, 선택을 받고 싶을 때
- 비밀번호를 일단 입력받고, 틀렸으면 다시 묻고 싶을 때

그럴 때는 조건 검사를 나중에 하는 `do-while`이 제격이다.
놀이공원 비유를 적용해보면, 일단 들어가서 좋은 시간을 보내고, 나갈 때 "한번 더?"라고 물어보는 것이다.

### 15.2.1 문법과 주의사항

구조는 `while`을 뒤집어 놓은 것과 비슷하다.

```c
do {
        /* 본문: 무조건 한 번은 실행된다 */
} while (조건수식);  /* 여기에 세미콜론 ; 을 꼭 써야한다 */
```

> [!WARNING]
> **`do`의 `while(조건수식);`에서 세미콜론을 잊지 말자**
>
> `if`, `for`, `while` 문법 뒤에는 세미콜론을 붙이지 않지만, `do`-`while`은 유일하게 끝에 세미콜론 `;`을 붙여야 한다. 명령이 여기서 끝난다는 것을 컴파일러에게 알려주는 약속이다.

### 15.2.2 실전 예제: yes/no 선택 받기

사용자가 `y` 또는 `n`을 누를 때까지 계속 메뉴를 보여주는 프로그램이다.

```c
char choice;

do {
        /* y 또는 n를 입력하라는 프롬프트를 보여준다. */
        /* choice에 키보드 입력을 받는다. */
} while (choice != 'y' && choice != 'n'); /* y나 n이 아니면 다시 */

/* choice에는 'y' 또는 'n'이 저장되어 있다. */
```

만약 그냥 `while`로 짰다면, 프롬프트를 보여주고 입력을 받는 코드가 중복으로 들어가게 된다. 

```c
char choice;
/* y 또는 n를 입력하라는 프롬프트를 보여준다. */
/* choice에 키보드 입력을 받는다. */

while (choice != 'y' && choice != 'n') {
        /* y 또는 n를 입력하라는 프롬프트를 보여준다. */
        /* choice에 키보드 입력을 받는다. */
}

/* choice에는 'y' 또는 'n'이 저장되어 있다. */
```

확실히 `do`-`while` 쪽이 코드도 짧고 의도가 명확해진다.

> [!NOTE]
> **논리 연산에서의 드 모르간 법칙 _de morgan's law_**
>
> `choice != 'y' && choice != 'n'` 이 조건은 `choice`가 `y`도 아니고, `n`도 아닐 때 참인 식이다.
> `||`를 써서 만들면 `!(choice == 'y' || choice == 'n')`으로도 쓸 수 있다.

### 15.2.3 상자의 유효범위: 괄호 밖에서 선언하라

`do`-`while`을 처음 쓸 때 가장 많이 만나는 컴파일 에러가 있다. 
바로 상자의 유효범위 문제다.

다음 코드는 아주 그럴싸해 보이지만, 에러가 발생한다.

```c
do {
    int input; /* 여기서 상자 만들기 - 나쁜 위치 */
    
    /* input에 입력을 받아서 사용 */   

} while (input != 0); /* 에러! input이 뭐야? */
```

**왜 에러가 날까**

C언어에서 중괄호 `{ }` 안에서 만든 상자는, 닫는 중괄호 `}`를 만나는 순간 사라진다. 범위 바깥이라 이름도 의미를 잃고, 메모리에서도 저장기간이 종료된다.

그런데 `do-while`의 조건 검사 `input != 0`는 중괄호가 닫힌 바깥쪽에서 일어난다. 
상자 `input`이 소멸된 뒤라 컴퓨터는 `input`이 뭔지 모르겠다고 불평하는 것이다.

**해결책**
조건 수식에서 검사해야 할 변수는 반드시 `do` 블록 바깥 이전에 미리 선언해야 한다.

```c
int input; /* 밖에서 상자를 만들어서 유효한 범위를 늘려준다 */

do {
        /* input에 입력을 받아서 사용 */   
} while (input != 0); /* 유효한 범위 안이라 input이 보인다 */
```

조건 검사에 쓰일 변수는 반복문보다 범위가 같거나 넓어야 한다고 기억하면 쉽다.

## 15.3 부수 효과 _side effect_: 계산의 흔적을 남기다

부수 효과라는 말은 계산 과정에서 일어나는 상태의 변화를 뜻한다.

C언어의 수식 _expression_ 은 두 가지 일을 한다

- **주요 효과 _main effect_:** 계산해서 결과값을 만들어 낸다.
- **부수 효과 _side effect_:** 변수의 값을 바꾸거나 화면에 무언가를 남긴다.

**순수한 계산은 부수 효과가 없다**

다음 식은 값만 만들어낼 뿐, 메모리에 아무런 흔적을 남기지 않는다.

```c
3 + 5;
```

코드는 **8**이라는 값을 계산하지만, 어디에도 저장하지 않는다. 
컴퓨터는 계산하고 나서 결과인 **8**을 쿨하게 버린다. 
아무 일도 일어나지 않은 것과 같다.

**흔적을 남기는 계산은 부수 효과가 있다**

반면에 대입 연산자 `=`는 강력한 부수 효과를 가진다.

```c
a = 20;
```

- **주요 효과 _main effect_:** 수식의 결과값은 **20**이다.
- **부수 효과 _side effect_:** 상자 `a`의 메모리 공간에 **20**이 저장된다.

### 15.3.1 `i = i + 1` vs `i += 1` vs `++i`

프로그래머들은 "변수를 1 증가시키는 일"을 숨 쉬듯이 한다. 그래서 C언어는 이를 위한 다양한 도구를 제공한다.

셋 다 결과적으로 `i`가 **1** 커지고, 커진 값을 결과값으로 낸다.

**`i = i + 1`: 정석**

가장 정직한 표현이다.

- **동작:** `i`에 **1**을 더한 값을 계산하고, 다시 `i`에 넣는다.

- **단점:** `i`를 두 번이나 써야 해서, 변수 이름이 길면 귀찮다. 
  `player_score = player_score + 1`

**`i += 1`: 복합 대입**

"어차피 내 값에 더할 건데 줄여 쓰자"고 해서 나온 것이다.

- **의미:** `i = i + 1`과 완전히 똑같다.

- **장점:** 코드가 짧아지고, 중복을 피할 수 있다. `player_score += 1`

- **확장:** `+= 3`, `+= 10` 처럼 **1**이 아닌 수도 더할 수 있다.

**`++i`: 증가 연산자**

"1 더하는 게 제일 많이 쓰이니까, 아예 전용 문법을 만들자"고 해서 나온 가장 짧은 표현이다.

- **부수 효과:** `i`가 **1** 증가한다.

- **결과값:** 원래 `i`에서 **1** 증가된 값이다.

무슨 말일까? 코드로 확인해 보자.

```c
int i = 5;
int result = ++i;  /* i가 1 증가해서 6이 되고, result에도 6이 저장된다 */
```

-  **결과값:** `i`가 **1** 증가한 **6**을 `result`에 준다.
-  **부수 효과:** `i`도 `6`이 된다.

결과적으로 `result`도 **6**, `i`도 **6**이 된다.

### 15.3.2 `++i` vs `i++`: 타이밍의 싸움

`++`가 상자의 앞에 붙느냐 뒤에 붙느냐에 따라 값을 사용하는 타이밍이 달라진다.

**혼자 쓸 때는 똑같다**

다른 식 없이 단독으로 한 줄에 쓸 때는 둘의 차이가 없다.

```c
i++;  /* i가 1 증가한다 */
++i;  /* i가 1 증가한다 */
```

예를 들면, `for` 문의 증감식에서 `i++`를 쓰든 `++i`를 쓰든 똑같다.

**대입할 때는 다르다**

다른 상자에 값을 대입하거나, 함수에 값을 넘겨주는 등 결과값을 쓸 때는 순서가 중요해진다.

**먼저 증가 `++i`: 변하고 나서 쓴다**

- **순서:** 증가 먼저시키고, 그 값을 사용한다.

- **느낌:** 최신 버전을 가져다 쓴다.

```c
int i = 5;
int a = ++i;
/* 1. i가 6이 된다. 값이 증가한다 */
/* 2. a에 6이 들어간다. 증가된 값을 사용한다 */
```

**나중 증가 `i++`: 쓰고 나서 변한다**

- **순서:** 현재 값을 먼저 쓰고 사용하고, 나중에 증가시킨다.
- **느낌:** 기존 값 백업본을 먼저 주고, 뒤돌아서서 증가한다.

```c
int i = 5;
int a = i++;
/* 1. a에 5가 들어간다. 사용되는 값은 증가하기 전 값이다 */
/* 2. i가 6이 된다. 뒷북으로 증가한다 */
```

> [!NOTE]
> **`i++`와 `++i`를 쉽게 쓰는 방법**
>
> `a = i++` 같은 코드는 한 줄에 두 가지 일 대입과 증가를 모두 하려다 보니 헷갈리는 것이다. 
> 헷갈리면 두 줄로 나누면 된다.
>
> - `a = ++i;` 는 `++i; a = i;`로 쓰면 된다. 
> - `a = i++;` 는 `a = i; i++;`로 쓰면 된다. 


### 15.3.3 대입도 '수식'이다: `while ((c = s[i]) != '\0')`

5장에서 우리는 대입 `=`을 오른쪽 값을 왼쪽에 저장하는 방법이라고 배웠다.
그런데 C언어에서 대입은 단순한 명령이 아니라 값을 반환하는 수식이다.

`x = 3`을 실행하면 두가지 일이 일어난다:
- **저장:** 변수 `x`에는 3을 대입하고,
- **반환:** 결과값으로 3을 뱉어낸다.

두가지 일을 수행하는 특성을 이용하면 대입과 검사를 한 줄로 처리하는 C언어 패턴이 가능해진다.

```c
while ((c = s[i]) != '\0') {
        /* ... */
}
```

이 한 줄의 코드는 내부적으로 세가지 일을 처리한다.

- **대입:** `c = s[i]`는 `c`에 `s[i]` 문자를 저장한다.
- **계산:** 대입 수식의 결과값은 "방금 대입된 문자값"이다.
- **비교:** 그 값이 `\0` null 문자가 아닌지 검사한다.

**왜 대입 수식에 괄호 `(...)`를 썼을까?**

C언어의 연산자 우선순위 때문에 괄호를 썼다.
비교 연산자 `!=`가 대입 연산자 `=`보다 우선순위가 세기 때문에 대입한 결과를 비교하려면 `=`에 괄호를 쳐야 한다.

만약 괄호를 빼먹으면 이런 일이 벌어진다.

```c
while (c = s[i] != '\0')  /* 괄호 실수 */
```

-  우선순위에 따라 `s[i] != '\0'`이 먼저 계산된다. 
   결과는 참이면 **1**, 거짓이면 **0**이다.
-  계산 결과인 **0** 또는 **1**을 변수 `c`에 저장한다.
-  그래서 `c`에는 문자가 아니라 엉뚱한 숫자인 **0** 또는 **1**이 들어가게 된다.

실수를 막기 위해, 컴파일러는 괄호 없이 대입을 조건에 넣으면 경고를 보낸다.

> [!TIP]
> **의도를 괄호로 묶자**
>
> 조건 수식 맥락에서 대입을 할 때는, 
> "지금 일부러 대입하는 거야, 오타 아니야"라는 뜻으로 반드시 전체를 괄호로 감싸자.
>
> - **`if (x = 3)`:** 같은지 비교 `==`를 의도했던 오타처럼 보인다.
> - **`if ((x = 3))`:** 의도된 코드로 보인다.

> [!TIP]
> **너무 어렵다면 풀어서 써도 된다**
>
> 패턴이 너무 복잡해 보인다면, 굳이 억지로 쓸 필요는 없다. 
> 풀어서 쓰는 것이 가독성이 더 좋을 수 있다.
> ```c
> c = s[i];           /* 1. 먼저 대입하고 */
> while (c != '\0') { /* 2. 검사한다 */
>         /* ... */
>         ++i;        /* 3. 루프 변수 i의 값을 하나 증가시킨다 */
>         c = s[i];   /* 4. 루프 끝에서 다시 대입한다 (조금 번거로움) */
> }
> ```
> 하지만 남들이 짠 코드를 읽기 위해서라도 이 패턴 `while((c=...))`은 눈에 익혀두자.

### 15.3.4 고전적인 C 문장: `while ((c = getchar()) != EOF)`

대입 수식을 활용한 패턴이 빛을 발하는 순간이 외부로부터 입력을 받을 때다. 
C 프로그램 코드에서 관용구 _idiom_ 처럼 굳어진 문장을 소개한다.

C언어에는 키보드나 파일에서 글자를 하나씩 읽어오는 `getchar()`라는 함수가 있다. 
보통은 문자를 하나 읽어서 반환하는데, 파일의 끝에 도달하거나 오류가 발생하면 `EOF` _end of file_ 이라는 특별한 값을 뱉어낸다.

입력이 언제 끝날지 모르는 상황에서 모든 문자를 하나씩 읽어 들이려면 어떻게 해야 할까? 바로 여기서 대입과 비교를 동시에 하는 패턴이 등장한다.

```c
int c;

/* 글자를 하나 읽어서 c에 넣고, 그 값이 EOF가 아닌 동안 반복한다 */
while ((c = getchar()) != EOF) {
        /* 읽어온 문자 c를 처리한다 */
        putchar(c); /* 예: 화면에 그대로 다시 출력하기 */
}

```

이 짧은 문장 하나가 다음 세 가지 작업을 매끄럽게 처리한다.

1. **입력:** `getchar()`가 문자를 하나 읽어온다.
2. **저장:** 읽어온 문자를 변수 `c`에 대입한다.
3. **검사:** 방금 `c`에 대입된 값이 입력의 끝을 알리는 `EOF`인지 확인하고, 아니라면 반복문 안으로 들어간다.

만약 이 패턴을 쓰지 않고 코드를 길게 풀어 쓴다면 다음과 같이 지저분해진다. `getchar()`를 두 번이나 호출해야 하는 것을 볼 수 있다.

```c
int c;

c = getchar();          /* 반복문 진입 전 첫 번째 입력 */
while (c != EOF) {
        putchar(c);
        c = getchar();  /* 다음 문자를 위한 두 번째 입력 (코드 중복!) */
}

```

> [!WARNING]
> **왜 문자를 담는데 `char c`가 아니라 `int c`일까?**
>
> 초보자들이 가장 많이 하는 실수 중 하나가 `char c;`로 선언한 뒤 `c = getchar()`를 받는 것이다.
> `getchar()`라는 이름만 보면 당연히 `char`를 뱉을 것 같지만, 사실 이 함수는 `int`를 반환한다.
> 왜냐하면 **`EOF`** 때문이다. 
> 시스템마다 조금씩 다르지만, 일반적으로 `EOF`는 `-1`로 정의되어 있다. 
> 만약 `char` 변수에 입력을 받으면, 
> 어떤 컴퓨터에서는 `-1`을 제대로 저장하지 못하거나, 정상적인 특수 문자, 
> 예를 들면 값이 255인 문자를 `EOF`로 착각하는 치명적인 버그가 발생할 수 있다.
> 따라서 `getchar()`의 결과를 담는 상자로 반드시 `EOF` **-1**을 
> 안전하게 담을 수 있는 `int` 타입을 쓴다.

**터미널에서는 어떻게 동작할까?**

만약 위 프로그램을 실행한 뒤 터미널에서 키보드로 `a`, `b`, `c`를 치고 엔터 _enter_ 키를 누르면 어떤 일이 벌어질까? `getchar()`가 한 글자씩 읽는다고 했으니 `a`를 누르자마자 화면에 `a`가 출력될 것 같지만, 실제로는 그렇지 않다.

1. **줄 버퍼링 _line buffering_:** 
   터미널 같은 대화형 환경에서 표준 입력은 보통 줄 버퍼 방식으로 동작한다. 
   유저가 `a`, `b`, `c`를 누르는 동안에는 운영체제의 임시 보관소 버퍼에 
   문자가 쌓이기만 할 뿐, 프로그램 내부의 `getchar()`로 곧바로 전달되지 않는다.

2. **엔터 키와 데이터 전송:** 
   유저가 엔터 키를 누르면 입력 버퍼 끝에 줄바꿈 문자 `\n`가 추가된다. 
   비로소 버퍼에 쌓여 있던 데이터 블록 `a`, `b`, `c`, `\n`이 프로그램으로 
   한 번에 전송되어 읽기 가능한 상태가 된다.

3. **루프의 맹렬한 반복 처리:** 이제 프로그램은 전송받은 문자 스트림을 하나씩 처리하기 시작한다.

   - **1~3번째 반복:** 
     `getchar()`가 차례대로 `a`, `b`, `c`를 읽어 변수 `c`에 대입한다. 
     정상적인 문자이므로 `EOF`가 아니라고 판정되어 루프 안으로 진입하고, 
     `putchar(c)`가 화면에 `a`, `b`, `c`를 차례대로 출력한다.

   - **4번째 반복:** 
     엔터 키가 만들어낸 개행 문자 `\n`를 읽는다. 
     개행 문자 역시 `EOF`가 아니므로 루프에 진입하여 `putchar('\n')`을 실행하고,
     터미널 화면의 커서가 다음 줄 처음으로 이동해서 줄바꿈이 된다.

4. **다시 대기 상태로 _blocking_:** 
   버퍼에 들어있던 4개의 문자를 모두 처리하고 나면 프로그램이 종료될까? 아니다. 
   사용자가 친 엔터 키는 단순한 줄바꿈 문자일 뿐, 
   '입력의 끝(`EOF`)' 신호가 아니기 때문이다. 
   프로그램은 루프를 종료하지 않고 다시 `while`문 조건식의 `getchar()` 지점에서
   멈춰서 _blocking_, 새로운 입력이 들어오기를 기다리게 된다.

요약하자면, 화면에는 사용자가 입력한 `abc` 바로 밑에 프로그램이 출력한 `abc`가 똑같이 찍히고, 줄이 바뀐 뒤 커서가 깜빡거리며 대기하는 상태가 된다. 

> [!TIP]
> **`while ((c = getchar()) != EOF)` 루프 탈출하는 방법**
> 
> 터미널에서 대기 중인 프로그램에게 더 이상 읽을 입력이 없다는 
> 진짜 `EOF` 신호를 보내려면, 운영체제가 제공하는 특별한 단축키를 눌러야 한다.
> - **Windows:** `Ctrl + Z`를 누르고 Enter
> - **Mac / Linux:** `Ctrl + D`
> 
> 단축키를 누르면 `getchar()`가 비로소 음의 정수 `EOF`를 반환하게 되고, 
> 조건식이 거짓이 되면서 길었던 반복문이 마침내 종료된다.

**응용: 엔터 `'\n'`도 처리하기**

터미널에서 `getchar()`로 엔터까지 한 줄을 처리하려면, 
`&&` 논리 _and_ 연산을 써서 `EOF` 뿐만 아니라 `'\n'`도 확인해주면 된다. 

```c
/* 엔터 '\n'나 EOF를 만날 때까지 계속 읽는다 */
while ((c = getchar()) != '\n' && c != EOF) {
        /* 읽어온 문자 c를 처리한다 */
        putchar(c); /* 예: 화면에 그대로 다시 출력하기 */
}
```

> [!WARNING]
> **왜 `\n`만 검사하지 않고 `EOF`도 검사할까?**
>
> `while ((c = getchar()) != '\n')` 이라고만 쓰면 안 될까? 절대 안 된다.
> 만약 사용자가 엔터키를 치지 않고 갑자기 `Ctrl + D`로 `EOF` 를 눌러버리거나, 
> 읽어 들이던 파일이 엔터 없이 갑자기 끝나버리면 큰일 난다. 
> `getchar()`는 더 읽을 것이 없으므로 계속해서 `EOF` **-1**를 반환하고, 
> 한번 `EOF` 처리가 된 다음에는 해당 입력 채널이 닫혀서 
> `getchar`는 항상 `EOF`를 반환한다.
> 그래서 `EOF`는 줄바꿈 문자 `\n`가 아니므로 조건식은 영원히 참이 된다.
> 결국 프로그램은 오지 않을 엔터키를 기다리며 빛의 속도로 도는 무한 루프에 빠진다.
> 입력을 받을 때 반드시 "엔터 `\n`를 만나거나, 입력이 끊기면 `EOF` 멈춰라고 
> 이중 안전장치를 걸어두는 습관을 들여야 한다.

### 15.3.5 위험: 한 문장 안에서 같은 변수를 여러 번 흔들지 말자

C언어는 실행 속도의 최적화를 위해 컴파일러에게 자유를 준다.
컴파일러는 계산 순서를 효율적으로 바꾸기 위해, 
세미콜론 `;`을 만나거나 하기 전까지 상수에 저장된 값의 업데이트를 미룰 수도 있고, 당길 수도 있다.

그래서 한 문장 안에서 같은 변수를 건드리면 
"정의되지 않은 동작 _undefined behavior_"이라는 재앙이 일어난다.

**대표적인 "악마의 코드"**

다음 코드는 실행 결과가 10일까, 11일까, 12일까?

```c
int i = 5;
int result = i++ + i++;  /* 절대 금지! */
```

- **컴파일러 A:** `5 + 5 = 10` 옛날 값을 쓰고 나중에 두 번 증가한다.
- **컴파일러 B:** `5 + 6 = 11` 하나 증가시키고 더한다.
- **컴파일러 C:** 프로그램이 강제 종료 _crash_ 된다.

정답은 아무도 모른다. 표준 문법이 결과를 보장하지 않기 때문이다. 
내 컴퓨터에서 잘 돌아간다고 안심하면 안된다. 다른 컴퓨터에서는 폭발할 수 있다.

특히 배열 인덱스를 다룰 때 가장 많이 실수한다.

```c
a[i] = i++; /* (X) i가 먼저 증가할지, 인덱스를 먼저 잡을지 모른다. */
```

**프로그래밍 안전 수칙**

문제를 피하는 방법은 단순하다. 한 줄에 욕심내지 않는 것이다.

- **한 줄에 `++/--`는 한 번만:** 상자 하나당 한 번만 쓴다.
- **섞어 쓰기 금지:** `i`를 바꾸는 `i++` 동시에 `i`를 읽는 `a[i]` 코드를 한 줄에 쓰지 말자.
- **함수 입력값 주의:** `add(i++, i++)`에서 입력값 계산 순서도 랜덤이다. 쓰지 말자.
- **헷갈리면 쪼개기:** 한 줄을 두 줄로 나누는 것이 부끄러운 게 아니다. 명확한 게 최고의 코드다.

```c
/* 나쁜 예: 동작이 정의되지 않는다 */
a[i++] = b[i];

/* 좋은 예: 명확하다 */
a[i] = b[i];
i++;
```

**단락 평가 _short-circuit_과 부수 효과**

`&&` _and_ 와 `||` _or_ 연산자는 효율성을 위해 결과가 확실하면 뒤쪽을 실행하지 않는다.
여기에 부수 효과 `++`를 넣으면, 코드가 실행될 때도 있고 안 될 때도 있는 유령 같은 버그가 생긴다.

```c
/* 위험한 코드 */
if (is_safe && count++ < 10) { ... }
```

- **`is_safe`가 참이면**: `count++`가 실행되어 `count` 가 증가된다.
- **`is_safe`가 거짓이면**: `count++`가 실행되지 않아서 `count`는 그대로)=다.

논리 흐름에 따라 상자에 저장된 값이 변했다 변하지 않았다 하면, 디버깅하기 어려워진다.

조건 수식 안에서는 되도록 변수를 변경 `++`하지 말자.

## 15.4 심화: 제어 문법과 부수 효과의 융합: 더프의 장치

지금까지 우리는 `while`, `do-while`, 그리고 `++` 연산자가 만드는 부수 효과를 배웠다. C언어는 프로그래머를 전적으로 믿기 때문에, 문법의 규칙만 맞으면 상상을 초월하는 제어 흐름의 결합을 허용한다.

1983년, 루카스필름 _lucasfilm_ 에서 일하던 톰 더프 _tom duff_ 는 비디오 데이터를 그래픽 장치로 아주 빠르게 전송해야 하는 최적화 문제에 직면했다. 이때 그가 작성한 코드는 C언어 역사상 가장 천재적인 코드로 불리는 더프의 장치 _duff's device_ 다.

### 15.4.1 루프 전개 _loop unrolling_ 의 딜레마

데이터를 대량으로 전송할 때, 루프를 도는 것에 필요한 조건 검사, 인덱스 증가도 상당한 성능 저하를 가져온다.

```c
/* 평범한 전송 루프 */
int n = count;
while (n > 0) {
        *to = *from++;  /* 부수 효과를 이용해 값을 전송하고 포인터를 이동 */
        n--;
}
```

단순한 코드이지만, 데이터를 1개 전송할 때마다 `n > 0`을 검사해야 한다. 그래서 프로그래머들은 루프를 한 번 돌 때 데이터를 8개씩 통째로 전송하게 만드는 루프 전개 _loop unrolling_ 기법을 사용한다. 이렇게 하면 조건 검사 횟수를 8분의 1로 줄일 수 있다.

```c
/* 8개씩 묶어서 전송 (루프 전개) */
int n = count / 8;
while (n > 0) {
        *to = *from++;
        *to = *from++;
        /* ... (총 8번 전송) ... */
        *to = *from++;
        n--;
}
```

하지만 치명적인 문제가 남는다. 전송해야 할 개수 `count`가 딱 8의 배수로 떨어지지 않고 나머지가 남는 경우다. 자투리를 처리하기 위해 루프 바깥에 별도의 코드를 또 짜야 하니 코드가 지저분해진다.

### 15.4.2 `switch`와 `do`-`while`의 결합

톰 더프는 자투리를 처리하는 코드를 중복해서 작성하기 싫었다. 그래서 `switch`문과 `do-while`문을 한 덩어리로 섞어버리는 코드를 작성한다.

```c
/* 더프의 장치 (Duff's Device) */
int n = (count + 7) / 8;  /* 8개씩 묶었을 때 반복할 총 횟수 (올림 처리) */

switch (count % 8) {      /* 자투리 개수만큼 먼저 점프하여 진입! */
        case 0: do { *to = *from++;
        case 7:      *to = *from++;
        case 6:      *to = *from++;
        case 5:      *to = *from++;
        case 4:      *to = *from++;
        case 3:      *to = *from++;
        case 2:      *to = *from++;
        case 1:      *to = *from++;
                } while (--n > 0); /* do-while의 끝, 다시 case 0으로 올라간다! */
}

```

**왜 컴파일이 될까?**

다른 프로그래밍 언어를 배운 사람이라면 이 코드를 보고 경악할 것이다. `do {` 블록이 열렸는데 그 안에 `case` 라벨들이 들어있고, 심지어 `switch` 블록이 닫히기도 전에 `while`이 튀어나왔기 때문이다. 논리적인 중괄호 `{ }`의 구역이 완전히 파괴된 것처럼 보인다.

하지만 C언어에서 `switch`문의 `case`는 논리적인 블록을 나누는 벽이 아니라, 단순히 점프할 위치를 나타내는 이름표일 뿐이다. 컴파일러는 중괄호의 경계를 무시하고 조건에 맞는 `case` 이름표가 있는 곳으로 실행 흐름을 이동한다.

**어떻게 동작할까?**

만약 `count`가 **9**라고 가정해 보자.

1. `n = (9 + 7) / 8`이므로 `n`은 **2**가 된다. 루프를 총 2번 돈다.
2. `count % 8`은 **1**이다. `switch`문은 `do`의 중괄호를 뚫고 `case 1:`로 곧바로 **점프**한다.
3. `case 1`부터 아래로 코드가 폭포수처럼 흘러내리며 _fall-through_ 남은 자투리인 1개의 데이터를 전송한다.
4. 아래로 내려가다 보니 `} while (--n > 0);`을 만난다. `n`이 1가 되고 조건이 참이므로 루프의 처음(`do`)인 `case 0`으로 훌쩍 올라간다.
5. 이제부터는 온전하게 8개의 데이터를 통째로 전송하는 과정을 실행하고, `n`이 0이 되어 루프를 빠져나온다.

단 하나의 루프와 제어문만으로 자투리 처리와 루프 전개를 완벽하게 해결한 것이다. 이 코드는 C언어의 `case`가 사실상 곧 배우게 될 `goto`문의 목적지 역할을 한다는 것을 보여준다.

> [!WARNING]
> **_duff's device_ 코드가 실무에서 쓰일까?**
>
> 그렇지 않다. 
> 오늘날의 최신 컴파일러 _gcc_, _clang_, _msvc_ 는 우리가 단순한 `while`문을 작성해도, 알아서 내부적으로 '루프 전개'를 적용해 기계어를 최적화해 준다.
