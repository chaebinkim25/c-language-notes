# 7. 반복문 `for`: 컴퓨터를 활용하는 최고의 기술

지금까지 우리는

- C 프로그램이 `main`에서 시작한다는 것
- 데이터를 저장하는 방법 `int`
- 문장이 순서대로 실행된다는 것과 `if`로 갈림길을 만드는 방법

을 배웠다.

다음 순서는 `if`에 이어 프로그래밍의 양대 산맥인 반복을 (*loop*) 정복할 차례다. 

> [!NOTE]
> **보안 관점: 왜 반복문이 무서운가?**
>
> 영화에서 해커가 비밀번호를 뚫을 때 숫자가 빠르게 돌아가는 장면을 본 적이 있는가?
> 사람은 비밀번호를 3번 틀리면 포기하지만, 컴퓨터는 반복문을 이용해 `0000`부터 `9999`까지 전부를 눈 깜짝할 새에 입력해볼 수 있다.
> 모든 경우를 시도해보는 방법을 무차별 대입 공격(*brute force attack*)이라고 한다.

> [!NOTE]
> **보안 관점: 반복되는 공격과 방어**
>
> 현실의 보안 문제는 한 번의 판단으로 끝나지 않는다.
> 공격은 수천 번 반복되고, 방어도 수천 번 검사해야 한다.
> 반복문은 바로 그 자동화된 검사와 공격의 기반이다. 

컴퓨터가 인간보다 월등히 뛰어난 점은 딱 하나, 지치지 않고 실수 없이 똑같은 일을 무한히 반복하는 것이다. 

7장은 이 컴퓨터의 파괴력을 내 손으로 다루는 법을 배운다. 

## 7.0 이 장의 목표

7장의 목표는 `for` 문법을 이용해 다음 세 가지 패턴을 익히는 것이다. 

1. **숫자 세기 (*counting*):** 1, 2, 3 ... 순서대로 셈을 한다 (나중에 분석할 해킹 툴의 기초가 된다).
2. **값 누적하기 (*accumulating*):** 1부터 10까지 더하는 것처럼 값을 차곡차곡 모은다.
3. **조건으로 거르기 (*filtering*):** 반복하는 도중에 원하는 값만 쏙쏙 골라낸다.

> [!NOTE]
> **미리 알림: 증감 연산자(`++`, `--`)는 아직 쓰지 않는다.**
>
> C 언어에는 `i++`처럼 1을 더해주는 편리한 연산자가 있지만,
> 우리는 반복문의 원리에 집중하기 위해 당분간 `i = i + 1`이라는 정직한 표현을 쓴다.

## 7.1 반복이 왜 필요한가

프로그램은 컴퓨터에게 건네는 작업 지시서라고 했다. 
컴퓨터는 스스로 알아서 일을 하지 못하므로, 우리가 무엇을 어떤 순서로 할지를 꼼꼼하게 문장으로 적어 줘야 한다. 

그런데 작업 지시서에는 이런 요구가 정말 자주 나온다.

- "비밀번호를 5회 틀리면 잠가라."
- "접속한 사용자 1,000명의 권한을 검사해라."
- "데이터 10,000개를 암호화해라."

만약 반복문이 없다면, 우리는 같은 문장을 계속 복사해서 붙여넣어야 한다. 
예를 들어 "0번부터 9번까지 사용자를 검사해라"를 반복문 없이 적으면 이렇게 된다. 

```c
check_user(0);
check_user(1);
check_user(2);
// ... (중략) ...
check_user(9);
```

너무 비효율적이다.
숫자가 10개면 10줄, 1000개면 1000줄을 써야 한다. 
게다가 1000줄을 다 썼는데 "사용자가 2000명으로 늘어났어"라고 한다면? 상상만 해도 끔찍하다.

**프로그래밍은 "손으로 하기 싫은 일을 컴퓨터에게 시키는 기술"이다.** 

그런데 이 방식은 컴퓨터에게 일을 시키기 위해 내 손이 먼저 지쳐버리는 꼴이다.
그래서 C 언어는 "같은 일을 여러 번 하라"를 짧게 표현할 수 있는 반복문 (*loop*)을 제공한다. 

> [!NOTE}
> **보안 관점: 복사 붙여넣기는 보안의 적이다**
>
> 보안 전문가들은 중복된 코드(*duplicated code*)를 싫어한다.
> 반복문을 쓰지 않고 코드를 복사해서 붙여넣으면 치명적인 보안 위협이 생기기 때문이다.
>
> 1. 패치 누락 (*inconsistent patching*)
>    만약 앞서 예제에서 본 `check_user` 함수에 보안 버그가 있어서 `check_user_secure` 함수로 코드를 수정해야 한다고 가정해 보자.
>    - **반복문이 없다면:** 함수를 호출한 코드를 일일이 찾아다니며 다 고쳐야 한다.
>      그러다 실수로 딱 한 줄을 빼먹는다면? 해커는 그 구멍을 찾아내 공격한다.
>    - **반복문을 썼다면:** 반복문 안의 코드 딱 한 곳만 고치면 모두 적용된다.
>      
> 2. 코드의 복잡성
>    코드가 길어질수록 개발자의 집중력은 떨어진다.
>    10줄로 끝날 코드가 10,000줄이 되면, 그 속에 숨어 있는 논리적 오류나 오타를 찾아내는 건 불가능에 가깝다.
>    코드가 짧을수록 보안성은 높아진다는 것이 정설이다 (*complexity is the enemy of security*).
>
> 결국 반복문은 개발자의 손목을 보호할 뿐만 아니라, 프로그램의 빈틈을 없애는 방패 역할도 한다.

## 7.2 반복을 만드는 재료 3가지: 시작, 끝, 그리고 갱신

반복은 겉보기에는 복잡해 보이지만, 사실은 딱 세 가지 재료로 만들어진다. 
세 가지 중 하나라도 빠지거나 틀리면, 프로그램은 멈추지 않고 폭주하거나 엉뚱한 곳을 건드리는 사고를 친다.

1. **초기화 (*initialization*):** "어디서부터 시작할 것인가?" (출발점)
2. **반복 조건 (*termination condition*):** "언제 멈출 것인가?" (브레이크)
3. **갱신 (*update*)**: "한 번 돌 때마다 무엇이 어떻게 바뀔 것인가?" (엑셀)

우리는 5장에서 "갱신"의 핵심을 미리 연습했다.

```c
i = i + 1;
```

갱신은 반복의 뼈대다. 값이 조금씩 바뀌어야 반복이 앞으로 진행할 수 있고, 언젠가는 반복 조건을 끝낼 수 있다. 

앞으로 볼 `for`는 이 세가지를 한 줄에 모아 놓은 안전하고 강력한 반복 문법이다.

> [!NOTE]
> **보안 관점: 반복 조건은 안전 브레이크다**
>
> 보안 전문가들은 이 세가지 재료 중 반복 조건을 가장 유심히 본다. 시스템을 보호하는 최후의 안전장치 (*fail-safe*)이기 때문이다.
>
> 1. 브레이크 고장 (무한루프로 인한 가용성 침해)
>    만약 반복 조건이 영원히 `참 (true)`이 되도록 실수한다면?
>    프로그램은 멈추지 않고 CPU를 100% 사용하여 컴퓨터를 뜨겁게 달군다.
>    결과적으로 해커가 의도적으로 시스템을 마비시키는 서비스 거부 (*DoS, Denial of Service*) 공격과 똑같은 효과를 낸다.
>
> 2. 브레이크가 늦게 밟힘 (*OOB*로 인한 무결성 침해)
>    10번만 돌아야 하는데 조건식을 잘못 써서 11번 돌게 된다면?
>    마지막 한번의 실행이 허가되지 않은 메모리를 건드려 정보를 유출하거나 데이터를 오염시킬 수 있다 (*OOB, Out-of-Bounds*). 

## 7.3 `for`문의 기본 모양: 3단 합체 로봇
   
`for`는 C 언어가 미리 정해 둔 예약어 (*keyword*)다.

`for`문의 모양은 다음과 같이 세 가지 부품이 합체된 형태다.

```c
for (초기화; 반복_조건; 갱신) {
        반복해서 실행할 문장들
}
```
여기서 헷갈리기 쉬운 포인트가 하나 있다.

**`for ( )` 안의 세미콜론 `;`은 문장 끝이 아니다**

우리는 지금까지 `;`을 문장이 끝났다는 마침표로 봤다.

하지만 `for` 괄호 안의 `;` 두 개는 칸막이 (*separator*) 역할이다.

- **첫 번째 칸:** 시작할 때 딱 한 번 실행할 코드
- **두 번째 칸:** 매번 검사할 보안 조건
- **세 번째 칸:** 한 바퀴 돌 때마다 실행할 갱신 코드

> [!WARNING]
> **보안 관점: *off-by-one error***
>
> `for`문의 세 칸 중 보안에서 가장 중요한 곳은 가운데 조건 칸이다.
> 초보자는 종종 `<=`와 `<`를 헷갈리는 실수를 한다.
>
> - **안전:** `for (i = 0; i < 10; i = i + 1)` 코드는 `i`가 `0`부터 `9`까지 10번 반복된다.
> - **위험:** `for (i = 0; i <= 10; i = i + 1)` 코드는 `i`가 `0`부터 `10`까지 11번 반복된다.
>
> "겨우 1번 더 도는 게 무슨 문제냐?"라고 할 수 있다.
> 하지만 이 '1번'의 차이가 할당되지 않은 메모리를 건드리는 치명적인 버그로 이어진다.
> 보안 용어로 오프-바이-원(*off-by-one error*)이라고 하며, 수많은 해킹 사고가 이 작은 부등호 실수 하나에서 시작되었다.
> 조건 칸을 작성할 때는 "항상 경계선이 어디인가?"를 긴장하며 확인해야 한다.

## 7.4 `for`문은 어떤 순서로 실행될까?

`for`문은 구조적으로 딱 한 번만 실행되는 부분과 매번 반복되는 부분이 섞여있다. 
반복문 정복의 첫걸음은 이 흐름을 이해하는 것이다. 

**실행 순서의 핵심 (*flow*)**

`for`문의 실행 흐름은 다음과 같은 루프(*loop*) 를 그린다.

```c
for (초기화; 반복_조건; 갱신) {
        반복해서 실행할 문장들
}
```

1. **반복 시작값 초기화:** 맨 처음에 딱 한 번만 실행된다.
2. **반복 조건 검사:**
   - **참 (*true*)이면:** 블록 `{ }` 안의 반복해서 실행할 문장들로 이동한다.
   - **거짓 (*false*)이면:** 반복을 즉시 종료하고 `for`문 밖으로 나간다.
3. **본문 실행:** `{ }` 안의 코드를 실행한다.
4. **갱신:** 반복 상태를 모니터링하는 변수의 값을 변화시킨다.
5. 다시 2번(조건 검사)로 돌아간다.

요약하자면: `초기화`, (`조건(참)`, `본문`, `갱신`), (`조건(참)`, `본문`, `갱신`), ... `조건(거짓)`, `반복끝` 순서로 진행된다.

**예제로 실행 흐름 따라가기 (*trace*)**

변수 `i`와 `last`를 쓰는 작은 `for`문 예제 코드에서 변수의 값이 어떻게 변하는지 추적해보자.

```c
int main()
{
        int i = -1;
        int last = -1;

        /* i가 0부터 2까지, 총 3번 반복 */
        for (i = 0; i < 3; i = i + 1) {
                last = i;    /* 현재 반복 중인 i 값을 기록 */
        }

        /* 반복 종료 후 i와 last의 값은 얼마일까? */
        return 0;
}
```

`for`문에서 반복을 시작하기 전, `i`와 `last`에 저장된 값은 모두 **-1**이었다. 

`for`문이 시작될 때, `i`에는 **0**이 저장된다. 

`for`문 반복 1회차에서, 반복 조건 `i < 3`이 참이므로 본문과 갱신이 실행된다. 
본문에서 `last`에 `i`의 값인 **0**이 저장된다. 
갱신에서 `i`에 `i + 1`인 **1**이 저장된다.

`for`문 반복 2회차에서, 반복 조건 `i < 3`이 참이므로 본문과 갱신이 실행된다. 
본문에서 `last`에 `i`의 값인 **1**이 저장된다. 
갱신에서 `i`에 `i + 1`인 **2**가 저장된다.

`for`문 반복 3회차에서, 반복 조건 `i < 3`이 참이므로 본문과 갱신이 실행된다. 
본문에서 `last`에 `i`의 값인 **2**가 저장된다. 
갱신에서 `i`에 `i + 1`인 **3**이 저장된다.

다음 번에, 반복 조건 `i < 3`이 거짓이 되어 반복문이 종료된다.

|단계|현재 `i`|조건식 (`i < 3`)|실행 내용|`last`의 값|`i`의 변화(갱신)|
|:---|:---|:---|:---|:---|:---|
|1회차|0|참|본문 실행|-1에서 **0**으로|0에서 **1**로|
|2회차|1|참|본문 실행|0에서 **1**로|1에서 **2**로|
|3회차|2|참|본문 실행|1에서 **2**로|2에서 **3**으로|
|종료|3|거짓|루프 탈출|**2**(유지)|**3**(유지)|

**여기서 꼭 잡아야 할 감각 (보안 핵심)**

반복문이 끝난 뒤의 상황을 정확히 이해해야 한다.

- **본문이 실행되는 동안:** `i`는 **0, 1, 2**였다.
- **반복이 완전히 끝난 뒤:** `i`는 **3**이다. 

반복문 종료 후의 `i`는 마지막으로 실행된 값 **2**가 아니라, 조건을 거짓으로 만들어 반복을 끝내버린 값 **3** 이 되는 것이다.

> [!WARNING]
> **보안 관점: 반복이 끝난 뒤 *off-by-one error***
>
> 보안에서 이 개념이 왜 중요할까?
> 많은 초보자가 반복문이 끝난 후 `i`를 재사용하다가 논리적 버그를 만든다.
>
> **시나리오: 비밀번호 3회 제한 시스템**
>
> ```c
> int i;
> int max_try = 3;       /* 최대 3번까지 시도 허용 */
>
> /* 0, 1, 2회차 (총 3번) 시도한다 */
> for (i = 0; i < max_try; i = i + 1) {
>         try_login();
> }
>
> /* 개발자의 착각: 반복문이 끝났으니 i는 2 (마지막 횟수)겠지? */
>
> /* 논리적 버그: 실제 i는 3이기 때문에 해커에서 보너스 기회를 한번 더준 샘이 된다. */
> if (i == max_try) {
>         try_login();       /* 4번째 시도가 실행됨 (보안 정책 위반)
> }
> ```
>
> 루프가 끝난 변수 `i`는 이미 제한선인 **3**에 도달한 상태다.
> 개발자가 `i`의 값을 **2**라고 착각하고 코드를 짰기 때문에, 3회 제한이라는 보안 정책을 스스로 무너뜨리게 되었다.

## 7.5 C99 이후: 범위 축소로 더 안전해진 변수 선언

**C89 방식**

C89 표준에서는 변수 선언을 반드시 블록의 맨 앞에 모아야 했다.

```c
int main(void)
{
        int i;  /* 선언 구역 */

        for (i = 0; i < 10; i = i + 1) {
                /* ... */
        }

        /* 루프 종료 후에도 i는 여기서 유효하다 */

        return 0;
}
```

- 선언 구역과 실행 구역이 분리된다.
- 반복문이 끝난 뒤에도 루프 기록 변수 `i`는 유효하다.
- `i`를 나중 코드에서 실수로 재사용할 가능성이 있다.

**C99 방식**

C99 표준 이후부터는 `for`문에서만 쓸 변수는 `for`문의 초기화 부분에서 선언하는 것이 허용된다.

```c
for (int i = 0; i < 10; i = i + 1) {
        /* ... */
}

/* 여기서는 i를 사용할 수 없다 (컴파일 에러) */
```

- 루프 기록 변수 `i`의 유효 범위가 `for`문 내부로 제한된다.
- 루프 밖에서 `i`에 접근할 수 없다.

변수의 유효 범위가 줄어들면

- 우발적 재사용 방지
- 상태 오염 감소
- 유지보수성 향상
- 코드 의도 명확화

효과가 발생한다. 
결과적으로 더 안전한 코드 작성을 유도한다.

> [!NOTE]
> **보안 관점: 변수가 유효한 범위(*scope*)를 최소화하라**
>
> 세계적인 보안 가이드라인 CERT C는 DCL19-C 규칙을 통해 "변수의 유효 범위를 가능한 한 작게 줄여라"고 권고한다.
> 범위를 최소화하면
> - 실수 가능성이 감소하고
> - 코드의 의도가 명확해지며
> - 유지보수가 쉬워진다.



## 7.6 C89 규칙: `<`와 `<=`: 반복에서 가장 흔한 실수

반복문을 작성할 때 가장 많이 하는 실수는 **딱 한 번 더 돌거나, 딱 한 번 덜 도는** 실수다. 
프로그래머들은 이를 **Off-by-one Error**라고 부른다.
특히 `0`부터 시작할 때 `<`를 쓸지 `<=`를 쓸지 헷갈리기 쉬운데, 이 작은 차이가 결과에 큰 영향을 미친다.

1. `<` 를 쓰는 경우 (기준: 횟수 중심)
보통 **n번 반복하고 싶다**고 할 때 가장 많이 쓰는 표준적인 방식이다.

```c
/* 0부터 9까지 (총 10번 실행) */
for (i = 0; i < 10; i = i + 1) {
    /* ... */
}
```
- 실행되는 `i` 값: 0, 1, 2, ..., 9
- 반복 횟수: 총 10번
- 종료 시점: `i`가 10이 되는 순간 `10 < 10`은 거짓이 되어 멈춘다.
- 특징: 0부터 시작해서 숫자를 반복할때 권장되는 형태다.

2. <= 를 쓰는 경우 (기준: 범위 중심)
끝나는 숫자를 포함하고 싶을 때 사용한다.
```c
/* 0부터 10까지 (총 11번 실행) */
for (i = 0; i <= 10; i = i + 1) {
    /* ... */
}
```
- 실행되는 `i` 값: 0, 1, 2, ..., 9, 10
- 반복 횟수: 총 11번
- 종료 시점: `i`가 11이 되는 순간 `11 <= 10`은 거짓이 되어 멈춘다.
- 특징: 1부터 시작해서 숫자를 반복할때 권장되는 형태다.

3. 결론: 무엇을 써야 할까?
C언어는 **0부터 숫자를 세는 문화(Zero-based numbering)** 를 가지고 있다.
따라서 특별한 이유가 없다면 **작다 `<`**를 사용하는 것이 실수를 줄이는 가장 좋은 습관이다.

   **공식처럼 외우기**: "0부터 시작해서 10번 돌고 싶다면? i < 10" (등호 =를 뺀다!)



## 7.7 예제 1: "0부터 9까지 돈다"를 변수 값으로 확인하기

처음 반복문을 쓸 때는 "정말 10번 돌았나?"가 불안할 수 있다.
그래서 여기서는 반복 횟수를 `count`에 저장해서 확인해본다.

```c
int main()
{
   int i;           /* 루프 변수. 0부터 반복할 계획 */
   int count = 0;   /* 반복 횟수. 처음에는 0번 실행됨 */
   int last = -1;   /* 마지막에 실행된 i. -1은 루프가 돌지 않았음을 표시 */

   /*  i를 0부터 9까지(10보다 작을 때까지) 반복 */
   for (i = 0; i < 10; i = i + 1) {
      count = count + 1;   /* 반복 횟수 업데이트 */
      last = i;            /* 현재 처리 중인 i값 저장 */
   }

   /* 반복이 끝났을 때:
      - count는 10
      - last는 9
      = i는 10
   */

   return 0;
}
```



## 7.8 예제 2: 1부터 10까지 합계 구하기

반복문은 누적해서 계산하는 데에 많이 쓰인다.

```c
int main()
{
   int i;         /* 루프 변수. 1부터 반복할 계획 */
   int sum = 0;   /* 합계. 초기값은 +의 항등원인 0 */

   /* i를 1부터 10까지(10보다 작거나 같을 때까지) 반복 */
   for (i = 1; i <= 10; i = i + 1) {
      sum = sum + i;   /* 기존 sum에 현재 i를 더한 값을 sum에 저장한다. */
   }

   /* 반복이 끝나면 sum은 55가 된다 (1+2+...+10) */
   return 0;
}
```

중요한 습관: 누적 변수는 반드시 항등원으로 초기화

`sum = 0;`을 빼먹으면 결과가 망가질 수 있다. 
`sum`이 알 수 없는 값인 상태에서는 `sum + i`도 알 수 없는 값이기 때문이다. 

누적은 계산할 때 이전 값이 쓰이기 때문에, 처음 시작한 값이 틀리면 끝의 값도 틀린다. 


## 7.9 예제 3: 1부터 10까지 짝수들의 합계 구하기

프로그래밍의 가장 강력한 패턴 중 하나는 스캔(Scan) 후 필터링(Filter)하는 것이다. 

6장에서 한 `if`를 반복문 안에서 쓰면, 원하는 값에 대해서만 작업을 할 수 있다. 

1부터 10까지 중에서 짝수만 골라서 개수(`even_count`)와, 합계(`even_sum`)을 구해본다.

```c
int main()
{
   int i;                /* 루프 변수. */
   int even_count = 0;   /* 개수. 초기값은 0 */
   int even_sum = 0;     /* 합계. 초기값은 +의 항등원인 0 */

   /* i를 1부터 10까지(10보다 작거나 같을 때까지) 반복 */
   for (i = 1; i <= 10; i = i + 1) {
      /* i가 2의 배수일 때 실행*/
      if ((i % 2) == 0) {
         even_sum = even_sum + i;   /* 기존 even_sum에 현재 i를 더한 값을 even_sum에 저장한다. */
      }
   }

   /* 반복이 끝나면:
      - even_count는 5 (0,2,4,6,8)
      - even_sum은 30 (2+4+6+8+10)
   */
   return 0;
}   
```

예제 코드에서,

- `for`는 "대상을 하나씩 꺼내는 역할"
- `if`는 "그중에서 필요한 것만 고르는 역할"

을 하고, 비슷한 패턴으로 대부분의 검사 작업이 가능해진다.


## 7.10 `for`에서 자주 하는 실수 2가지

### 7.10.1 무한 루프: "어? 프로그램이 멈췄네!"

반복 변수의 갱신 (`i = i + 1`)을 빼먹으면, 화면에 아무것도 안뜨고 커서만 깜빡이거나, 프로그램이 먹통이 된다(응답 없음).

```c
for (i = 0; i < 10; ) {
   /* i가 바뀌지 않으면 i < 10이 계속 참이 되어 영원히 반복한다. */
}
```

반복문은 앞으로 진행해야 끝난다. 그 진행을 만드는 것이 보통 `i = i + 1` 같은 갱신이다.

> Tip: 반복문이 끝나지 않고 멈춰버렸다면? 당황하지 말고 윈도우/리눅스에서는 키보드에서 `Ctrl` 키와 `C` 키를 같이 누르고, 맥에서는 `Control(^)` 키와 `C` 키를 같이 누르면 터미널에서 실행 중인 프로그램이 강제로 종료된다.

### 7.10.2 부등호 실수로 "하나 더/덜" 돌기

보통 0부터 시작하는 반복문은 10번 돌리려면 `< 10`을 쓴다.
```c
/* 0부터 9까지 반복 (총 10번) */
for (i = 0; i < 10; i++) ...

/* 0부터 10까지 반복 (총 11번) */
for (i = 0; i <= 10; i++) ...
```

헷갈릴 때는 마지막 값으로 부등호 결과를 검산해본다. 

## 7.11 `for`의 권장 코딩 스타일

`for`문의 장점은 반복문이 어떻게 돌지 첫 줄만 보면 알 수 있다는 점이다. 그런데 본문 안에서 `i`의 값을 바꾸면 이 장점이 사라진다.

```c
/* for의 괄호 안만 보면 i를 0부터 9까지 반복하는 것 같지만... */
for (i = 0; i < 10; i = i + 1) {

   if (i == 3) {
      i = 8;    /* i가 점프한다. 중간에 있는 값들은 어떤 처리도 하지 않는다.
   }

   i = i - 1;   /* i를 거꾸로 이동시킨다. 다음에 실행될 i = i + 1을 상쇄하여 반복이 진행될 수 없다.
}
```

`for`문의 괄호 안에 써져 있는 내용은 되도록이면 지켜주는 것이 코드를 읽을 다른 프로그래머들이나 훗날 코드를 다시 볼 본인을 위해서 좋다. 

하지만 반복 방식을 바꾸는 것이 필요할 때도 있으며, 이를 위해 `break`와 `continue`라는 정식 도구가 제공된다.

## 7.12 반복의 제어

반복을 중간에 멈추거나, 특정 순서만 건너뛰려고 할 때를 위해 다음 제어문이 제공된다.

### 7.12.1 찾았으면 그만! 탈출구 `break`

`break`를 만나면 반복문은 조건을 따지지도 않고 즉시 종료된다. 주로 검색을 할 때 쓰인다. 찾으려던 걸 찾았으니 나머지는 볼 필요 없다는 뜻이다.

[상황] 1부터 10까지 돌면서 7을 찾으면 멈추려고 한다.
```c
/* 나쁜 예: 루프 변수를 바꿔서 억지로 종료 */
for (i = 1; i <= 10; i = i + 1) {
   if (i == 7) {
      i = 100; /* 루프를 끝내려고 i를 억지로 키움 (비추천) */
   }
}
/* i에는 for가 종료될 때의 값인 100이 저장되어 있다. */

/* 좋은 예: break 사용 */
for (i = 1; i <= 10; i = i + 1) {
   if (i == 7) {
      break;  /* 즉시 for문이 종료됨 */
   }
}
/* i에는 for가 종료될 때의 값인 7이 저장되어 있다. */
```

> `for`문 안에 `for`문이 있는 경우 break을 했을 경우에는 가장 안쪽의 `for`문이 종료된다.

### 7.12.2 이번만 패스! 건너뛰기 `continue`

`continue`를 만나면 본문의 남은 부분을 무시하고 바로 갱신(`i = i + 1`과 같은) 단계로 점프한다. 주로 필터링을 할 때 쓰인다. 
현재 루프 변수는 내가 처리할 것이 아니니 다음으로 넘어가자는 뜻이다.

[상황] 1부터 10까지 중에서 3의 배수는 빼고 처리하려고 한다.
```c
/* continue를 쓰지 않았을 때 */
for (i = 1; i <= 10; i = i + 1) {
   if (i % 3 == 0) {
      /* 아무 것도 하지 않는다 */
   } else {
      /* 처리를 한다 */
   }
}

/* continue를 썼을 때 */
for (i = 1; i <= 10; i = i + 1) {
   if (i % 3 == 0) {
      continue; /* 남은 내용은 전부 실행하지 않고, 바로 i = i + 1로 이동한다 */
   }
   /* continue되지 않았을 때만 처리가 된다 */
}
```

`continue`는 이번 회차의 나머지만 생략하고 다음 회차로 넘어간다.
