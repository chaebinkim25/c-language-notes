# 9. 메모리와 주소: 값이 어디에 있는가

지금까지 우리는...
9장까지 숨 가쁘게 달려오며 다음 문법들을 익혔다.

- `int`로 값을 만들고 계산하기
- `if`로 갈림길을 만들기
- `for`로 반복 작업 수행하기
- 배열로 여러 값을 한꺼번에 다루기

**초보자가 부딪히는 2가지 의문**

여기까지 오면, 초보자라면 거의 반드시 이런 의문을 갖게 된다.

1. "다른 변수는 `=`로 대입이 되는데, 배열은 왜 통째로 복사가 안되지?"
2. "도대체 내가 만든 변수들은 컴퓨터 어디에 저장되는 거지?"

이 질문들의 핵심은 하나로 귀결된다. 우리는 값 _value_ 만 봤지, 그 값이 어디에 저장되는지 위치를 아직 제대로 보지 못했다.

## 9.0 이번 장의 목표

이번 장의 목표는 이미 배운 문법들이 왜 그렇게 동작하는지를 메모리라는 관점에서 시원하게 파헤치는 것이다.

- 컴퓨터가 값을 저장하는 공간인 메모리 _memory_ 를 이해한다
- 메모리의 각 칸을 구분하는 번호인 _address_ 의 개념을 잡는다
- 변수는 단순한 값이 아니라 값이 들어있는 자리 + 그 자리의 이름표임을 이해한다
- 배열 이름이 왜 배열 전체가 아니라 첫 칸의 위치 정보처럼 행동하는지 밝혀낸다
- C 언어의 가장 큰 산인 포인터를 배우기 위한 준비 운동을 마친다

## 9.1 메모리: 컴퓨터는 기억을 어디에 해둘까?

우리는 일상에서 뭔가를 기억할 때 다음과 같은 방법을 쓴다.

- 머릿속에 저장하거나
- 메모장에 적어두거나
- 파일에 저장한다

컴퓨터도 비슷하다. 그리고 프로그램이 실행되는 동안 컴퓨터가 가장 자주 쓰는 메모장이 따로 있다. 그게 바로 메모리 _memory_ 다. 

### 메모리는 칸이 엄청나게 많은 긴 서랍장이다

메모리를 가장 쉽게 상상하는 방법은 다음과 같다.

- 아주 길게 이어진 칸들
- 각 칸에는 **0** ~ **255** 같은 작은 숫자 하나가 들어갈 수 있다
- 프로그램은 이 칸들에 값을 써두고, 필요할 때 다시 읽는다

여기서 놓치지 말아야 할 중요한 포인트가 있다. 컴퓨터의 데이터는 공중에 둥둥 떠 있지 않다. 항상 메모리 어딘가의 칸들에 들어 있다.

### 가장 작은 칸을 뭐라고 부를까?

메모리 칸의 기본 단위를 보통 바이트 _byte_ 라고 부른다.

처음에는 바이트라는 용어가 낯설 수 있으니, 지금은 그냥 딱 두가지만 기억해도 충분하다.

1. 바이트는 메모리의 한 칸
2. 데이터는 이 칸을 여러 개 묶어서 저장하기도 한다.

예를 들어 보면,
- `int`는 컴퓨터마다 다르지만 여러 칸을 묶어서 쓴다

> [!NOTE]
> **`int`의 크기는 고정이 아니다.**  
> 어떤 컴퓨터에서는 2바이트를 쓰고, 어느 컴퓨터에서는 4바이트를 쓸 수도 있다. 
> 우리는 정확히 몇 바이트인가를 외우는 대신, 타입에 따라 필요한 칸의 개수가 다를 수 있다는 사실만 확실히 잡고 가면 된다.

## 9.2 주소: 메모리 칸에 붙은 ‘번지 번호’

앞서 메모리를 칸이 엄청나게 많은 서랍장이라고 했다. 그런데 서랍이 수천만 개나 된다면, 당연히 이런 문제가 생긴다.

"내가 방금 적어둔 숫자 10이 몇 번째 칸에 있더라?”

그래서 컴퓨터는 메모리의 각 칸마다 고유한 위치 번호를 붙여 두었다. 그 번호를 주소라고 부른다.

**주소는 데이터의 좌표다**

주소는 일상 생활의 위치찾기와 똑같다.

- **집 주소:** 수원시 팔달구… - 집이 어디 있는지
- **사물함 번호:** 108번 사물함 - 내 짐이 어디 있는지
- **좌석 번호:** E열 7번 - 내가 앉을 자리가 어디인지

메모리도 마찬가지다.

- **메모리 주소:** 이것이 메모리 맨 앞에서부터 몇 번째 칸인가?

우리는 지금까지 `a = 10`이라고 하면 단순히 `10`인 값만 생각했다. 하지만 컴퓨터 내부에서는 항상 두가지 정보가 함께 따라다닌다.

1. **값:** 그 칸 안에 무엇이 들어있는가? `10`
2. **주소:** 그 값이 어디에 있는가? `1000번지`

예를 들어 `int` 값이 **4**칸을 쓰는 컴퓨터에서, **0** ~ **255**까지 숫자는 제일 앞칸에 저장된다고 하면,
```
 1000   1001   1002   1003
[ 10]  [  0]  [  0]  [  0]   ← 이 4칸 묶음이 int 타입 변수인 a의 자리
```

주소는 한 칸의 번호이고, 타입은 몇 칸을 묶어서 값 하나를 저장할지를 결정한다.

## 9.3 변수의 정체: 값이 아니라 자리 + 이름표다

우리는 5장에서 변수를 이름 붙인 상자라고 배웠다.  
그 상자를 좀 더 컴퓨터 내부의 시선에서 들여다보자.

**선언 `int x;`는 무슨 뜻인가?**

이 코드는 단순히 x라는 변수를 뿅 하고 만든다고 하는 추상적인 마법의 주문이 아니다. 컴퓨터 입장에서 `int x;`는 아주 구체적인 작업 지시서다. 

1. **공간 확보**: 메모리 어딘가에 `int` 하나를 저장할 만큼의 자리를 찾아서 확보한다.
2. **이름표 부착**: 확보한 그 자리에 `x`라는 이름표를 붙인다.

그래서 변수의 정체는 어딘가의 자리 + 그 자리를 부르는 이름이다. 

**대입 `x = 10;`은 무슨 뜻인가?**

이 문장은 컴퓨터에게 다음과 같이 명령하는 것이다.

- `x`라는 이름표가 붙은 자리를 찾아서,
- 그 자리에 값 `10`을 저장하라.

### `y = x;`는 무슨 뜻인가?

초보자들이 자주 착각하는 포인트가 바로 여기다. `y = x;`는 `x` 상자 자체를 `y`로 옮겨라거나 두 상자를 합쳐라는 뜻이 절대 아니다.

철저하게 내용물만 복사하는 과정이다.

1. `x` 자리에 가서 그 안에 들어있는 값을 읽는다.
2. 읽어온 값을 `y` 자리에 그대로 복사한다.

변수끼리의 대입 `=`은 항상 값 복사다. `x`와 `y`는 여전히 별개의 자리이며, 서로의 위치에는 전혀 영향을 주지 않는다.

## 9.4 알 수 없는 값: 초기화하지 않은 변수의 위험성

우리는 5장에서 이미 이런 경고를 들었다.

> *초기화 없이 `int x;`만 선언하면,*
> * `x` 안에는 알 수 없는 값이 들어있을 수 있다.*

메모리의 원리를 알았으니 그 이유가 명확해진다.

**메모리는 재활용되는 공책이다**

컴퓨터 입장에서 `int x;`라는 명령은 앞에서 봤듯이, 

> *메모리에서 `int` 크기만큼 빈칸을 잡아서 `x`라는 이름을 붙여줘. 청소는 안해도 돼.*

새로운 자리를 확보하는 것일 뿐, 그 자리를 `0`으로 싹 지우라는 말이 아니다.

우리가 변수를 선언해서 받은 그 메모리 칸은, 방금 전까지 다른 프로그램이나 함수가 쓰다가 반납한 메모리 칸일 확률이 매우 높다.
그 칸에는 예전에 쓰던 데이터가 그대로 남아있다.

남겨진 흔적을 우리는 알 수 없는 값이라고 부른다.

**초기화를 안 하면 생기는 일**

초기화 `=`를 하지 않고 변수를 읽는다는건 이런 상황과 같다. 

1. **변수 선언:** 공책의 비어있는 페이지를 펼쳤다.
2. **알 수 없는 값 참조:** 이전 사람이 지우지 않고 남겨둔 낙서를 읽었다.
3. 내가 적은 적이 없는 값인데 내가 적은 값으로 착각하고 계산에 썼다.

프로그램은 운 좋게 문제 없이 동작하거나, 엉뚱한 결과를 내거나, 오류가 발생하여 갑자기 종료될 수 있다.

> [!NOTE]
> **보안 관점: 정보 노출 _information disclosure_ 취약점**
>
> 만약 이전 프로그램이 지우지 않고 남겨둔 낙서가 로그인 비밀번호, 유저의 개인정보, 또는 암호화 키라면 어떨까?
> 해커들은 메모리가 0으로 청소되지 않는다는 점을 악용한다.
> 초기화되지 않은 메모리를 고의로 읽어들여 그 안에 남아있는 민감한 정보를 몰래 빼낸다.
> 보안 용어로 정보 노출 _information disclosure_ 취약점이라고 부른다.

> [!CAUTION]
> **프로그래밍 규칙: 변수는 만들자마자 초기값을 저장하라**
> 
> C언어는 프로그래머를 믿기 때문에 자동으로 청소를 해주지 않는다. 청소는 온전히 프로그래머의 몫이다.
> 특히 C89 규칙에 따라 블록 앞에서 선언을 모두 하면,
> 나중에 변수를 쓸 때가 초기화를 하려다가 잊어버리고 청소를 안하는 일이 더 자주 발생한다.
>
> - **방법 1 - 선언과 동시에:** `int x = 0;` 가장 추천
> - **방법 2 - 쓰기 직전에:** `x = 10;` 사용하기 전에는 무조건 대입
> 
> 관련 코딩 가이드라인: **CERT EXP33-C**, **MISRA C:2025 Rule 9.1** 초기화되지 않은 메모리를 읽지 마라, **MISRA C:2025 Rule 9.3** 배열을 일부만 초기화하지 마라
>
> 관련 보안 사례: **CWE-456** 변수를 초기화 하지 않음, **CWE-457** 초기화되지 않은 변수 사용

## 9.5 블록과 함수: 작업대가 생겼다가 사라진다

우리는 2장에서 중괄호 `{ }`의 의미를 배웠다.

- `{ }`는 여기부터 여기까지가 한 묶음이다는 의미다.
- 블록 안에서 만든 이름은 블록 밖에서 쓸 수 없다.

여기에 9장에서 배운 메모리 관점을 더하면 정의가 이렇게 바뀐다. 
블록은 메모리 위에 잠깐 쓰는 작업대를 펼치는 것이다.

**변수에는 저장 기간이 있다**

우리가 만든 변수는 영원히 존재하지 않는다. 변수는 자기가 속한 블록이 유지되는 동안에만 쓰인다. 
컴퓨터 용어로 변수의 저장기간 _lifetime_ 이라고 부른다. 

1. **블록 시작 `{`:** 새 작업대를 펼친다. 여기서 만든 변수는 이 작업대 위에 올라간다.
2. **블록 끝 `}`:** 작업대를 통째로 치워버린다. 그 위에 있던 변수들도 모두 메모리에서 사라진다.

```c
int main(void)
{
        /* 함수가 시작되며 새 작업대가 펼쳐진다 */
        int x;      /* 메모리에서 int만큼을 새로 확보하고, 이름표 x를 붙여놓는다. */
        x = 1;
        
        {
                /* 블록이 시작되며 새 작업대가 펼쳐진다 */
                int y;   /* 메모리에서 int만큼을 새로 확보하고, 이름표 y를 붙여놓는다. */
                y = 2;
        }
        /* 블록이 끝나며 작업대를 치운다. 이름표 y가 붙어있던 메모리 공간도 재활용 할 수 있게 반납한다. */
                
        {
                /* 블록이 시작되며 새 작업대가 펼쳐진다 */
                int z;   /* 메모리에서 int만큼을 새로 확보하고, 이름표 z를 붙여놓는다. */
                         /* 컴퓨터에 따라서 이전 블록의 y가 붙어있던 메모리 공간이 할당될 수도 있다. */
                z = 3;
        }
        /* 블록이 끝나며 작업대를 치운다. 이름표 z가 붙어있던 메모리 공간도 재활용 할 수 있게 반납한다. */

        return 0;
}
/* 함수 실행이 끝나며 작업대를 치운다. 이름표 x가 붙어 있던 메모리 공간도 재활용 할 수 있게 반납한다. */
```

블록 안에서 변수를 선언해서 확보한 메모리 공간은, 블록이 닫히는 순간 다른 코드에서 쓸 수 있게 반납된다. 

> [!NOTE]
> **보안 관점: 메모리 반납은 데이터 삭제가 아니다**
>
> 여기서 초보자들 뿐만 아니라 때로는 숙련된 개발자들도 놓치는 치명적인 함정이 있다.
> 블록이 끝나서 변수가 사라진다는 것은, 컴퓨터가 이 공간을 다른 사람이 써도 좋다고 팻말만 바꿔 다는 것일 뿐이다.
>
> C 언어는 친절하게 지우개를 들고 와서 그 공간에 쓰여 있던 값을 지워주지 않는다.
> 이름표만 떼어질 뿐 데이터는 메모리 어딘가에 고스란히 남아 정보 노출 취약점이 발생할 수 있다.
>
> 따라서 비밀번호, 개인정보 같은 중요 보안 데이터를 다룬 변수는,
> 블록이 끝나기 전에 프로그래머가 직접 `0`으로 덮어써서 흔적을 지워야 한다.
> 아무도 내 작업대를 대신 치워주지 않는다는 것,
> 이것이 C언어가 프로그래머에게 요구하는 무거운 책임감이자 안전한 소프트웨어를 만드는 습관의 첫걸음이다. 

## 9.6 배열: 연속된 자리와 첫 칸의 위치 정보

8장에서 배열을 "같은 타입 상자들이 줄지어 있는 것"이라고 배웠다. 여기서는 배열을 메모리 관점으로 다시 본다. 

1. `int a[5];`는 메모리에 무엇을 만드는가?

이 배열 선언은 컴퓨터에게 다음과 같이 명령한다: "`int` `5`개를 담을 자리를 연속으로 확보하고, 제일 앞에 `a`라는 이름표를 붙여라." 

여기서 핵심은 연속성 _contiguous_ 이다. 중간에 빈틈이 전혀 없이 붙어있어야 한다. 

만약 `int`가 **4**바이트고, 시작 주소가 `1000`번지라면, 

| 배열 요소 | `a[0]` | `a[1]` | `a[2]` | `a[3]` | `a[4]` |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **저장된 값** | `[ 값 ]` | `[ 값 ]` | `[ 값 ]` | `[ 값 ]` | `[ 값 ]` |
| **메모리 주소** | **1000** | 1004 | 1008 | 1012 | 1016 |

- 상자들은 물리적으로 다닥다닥 붙어 있다.
- 컴퓨터는 시작점 `1000`만 알면 수학 계산 `1000 + 4 * 칸수`로 모든 위치를 즉시 찾아낼 수 있다.

2. 배열 이름 `a`는 왜 특별한가?

배열에서 가장 오해하기 쉬우면서도 중요한 성질은 이것이다: 배열의 이름 `a`는 배열 전체 값들의 모임이 아니다.

대부분의 문맥에서 배열 이름 `a`는 첫 번째 칸 `a[0]`의 주소값으로 변환되어 사용된다.

비유: `a`는 집 그 자체가 아니라, 그 집의 도로명 주소다.

## 9.7 고정된 메모리 주소: 왜 `b = a;` 같은 배열 대입이 안 될까?

8장에서는 규칙이니까 외우자고 했던 내용을, 이제 메모리 구조를 통해 이해해보자.

```c
int a[3] = { 1, 2, 3 };
int b[3];

b = a;  /* 컴파일 에러 발생! */
```

배열을 선언하는 순간 `int b[3];` 컴파일러는 2000번지와 같이 메모리의 특정 위치를 `b`라고 딱 정해버린다.

배열 `a`도 선언을 해서 다음과 같은 상황이 되었다고 가정해보자.

- `a`: 1000번지에 고정된 이름표
- `b`: 2000번지에 고정된 이름표

그런데 `b = a;`라는 코드는 이런 의미가 된다: "지금부터 2000번지 `b`를 1000번지 `a`로 바꿔라."

C언어에서 배열의 이름은 메모리상의 고정된 위치다. 태어날 때부터 정해진 주소를 바꿀 수 없으므로 대입 연산자 `=`의 왼쪽에 올 수 없다.
전문 용어로는 수정 가능한 l-value가 아니다라고 한다.

## 9.8 C가 무서운 이유: 메모리 안전장치가 없다

마지막으로, 이 장의 가장 중요한 경고를 한다.

1. 무한한 자유, 그리고 책임

C언어는 메모리 접근에서 안전장치가 거의 없다. 대표적인 예가 배열 범위 초과다.

```
int a[3] = { 1, 2, 3 };

/* a는 0, 1, 2번 방만 있다.
 * 그런데 3번 방에 짐을 넣으라고 한다면?
 */

a[3] = 10; 
```

2. 왜 에러가 안 날까? 이웃집 침범과 버퍼 오버플로우 _buffer overflow_

많은 초심자가 "컴파일 에러가 나겠지"라고 생각하지만, C언어는 에러를 내지 않는다.

메모리 관점에서 보면, `a[3]`은 `a[2]` 바로 뒷집 주소를 계산해서 접근한 것 뿐이다. 
보안 용어로 버퍼 오버플로우 _buffer overflow_ 또는 메모리 덮어쓰기 _memory overwrite_ 라고 부른다.

만약 그 뒷집에 다른 중요한 변수 `b`가 살고 있었다면 어떻게 될까?

메모리 상황: `int a[3]` 뒤에 `int b = 99;`가 선언되어 있었다고 가정해보자. 

| 메모리 주소 | 1000 | 1004 | 1008 | 1012 |
| :--- | :--- | :--- | :--- | :--- |
| **주인** | `a[0]` | `a[1]` | `a[2]` | `b` |
| **값** | 1 | 2 | 3 | 99 |

여기서 `a[3] = 10;`을 실행하면, 컴퓨터는 1012번지에 `10`을 쓴다. 나는 `a`만 건드렸는데, 갑자기 `b`의 값이 10으로 바뀐 것이다.

> [!NOTE]
> **보안 관점: 프로그램의 흐름 납치 _hijacking_**
>
> 함수가 실행될 때, 메모리 뒤쪽 어딘가에는 이 함수가 끝났을 때 원래 코드로 돌아갈 위치가 은밀하게 저장되어 있다.
>
> 해커들은 바로 이 점을 노린다. 배열의 인덱스를 고의로 크게 초과 입력하여, 이웃집 변수 `b`를 지나 '돌아갈 위치'가 저장된 메모리까지 덮어써 버리는 것이다.
>
> 해커가 이 돌아갈 위치를 '자신이 몰래 심어둔 악성 코드가 있는 주소'로 교묘하게 조작하면 어떻게 될까?
> 함수가 정상적으로 끝난 뒤, 프로그램은 원래 코드로 돌아가는 대신 해커의 악성 코드를 실행하게 된다.
> 정상적인 프로그램의 실행 흐름을 통째로 빼앗기는 것, 이를 보안 용어로 흐름 납치 _hijacking_ 이라고 부른다.

3. 미정의 동작

이것이 1장에서 언급했던 _undefined behavior_ 의 가장 무서운 사례다. 

- 운이 좋으면: 프로그램이 즉시 오류를 발생시키고 종료되어서 _segmentation fault_ 버그를 찾을 수 있다.
- 운이 나쁘면: 되는 것처럼 보이다가, 중요한 순간에 데이터가 꼬인다.
- 최악의 경우: 해커가 이 빈틈을 이용해 악성 코드를 심는다.

교훈: C언어에서 인덱스 관리는 전적으로 프로그래머의 몫이다. `0`부터 `크기 - 1`까지만 쓰도록 스스로 감시해야 한다.

> [!NOTE]
> **보안 관점: 버퍼 오버플로우를 막기 위한 방어막**
>
> C언어 문법 자체에는 메모리를 보호하는 안전장치가 없지만,
> 다행히 오늘날 우리가 코드를 빌드하고 실행하는 현대의 컴파일러와 운영체제는
> 이런 버퍼 오버플로우 공격을 막기 위해 든든한 방어막을 겹겹이 쳐두고 있다.
>
> 해커들의 공격 기법이 교묘해진 만큼, 방어 기술도 치열하게 발전해 왔다. 대표적인 두 가지 방패는 다음과 같다.
>
> **주소 공간 배치를 섞어버린다 _address space layout randomization, ASLR_**
>
> 해커가 흐름을 납치 _hijacking_ 하려면 제일 먼저 '돌아갈 위치'가 정확히 몇 번지에 있는지 알아야 한다.
> 그래서 현대의 운영체제는 프로그램을 실행할 때마다 메모리 안의 구조와 주소를 무작위로 뒤섞어버린다 _ASLR_.
> 해커 입장에서는 공격 목표의 주소가 실행할 때마다 바뀌니 제어권을 빼앗기가 극도로 어려워진다.
>
> **배열 끝에 세워둔 감시자 _stack canary_**
>
> 과거 광부들이 탄광에 들어갈 때 유독가스를 먼저 감지하기 위해 카나리아 새를 데려갔던 것에서 유래한 기술이다.
> 현대의 컴파일러는 배열과 '돌아갈 위치' 사이에 몰래 무작위 카나리아 숫자를 하나 끼워 넣는다.
> 만약 버퍼 오버플로우 공격으로 메모리를 덮어쓰며 밀고 들어온다면, 이 카나리아 숫자 역시 덮어씌워져 변형될 것이다.
> 프로그램은 함수를 끝내기 직전에 이 카나리아 숫자가 온전한지 확인하고,
> 조금이라도 값이 변했다면 즉시 프로그램을 강제 종료시켜 악성 코드의 실행을 원천 차단한다.
>
> **가장 강력한 방패는 방어적 프로그래밍 _defensive programming_**
>
> 그렇다면 이런 훌륭한 시스템만 믿고 인덱스 관리를 대충 해도 될까? 절대 그렇지 않다.
> 창과 방패의 싸움에서 해커들은 언제나 시스템의 빈틈을 파고드는 새로운 우회로를 찾아낸다.
>
> 결국 가장 근본적이고 뚫리지 않는 보안은 프로그래머가 처음부터 방어적으로 코드를 작성하는 것이다.
> 우리가 기초적인 문법을 배우면서 메모리의 동작 원리를 투명하게 들여다보는 이유도 바로 여기에 있다.
>
> 시스템이 대신 막아주기를 기대하기 전에,
> 애초에 배열의 크기를 넘어서지 않도록 스스로 경계를 꼼꼼히 검사하고 변수를 쓰기 전에 항상 초기화하는
> 올바른 습관을 들이자.
> 이것이 안전하고 단단한 소프트웨어를 만드는 가장 중요한 첫걸음이다.
