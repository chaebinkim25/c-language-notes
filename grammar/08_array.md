# 8. 여러 값을 한꺼번에 다루기: 배열 _array_

우리는 지금까지 `int x;`처럼 값 하나를 담는 상자인 변수를 만들고, 

`if`로 조건에 따라 다른 일을 하고,

`for`로 같은 일을 여러 번 반복하는 법을 배웠다.

그런데 현실 문제는 값이 하나만 있는 경우가 거의 없다.

- 장비에서 센서가 1초마다 측정한 온도 60개
- 업체에서 12개월간 발생한 매출
- 게임에서 몬스터 100마리의 체력

같은 상황에서 변수로만 프로그래밍을 하려고 한다면, 

```c
int s0, s1, s2, s3, s4, s5, s6, s7 /* ... */ ; /* 지옥 시작 */
```

과 같이 필요한만큼 변수들을 일일이 만들어야 한다. 

여기서 필요한 것이 배열 _array_ 이다. 

배열이란? 같은 타입의 상자가 메모리 상에 일렬로 쭉 늘어선 것이다. 마치 똑같은 구조의 집이 벽을 맞대고 붙어 있는 아파트와 같다.

> [!NOTE]
> **보안 관점: C 배열에는 울타리가 없다**
>
> 여기서 C 언어 배열만의 무시무시한 특징을 미리 알아둬야 한다.
> 다른 최신 언어들의 아파트는 101호와 102호 사이에 단단한 콘크리트 벽에 해당하는 배열 경계 검사가 있다.
>
> 하지만 C 언어의 아파트는 벽이 종이장처럼 얇다.
> 여러분이 101호에 물건을 너무 많이 채워 넣으면, 102호로 물건이 넘쳐흘러 들어가 버린다.
>
> 그것이 바로 해킹 역사상 가장 많은 사고를 낸 주범, 버퍼 오버플로우 _buffer overflow_ 의 원리다.
> 배열은 강력하고 빠르지만, 내 구역이 어디까지인가를 개발자가 직접 챙기지 않으면 옆집 데이터를 파괴하는 시한폭탄이 된다.
>
> 위험하고도 매력적인 이 도구를 안전하게 다루는 법을 배워보자.

## 8.1 배열과 반복문: 좋은 파트너

`for` 문법은 **0**, **1**, **2**, **3**, ...처럼 루프 변수에 숫자들을 바꿔서 저장하며 같은 코드를 실행하는 강점이 있다.

배열은 **0**번째 값, **1**번째 칸, **2**번째 칸, **3**번째 칸, ... 처럼 번호를 붙은 여러 공간을 제공한다. 

`for`로 만들어낸 숫자를 배열의 방 번호로 쓰면, 시너지를 낼 수 있다.

비유를 하지면, 

- **배열:** 물건이 순서대로 정리된 긴 선반 _data_
- **`for`문:** 그 선반을 처음부터 끝까지 지나가는 자동 레일 위에 있는 엔진 _processor_

선반은 가만히 있지만, 엔진이 레일을 타고 이동하며 값을 하나씩 처리해준다. 덕분에 데이터가 100개든 100만 개든 코드 길이는 똑같다.

> [!NOTE]
> **보안 관점: 레일 끝에는 안전바가 없다**
>
> 하지만 이 완벽해 보이는 조합에는 치명적인 보안 허점이 있다.
> C 언어의 배열 끝에는 자동 멈춤 장치 _stopper_ 가 없다.
>
> 만약 실수로 `for`의 반복 횟수를 선반의 길이보다 길게 설정하면 어떻게 될까?
>
> - **다른 언어 _java_, _python_:** "레일 끝입니다!"라고 경고하며 엔진을 멈추고 에러를 발생시킨다.
> - **C 언어:** 엔진이 멈추지 않고 옆집 선반을 부수고 지나간다.
> 
> 그것이 바로 버퍼 오버플로우 _buffer overflow_ 다.
>
> `for`문이라는 강력한 엔진을 쓸 때는, 레일이 어디서 끝나는지를 의미하는 배열의 크기를 운전자인 여러분이 직접 확인하고 멈춰 세워야 한다.
> C 언어는 여러분의 운전 실력을 100% 믿고, 브레이크를 달아주지 않기 때문이다.

## 8.2 배열의 기본 모양: 한 줄로 늘어선 똑같은 타입 상자들

예를 들어 `int` 배열을 크기 5로 만들면 이런 느낌이다.

```
 a[0]    a[1]    a[2]    a[3]    a[4]    위험 지역
 상자    상자     상자    상자    상자     다른 상자
```

각 칸은 `int` 변수 하나와 똑같은 역할을 한다. 

- `a[0]`도 `int` 변수 하나
- `a[1]`도 `int` 변수 하나
- 나머지도 전부 `int` 변수 하나다.

여기서 대괄호 `[]` 안의 숫자를 인덱스 _index_ 라고 부른다. 
우리는 흔히 방 번호라고 부르지만, 보안을 위해 더 정확한 의미를 알아야 한다.

> [!NOTE]
> **보안 관점: 인덱스는 순서가 아니라 거리다**
>
> 많은 초보자가 첫 번째니까 1번 방이라고 생각하다가 해킹 사고를 낸다.
> C 언어에서 인덱스는 시작점에서 떨어진 거리 _offset_ 을 뜻한다.
>
> - **`a[0]`:** 시작점에서 `0`칸 떨어진 곳으로, 여기까지 **1**개 데이터 존재
> - **`a[1]`:** 시작점에서 `1`칸 떨어진 곳으로, 여기까지 **2**개 데이터 존재
> - **`a[2]`:** 시작점에서 `2`칸 떨어진 곳으로, 여기까지 **3**개 데이터 존재
> 
> 따라서 크기가 `n`인 배열의 마지막 인덱스는 `n`이 아니라 `n - 1`이다.

> [!WARNING]
> **5개짜리 배열에는 5번 방이 없다**
>
> `int a[5];`라고 선언하면 상자가 **5**개 만들어진다.
> 각 상자의 인덱스는 `[0]`, `[1]`, `[2]`, `[3]`, `[4]`가 된다.
> 그런데 초보자들은 무의식적으로 `a[5]`에 값을 넣으려고 할 때가 있다.
>
> ```c
> int a[5];
> a[5] = 100;        /* 대형 사고: 버퍼 오버플로우 발생! */
> ```
>
> 인덱스 `5`의 상자 `a[5]`는 내 땅이 아니고 엄연히 남의 땅으로, 다른 변수나 시스템에서 쓰고 있거나 앞으로 쓰려고 잡아놓았다.
> 여기를 건드리면 해커에게 뒷문을 열어주게 된다.
> 배열을 쓸 때는 항상 주문을 외우자. `0`부터 시작해서 `n - 1`에서 끝난다.

## 8.3 배열 선언: 몇 칸짜리 선반을 만들 것인가

배열을 만드는 문법은 다음과 같다.

```c
int a[5];
```

의미는 이렇게 해석하면 된다. 

- **`int`:** 이 선반의 각 칸은 정수 상자다.
- **`a`:** 선반 이름은 `a`다.
- **`[5]`:** 칸이 총 `5`개다. 

비유하자면, 설계도에 방 개수를 미리 확정하고 집을 짓는 것과 같다. 

> [!WARNING]
> **배열 크기는 `1` 이상이어야 한다**
>
> `int a[0];`처럼 크기가 `0`인 배열을 선언하는 것은 표준 C 언어에서 금지되어 있다.
> 상자를 만들라면서 **0**개를 달라고 하는 것은 논리적으로 모순이기 때문이다.
> 일부 컴파일러에서는 특수한 목적으로 허용하기도 하지만, 우리는 표준을 따르므로 반드시 `1` 이상의 양수를 써야 한다.

**선언한 직후의 상태: 비어있는 상자가 아니다**

배열을 방금 막 선언했다면, 그 안에는 무엇이 들어있을까?

일반 변수를 선언했을 때와 마찬가지로, 알 수 없는 값이 들어있다.

> [!WARNING]
> **보안 관점: 초기화 되지 않은 배열에는 누군가의 비밀이 들어있을 수 있다**
>
> 내가 방금 대여한 배열의 메모리 공간은, 방금 전까지 다른 블록에서 쓰던 공간일 수 있다.
>
> 초기화를 하지 않은 배열에 저장된 값을 실수로 노출하면, 그 정보가 유출될 수 있다 _information leak_.
>
> 근본적으로는 초기화되지 않은 변수에 저장된 값을 읽는 것 자체가 취약점이다.
>
> 그리고 일반적인 변수에서는 메모리에서 변수 하나에 해당하는 정보만 유출될 수 있지만,
>
> 배열에서는 변수 여러개에 해당하는 정보가 유출될 수 있기 때문에 더 위험하다.
>
> 배열의 크기를 아주 크게 잡은 경우는 더 많이 위험하다. 
> 
> 미국 국토부에서 관리하는 _common weakness enumeration_ 소프트웨어 취약점 목록 _CWE-457_ 에도 등재되어 있고,
>
> _CERT EXP33-C_ 와 _MISRA C Rule 9.1_ 에서도 금지한다. 

**현대 C의 유혹: 변수로 크기 정하기 _variable length array, VLA_**

C99 표준에서는 배열 크기를 변수로 정하는 것을 허용한다. 
프로그래밍 용어로 가변 길이 배열 _variable length array, VLA_ 라고 한다. 

```c
int n = get_input();        /* 유저에게 크기를 입력받았다. */
int a[n];                   /* 입력받은 n만큼 크기의 배열을 만든다. */
```

짓슨 순간에 방 개수를 정해서 바로 짓는 임시 건물과 같아서 매우 편리해 보인다.
하지만, 보안 전문가들은 이 편리한 기능을 시한폭탄 취급하며 사용을 말린다.

그런 이유로 C11 이후 표준에서는 _VLA_ 지원이 선택 사항으로 변경되었다. 

> [!WARNING]
> **보안 관점: 편한 만큼 위험한 _VLA_**
>
> 문법적으로 가능하다고 해서 실무에서 막 써도 되는 것은 아니다.
> 리눅스 _linux_ 운영체제를 만드는 개발팀의 공식 문서에 따르면, _VLA_ 는 더 나쁜 기계어를 만들고, 특정 상황에서는 프로그램 종료 _crash_ 를 일으키거나, 다른 변수의 값을 덮어쓸 수 있다.
> 
> 또한, 변수 상자가 만들어지는 작업 공간 _stack_ 은 보통 수 MB 정도로 작다.
> 가변길이 `a[n]`에서 작업 공간의 크기를 넘어서 배열을 만들려고 하면 _stack_ 메모리가 고갈되어 프로그램이 멈출 수 있다 _stack overflow_.
>
> 또한 보안의 핵심은 예측 가능성이다. 고정 크기 배열은 메모리를 얼마나 쓸지 미리 알 수 있어서 감사 _audit_ 가 쉽지만, 가변 길이 배열은 실행해보기 전까지 메모리를 얼마나 쓸지 모른다.
>
> 관련 보안 가이드라인: **CERT ARR32-C** 가변 길이 배열의 크기가 적절한지 보장하라, **MISRA C Rule 18.8** 가변 길이 배열을 사용하지 마라

## 8.4 배열 초기화: 만들면서 값 저장하기

배열도 변수를 만들 때처럼, 선언과 동시에 값을 저장할 수 있다.

**값을 꽉 채우기: 완전 초기화**

```c
int a[5] = {10, 20, 30, 40, 50};
```

배열을 선언할 때 중괄호 `{}` 안에 값을 나열하면, 순서대로 저장된다.

- `a[0] = 10;`
- ...
- `a[4] = 50;`

**값을 덜 채우기: 부분 초기화로 나머지는 `0`**

칸 수보다 적게 적으면 어떻게 될까?

```c
int a[5] = {1, 2};
```

그럴 경우 남는 칸은 자동으로 `0`이 된다는 규칙이 있다.

- `a[0] = 1;`
- `a[1] = 2;`
- 나머지 `a[2]`, `a[3]`, `a[4]`는 모두 `0`

> [!TIP]
> **배열의 모든 원소를 `0`으로 초기화하려면?**
>
> `int a[5] = {0};`처럼 초기값을 `{0}`이라고 적으면, 첫 칸은 `0`, 나머지도 규칙에 따라 자동 `0`이 되므로, 
> 배열 전체가 깔끔하게 `0`으로 초기화된다.

> [!NOTE]
> **자동차 소프트웨어의 규칙: MISRA C Rule 9.3**
>
> 자동차 업계 표준인 MISRA C는 배열이 부분적으로 초기화되어서는 안 된다 *arrays shall not be partially initialized* 라고 엄격히 규정한다.
>
> 그 이유는 개발자의 의도가 불분명하기 때문이다. `int a[5] = {1};`을 보면, 누군가는 "전부 `1`로 채우고 싶었나?"라고 오해할 수 있다.
>
> **해결책:**
> 1. **전부 다 적거나:** `{1, 0, 0, 0, 0}` 처럼 모든 값을 명시적으로 적어라.
> 2. **아예 비우거나:** `{0}`처럼 `0`으로 초기화하는 경우만 예외적으로 허용한다.

**보너스: 칸 수 생략하기**

초기화 값이 있으면, 굳이 칸 수를 세지 않고 컴퓨터에게 맡길 수도 있다.

```c
int a[] = {1, 3, 5};   /* 칸 수를 빈칸으로 두면, 컴파일러가 초기값이 3개 있으니 3칸으로 자동 설정해준다. */
```

대괄호를 비워두면, 컴파일러가 중괄호 안의 개수를 세어서 알아서 `int a[3]`으로 만들어준다. 

> [!CAUTION]
> **_VLA_ 는 초기화자를 쓸 수 없다.**
>
> 앞서 8.3절에서 경고한 가변 길이 배열 _VLA_ 의 또 다른 단점은 초기화 문법을 사용할 수 없다는 것이다.
> _VLA_ 를 쓰지 말아야 할 또 하나의 중요한 보안적 이유다.

> [!NOTE]
> **C 표준의 변화: 배열 초기값과 상수 식**
>
> C89 표준에서는 배열의 초기값 목록 `{}` 안에 들어가는 모든 값이 상수거나 상수를 계산한 식이어야 했다.
>
> C99 이후 표준에서는 변수나 변수를 쓴 수식도 쓸 수 있게 되었다. 

## 8.5 배열에 값 저장하기와 저장된 값 쓰기: `a[i]`

**배열에 값 저장하기**

`a[i]`는 배열 `a`의 `i`번째 칸이라는 뜻의 변수여서, 여기에 값을 저장할 때는 일반 `int` 변수에 하는 것처럼 대입 연산자 `=`를 쓴다.

```c
a[0] = 10;
a[1] = 20;
a[2] = 10 + 20;       /* 계산 결과인 30도 저장할 수 있다. */
```

여기서 핵심 포인트는 인덱스 _index_ 가 계산 가능하다는 것이다. 꼭 숫자만 쓰지 않아도 된다. 변수나 계산식도 인덱스로 쓸 수 있다.

```c
int a[5];
int i = 3;

a[i] = 40;            /* a[3]에 40 대입 */
a[i + 1] = 50;        /* a[4]에 50 대입 */
```

인덱스가 계산 가능하기 때문에 `for (i = 0; ...)` 반복문에서 `a[i]`를 이용해 모든 값을 처리할 수 있다.

**배열에 저장되어 있는 값 읽기**

저장된 값을 읽을 때도 일반 변수와 똑같다. 값을 읽는 맥락에서 `a[i]`라고 하면, 배열 `a`의 `i`번째 칸에 저장된 값이 읽어진다. 

```c
/* 상자 안에 저장된 값을 다른 변수에 저장하기 */
x = a[1];             /* 20 */

/* 덧셈에 사용하기 */
sum = a[0] + a[1];    /* 10 + 20 */
```

> [!NOTE]
> **보안 관점: 범위를 벗어나는 _out-of-bounds_**
>
> 만약 배열에 값 저장하기 예제에서 `i`가 `5`였다면 어떻게 되었을까?
>
> ```c
> int a[5];               /* 0, 1, 2, 3, 4번 방만 있음 */
> int i = 5;
>
> a[i] = 50;              /* a[5]에 대입은 범위 초과 */
> ```
>
> C 언어 표준에서 배열 범위를 벗어난 원소에 접근하려고 하는 것은 미정의 동작이다 _undefined behavior_. 
> 존재하지 않는 5번 방에 값을 저장하면 _out-of-bounds_, 운 좋게 컴파일러가 이를 막아주거나 프로그램 실행시 바로 종료될 수도 있지만, 다른 변수의 값을 덮어쓰면서 프로그램이 오동작하며 계속 돌아갈 수도 있다 _buffer overflow_.

> [!NOTE]
> **자동차 소프트웨어의 규칙: MISRA C Rule 4.14**
>
> 특히 사용자 입력, 파일, 네트워크 등 외부 소스에서 온 데이터가 배열 인덱스로 사용될 때는 반드시 유효성을 검사해야 한다.
>
> 올바른 인덱스 검사 패턴은 상한선 _upper bound_ 와 하한선 _lower bound_ 를 모두 검사하는 것이다.
>
> ```c
> int table[10];
> int index = -1;
> /* index에 확인할 배열 인덱스를 저장한다 */
> 
> if (index < 0) {
>         /* 하한선 에러 처리 */
> } else if (index >= 10) {
>         /* 상한선 에러 처리 */
> } else {
>         /* 여기서는 안전하게 table[index]에 접근할 수 있다. */
> }
> ```
>
> 또한 인덱스를 계산해서 사용하는 경우, 6.6장에서 본 것과 같이 정수 오버플로우를 주의해야 한다. 

## 8.6 배열 크기 관리: 매크로 상수의 도입 `#define`

앞에서 배열의 크기는 변수로 정하면 위험해서 _VLA_ 를 금지하기로 했다.
그렇다고 숫자를 직접 적자니, 나중에 크기를 `5`에서 `10`으로 바꿀 때 코드 구석구석을 찾아다니며 고쳐야 하는 불편함이 생긴다.

이때 사용하는 것이 전처리 지시 문법 `#define`이다.

**문법 _syntax_**

```c
#define 이름 값 줄바꿈
```

이것은 컴파일러에게 여기서부터 파일의 끝까지 "코드에 있는 '이름'을 전부 엔터 앞에까지의 '값'으로 바꿔치기 해줘!"라고 명령하는 것이다.

C 언어에서는 관습적으로 매크로 이름은 대문자로 쓴다.

**예제: 유지보수가 쉬운 방 만들기**

```c
#define SIZE 5      /* 주의: 끝에 세미콜론 ;을 붙이지 않는다! */

int main(void)
{
        /* 컴파일 전에 먼저 SIZE가 5로 바뀐다. int a[5]; 와 똑같아진다. */
        int a[SIZE]; 
        
        return 0;
}
```

이제 우리는 `SIZE` 뒤에 있는 숫자 하나만 고치면, 프로그램 전체에 있는 배열의 크기를 안전하게 변경할 수 있다.

> [!CAUTION]
> **전처리 문법에서는 세미콜론 `;` 절대 금지**
>
> `#define`은 C 언어의 문장이 아니라 전처리 _preprocessor_ 를 할 때 적용되는 지시 사항이다.
> 만약 `#define SIZE 5;` 라고 뒤에 세미콜론을 붙이면 어떻게 될까?
> 
> `int a[SIZE];`가 `int a[5;];`로 변환된다.
> 
> 괄호 안에 뜬금없이 세미콜론이 들어가서 문법 에러가 발생한다. 초보자가 많이 하는 실수니 주의하자.

> [!NOTE]
> **MISRA C Rule 11.4: 상수 표현식 사용 권장**
>
> 신뢰성이 중요한 소프트웨어에서는 숫자 리터럴 `3`, `100` 등을 코드 흐름 제어에 직접 쓰는 것을 피하고, 의미 있는 이름을 가진 상수 `MAX_USER`를 사용할 것을 권장한다.
> 코드를 읽는 사람에게 이 숫자가 무슨 뜻인지 알려줄 수 있고, 수정 시 실수를 막아주기 때문이다.

## 8.7 전체를 처리하는 가장 기본 패턴: `for` + 배열

배열의 진정한 힘은 반복문과 만났을 때 폭발한다.
값이 5개든, 5만 개든, 우리는 단 3줄의 코드로 모든 데이터를 처리할 수 있다.

**기본 패턴: `0`부터 `n - 1`까지**

```c
#define SIZE 5        /* SIZE 라는 토큰을 찾으면, 5로 바꾸기를 한다. */

int main(void)
{
        int i;
        int a[SIZE] = {10, 20, 30, 40, 50};
        int sum = 0;

        /* 공식: 배열 전체 반복 패턴 */
        for (i = 0; i < SIZE; i = i + 1) {
                sum = sum + a[i];        /* 값을 꺼내서 누적하기 */
        }

        /* 반복이 끝나면, sum에는 150이 저장된다. */

        return 0;
}
```

이 코드는 다음과 같이 작동한다.

1. **`i = 0` 시작점:** 배열의 인덱스는 항상 `0`부터 시작하므로, `i`도 `0`에서 출발한다.
2. **`i < SIZE` 종료 조건:** 배열 크기가 **5**라면, 인덱스는 **4**번까지 있다.
   따라서 `i`가 **5**가 되는 순간 즉시 멈춰야 한다.
4. **`i = i + 1` 증가:** 다음 칸으로 이동한다.

> [!NOTE]
> **역사 관점: 배열의 크기를 왜 코드에 따로 써야 할까? B 언어의 유산**
>
> 자바 _java_나 파이썬 _python_ 같은 현대 언어에서는 `for`문을 돌릴 때 배열 스스로가 자신의 길이를 알려준다.
> 하지만 C 언어에서는 우리가 직접 `#define SIZE 5`처럼 길이를 기억해두고 `for`에 적어주어야 한다.
> 도대체 왜 이렇게 불편하게 만들었을까?
> 
> 해답의 열쇠는 C 언어의 조상인 B 언어에 있다.
> 
> B 언어에는 C처럼 구분된 배열 타입이 없었다.
> 배열은 연속된 메모리 블록이었고, 배열 변수에 해당하는 상자에는 그 블록의 시작 주소가 저장되었다.
>
> 컴퓨터 실행 환경이 매우 열악했고, 몇 명이서 모든 소프트웨어를 개발하던 당시 상황을 생각하면 합리적인 선택이었다.
> 
> C 언어는 배열 타입을 도입했지만, 수식에서 배열 이름이 사용될 때는 B 언어에서와 같이 제일 앞 원소의 주소로 자동 변환되도록 설계했다 [^1].
> 개발자가 이미 배열의 크기를 알고 코드를 작성했기 때문에, 실행 중에 크기를 따로 저장하는 것은 당시 기준으로 낭비 _overhead_ 였다.

> [!CAUTION]
> **보안 관점: 부등호 `<` 와 `<=`**
>
> 초보자가 가장 많이 하는 실수가 종료 조건에서 `=`를 붙이는 것이다.
> 
> ```c
> int i;
> int a[5];
> 
> /* 위험: 크기가 5인데, 5번 방까지 가려고 한다. */
> for (i = 0; i <= 5; i = i + 1) {
>         /* a[i]에 접근 */
> }
> ```
>
> 이 코드는 `i`가 `0`, `1`, `2`, `3`, `4`, `5`일 때 실행된다. 마지막에 존재하지 않는 `a[5]`에 접근하는 _out-of-bound_ 문제가 발생한다.
> 더 나아가 다른 메모리를 침범하는 _buffer overflow_ 취약점으로 이어질 수 있다.
> 
> 배열을 다루는 `for`문에서는 습관적으로 `<` 작다를 쓰자.

[^1]: 배열 이름이 제일 앞 원소의 주소로 자동 변환되지 않는 예외는 `sizeof` 연산자나 `&` 연산자의 피연산자일 때, 그리고 문자열 상수인 배열의 경우에는 문자 배열의 초기값으로 쓰일 때이다.
